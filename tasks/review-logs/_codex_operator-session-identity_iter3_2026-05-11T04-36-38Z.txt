**Findings**

- **[critical] V1’s locked connect-state is internally contradictory.** [§7.4](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:374>) says the V1 OpenAI registry uses `self_declaration`, so “EVERY new operator_session connection in V1 lands in `connected_unverified`” and “EVERY connect attempt requires a `disclosureAcceptance` block.” But [§11.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:804>) still frames disclosure as “For Plus tier” only and inserts Plus rows as `connected_usable`, and [§17.5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1180>) hard-codes the Plus happy-path final state as `connected_usable`. Those cannot all be true at once for the stated V1 registry.

- **[critical] The consent uniqueness rule conflicts with the immutable `connection_id` back-pointer and multi-connection posture.** [§7.2](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:262>) makes `operator_session_consents.connection_id` a one-time back-fill, then immutable; [§7.2 DDL](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:276>) adds `UNIQUE (organisation_id, subaccount_id, user_id, disclosure_version)`; [§16.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1051>) and [§16.6](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1125>) treat duplicates as idempotent hits. That breaks as soon as the same user connects a second Plus/unverified subscription in the same subaccount/version: the second flow cannot create a new consent row, and it also cannot legally update the existing row’s immutable `connection_id` to point at the new connection.

- **[important] Disclosure-version supersession points at two different states.** [§7.5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:416>) defines `disabled` as covering “disclosure version supersession,” but [§11.4](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:842>) and [§17.5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1182>) transition bumped disclosures to `connected_needs_consent`. That is a direct state-machine contradiction.

- **[important] The router registration path is wrong in the spec.** [§8.9](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:487>) and [Chunk 5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:909>) say to register routers in `server/routes/index.ts`, but this repo mounts routers from `server/index.ts`, and there is no existing `server/routes/index.ts` registration surface.

- **[important] The existing web-login primitive is named incorrectly.** [§8.8](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:480>) says `webLoginConnectionsGovern.ts` “reuses existing `webLoginConnections.ts` service layer,” but `server/routes/webLoginConnections.ts` is the route file; the service primitive in-tree is `server/services/webLoginConnectionService.ts`. That is stale/misnamed guidance against the current repo.

- **[important] The spec defines two adjacent list producers for AI Subscriptions and never names the canonical one.** [§9.2](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:594>) says the `AiSubscriptionConnection` producer is `server/routes/operatorSessionConnections.ts`, but [§10.5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:790>) says the existing `GET /api/subaccounts/:id/connections` remains the Govern-surface list endpoint, and [§8.9 / Chunk 5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:486>) extend `listConnections` for that response. Right now the spec describes both a dedicated operator-session list route and the unified connections list route as producers, with no source-of-truth decision.

- **[important] Chunk 10 introduces a new client call with no named server producer.** [Chunk 10](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:965>) adds `governApi.ts: getAgentAllowedSubscriptions(agentId, subaccountId)` for the Model Access summary, but there is no corresponding server route, service method, or contract anywhere in [§8 File inventory](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:420>) or [§10.4 Route guards](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:776>). That is both file-inventory drift and an unnamed new primitive.

- **[important] The spec’s own testing-posture section contradicts the framing it cites.** [§15](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1018>) says “Allowed tests in this spec (pure functions only),” but the table includes `credentialBrokerService.test.ts` and `operatorSessionConsentService.test.ts`, and [§17.2](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1161>) / [§17.5](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1177>) make those non-pure tests acceptance-gating. That is a checklist/framing compliance miss inside the spec as written.

- **[minor] The file inventory miscounts the schema change on `integrationConnections.ts`.** [§7.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:215>) and [§8.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:429>) both say 0319 adds six columns, but [§8.3](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:442>) says `server/db/schema/integrationConnections.ts` adds “5 new columns.”

- **[minor] The schema acceptance text assigns the new tables to the wrong migration.** [§17.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:1155>) says “Migration 0319 applies cleanly; both new tables exist with ENABLE ROW LEVEL SECURITY + FORCE ROW LEVEL SECURITY,” but the new tables are created in 0318 per [§8.1](</c:/files/Claude/automation-v1-2nd/docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:428>). 0319 only alters `integration_connections`.
