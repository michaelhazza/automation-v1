- **[critical] `docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md` §11.1 contradicts its own sanctioned-tier path.** In the same section, the gate says sanctioned verified tiers require “no disclosure required” and can land directly in `connected_usable` (`§11.1`, lines 840-847), and `§17.5` repeats “no consent row required” for the introspection/sanctioned path (line 1249). But the “canonical one transaction” immediately below hard-codes `disclosureAcceptance` in the POST body and says step 4 always `INSERT`s an `operator_session_consents` row plus `granted` event before the connection row (lines 856-872). Those cannot both be true; the spec needs one canonical connect sequence for sanctioned verified connects vs disclosure-required connects.

- **[important] The spec gives two different single sources of truth for failover ordering.** `§9.7` says `label ASC NULLS LAST` “is implemented in the broker's SQL — not in JavaScript” (lines 742-744). But `§8.5`, `§12 Chunk 2`, `§12 Chunk 4`, and `§15` all say ordering is delegated to `server/services/credentialBrokerServicePure.ts` via `orderResolvedCredentials(rows)` and explicitly describe the helper as taking an unordered list and returning the ordered array (lines 480, 936, 953, 1098). That is an internal contradiction on where the ordering invariant actually lives.

- **[important] The broker retrieval invariant has conflicting verification targets.** `§9.1` says “A unit test proves that calling `issueCredential` … throws before any decryption step” (lines 602-607). `§15` then says the non-pure `credentialBrokerService.issueCredential` is only verified at “integration-test time only (Phase 2+)” and V1 uses only the extracted pure helper test (line 1095). `§17.2` follows the pure-helper version, not the `issueCredential` version (lines 1229-1231). Those sections disagree on whether V1 acceptance is on `issueCredential` itself or only on `assertCredentialUsableOrThrow`.

- **[important] `usability_state` ownership is internally inconsistent between services.** `§7.5` says “State transitions are written only by `operatorSessionLifecycleService`” (line 431), and `§8.5` describes `server/services/operatorSessionLifecycleService.ts` as “the only code allowed to write `usability_state`” (line 477). But the same inventory block describes `server/services/operatorSessionService.ts` as “Connect flow, plan detection, `usability_state` management” (line 475), and Chunk 3 says it owns “initial `usability_state` assignment” (line 946). If initial writes on insert are the only exception, the spec should say that explicitly; as written, the ownership boundary is contradictory.

- **[important] The allowlist/availability model is load-bearing in contracts but has no pinned schema mechanism.** `§9.2` exposes `availabilityScope` and `allowedAgentIds` on `AiSubscriptionConnection` (lines 644-645), and `§9.7` makes broker failover depend on them (line 738). But the only place the storage mechanism appears is Open Question 2: “stored in `configJson` on the `integration_connections` row” (line 1299). `§7` never defines that JSON shape, `§9.6` never declares it as a source of truth, and `§8` has no inventory entry for the `configJson` contract change. This is a load-bearing claim without a backing mechanism and an inventory gap around the actual persistence shape.

- **[important] Ownership framing is stale/inconsistent in the connection contract.** The spec’s final framing is user-attributed consent and user-bound offboarding: `§7.2` stores `user_id`, `§7.5` has `disabledReason: 'owner_inactive'`, and `§18` says re-auth identity mismatch must detect that “the re-auth-ing user is the same identity as the original consent owner” (line 1303). But `§9.2` still returns `owner: { kind: 'workspace'; id: string; name: string }` (line 647). That shape reflects workspace ownership, not the user-bound owner semantics the rest of the spec depends on.

- **[important] File-inventory lock has drift for the agent-side allowed-subscriptions route/method.** `§10.4` and Chunk 10 introduce `GET /api/subaccounts/:id/agents/:agentId/allowed-subscriptions` plus `operatorSessionService.listAllowedSubscriptionsForAgent(agentId, subaccountId)` (lines 815 and 1024). But `§8.8` describes `server/routes/operatorSessionConnections.ts` only as “Full CRUD: connect, list, get, update … disconnect, consent, re-auth trigger” (line 504), and `§8.5` describes `operatorSessionService.ts` only as connect flow / plan detection / `usability_state` management (line 475). Those newly introduced surfaces are not reflected in the file inventory lock entry that is supposed to be the single source of truth.

- **[minor] The deprecation/removal verdict for the legacy page is inconsistent.** `§5.2` says the legacy `IntegrationsAndCredentialsPage` and `CredentialsTab` “are deprecated and removed as part of this spec” (line 116), but `§5.6`, `§8.14`, `§12 Chunk 9`, and `§17.7` all soften that to “removed or converted to redirect” (lines 146, 559, 1015, 1272). The spec should pick one verdict for `client/src/pages/IntegrationsAndCredentialsPage.tsx`.

- **[minor] `OPERATOR_SESSION_DISCLOSURE_VERSION` is a new config dependency without a concrete owning file/path.** `§11.4` introduces “When `OPERATOR_SESSION_DISCLOSURE_VERSION` config increments” (line 898), but `§8` does not name where that config lives, and no config file other than `server/config/operatorSessionProviders.ts` is inventoried for it. That misses the checklist requirement to pin new primitives/configs to a concrete path.
