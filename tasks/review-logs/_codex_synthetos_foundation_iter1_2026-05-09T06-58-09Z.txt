  296: ## Section 10 â€” Execution-safety contracts (new writes and state machines)
  297: 
  298: Before sending any spec for review that introduces new write paths, state machine transitions, or externally-triggered operations, verify each of the following is pinned in the spec. These are routinely missing from first-draft specs and are the root cause of the most expensive post-ship bugs.
  299: 
  300: ### 10.1 Idempotency posture
  301: 
  302: For every externally-triggered write, state one of:
  303: 
  304: - `key-based` â€” a unique key (e.g. `(artefactId, decision)`) guarantees exactly-once with a DB unique constraint. Name the key and the index.
  305: - `state-based` â€” the write is guarded by an optimistic predicate (`UPDATE ... WHERE status = 'expected_pre_state'`). Name the predicate.
  306: - `non-idempotent (intentional)` â€” the operation is inherently non-idempotent; state why and what the caller's retry contract is.
  307: 
  308: Do not describe an operation as "idempotent" without naming which of the three applies. "We'll handle retries" is not an idempotency posture.
  309: 
  310: ### 10.2 Retry classification
  311: 
  312: For every write or external call, declare one of: `safe` (unconditionally retryable), `guarded` (retryable with an idempotency key or optimistic predicate), or `unsafe` (caller bears retry risk). Any `unsafe` operation must be wrapped by a `safe` or `guarded` boundary before the caller can retry it. Name the boundary.
  313: 
  314: ### 10.3 Concurrency guard for racing writes
  315: 
  316: If two concurrent callers can race to write the same terminal state (e.g. two approve requests for the same decision, two job instances for the same org), the spec must declare the concurrency guard:
  317: 
  318: - Optimistic predicate: `UPDATE ... WHERE status = 'review_required'` â†’ 0 rows affected = conflict
  319: - Unique constraint: DB-level (`UNIQUE (artefact_id, decision)`) + catch `23505` â†’ defined HTTP status
  320: - First-commit-wins: the 0-rows-updated path returns the winning decision to the losing caller
  321: 
  322: Name the guard, the DB mechanism, and the losing-caller response. "The DB will handle it" is not a guard.
  323: 
  324: ### 10.4 Terminal event guarantee
  325: 
  326: Every cross-flow chain that emits events must declare:
  327: 
  328: - Exactly one terminal event (the event that marks the logical run complete)
  329: - Post-terminal prohibition â€” no further events with the same correlation key after the terminal
  330: - The terminal event's `status` field: `success | partial | failed`
  331: 
  332: If the chain has multiple success paths or multiple error paths, each path gets exactly one terminal event â€” they are mutually exclusive.
  333: 
  334: ### 10.5 No-silent-partial-success
  335: 
  336: Every flow that can partially complete must emit an explicit `status: 'partial'` terminal event (not `status: 'success'` with a silent partial-failure). Name the conditions under which `partial` fires vs `failed`.
  337: 
  338: ### 10.6 Unique-constraint-to-HTTP mapping
  339: 
  340: For every DB unique constraint the spec introduces, pin the HTTP status returned to the caller when the constraint is violated. Never let a `23505 unique_violation` bubble as a 500 â€” map it to a named status (409, 422, or 200-idempotent-hit) and document which one and why.
  341: 
  342: ### 10.7 State machine closure (if the spec introduces or modifies a state machine)
  343: 
  344: If the spec introduces or modifies a state machine (step transitions, run aggregation, approval boundaries, status enums), include a State/Lifecycle subsection that pins:
  345: 
  346: - Valid transitions (and which transitions are forbidden)
  347: - What execution record must exist before a terminal state is written
  348: - Whether the status set is closed (adding a new status value requires a spec amendment)
  349: 
  350: A spec that describes behaviour without pinning valid transitions and forbidden transitions will have its state machine diverge from implementation within two feature cycles.
  351: 
  352: ### Reviewer signal this prevents
  353: 
  354: "No idempotency posture declared" / "What happens when two callers race here?" / "How does the caller know if this partially failed?" â€” all caught in the pre-launch hardening pre-implementation hardening pass (amendments v2â€“v5, Chunks 3, 4, 5). These gaps are architectural, not stylistic â€” they produce correctness bugs at production load.

 1695: ## 6. Schema Migrations Summary
 1696: 
 1697: ### 6.1 Migrations introduced by this spec
 1698: 
 1699: | # | File | Purpose | Reversible | Locks table? |
 1700: |---|---|---|---|---|
 1701: | 1 | `NNNN_agent_runs_controller_style.sql` | Add `controller_style` column with default `'native'`; partial index for operator runs | Yes | Briefly (column add) |
 1702: | 2 | `NNNN_agent_runs_policy_envelope.sql` | Add `policy_envelope_snapshot jsonb` column; nullable | Yes | Briefly |
 1703: | 3 | `NNNN_subaccount_agents_governance.sql` | Add `controller_style_allowed`, `allowed_environments`, `max_risk_tier`, `require_approval_at_tier` columns | Yes | Briefly |
 1704: 
 1705: All migrations have `.down.sql` counterparts in `migrations/_down/`.
 1706: 
 1707: ### 6.2 No new tables
 1708: 
 1709: This spec deliberately introduces zero new tables. All net-new state is captured as columns on existing tables (`agent_runs`, `subaccount_agents`). This minimises schema risk and preserves rollback simplicity.
 1710: 

 succeeded in 1388ms:
 1710: 
 1711: ### 6.3 Migration ordering
 1712: 
 1713: The three migrations are independent and can apply in any order. They are:
 1714: 
 1715: 1. Idempotent on re-apply (no-op if column exists).
 1716: 2. Reversible (drop column reverses).
 1717: 3. Non-destructive (no data loss).
 1718: 
 1719: ### 6.4 Backfill jobs
 1720: 
 1721: Two one-time backfill jobs run after migrations land:
 1722: 
 1723: | Job | Trigger | Idempotent | Effect |
 1724: |---|---|---|---|
 1725: | `backfill_controller_style_from_execution_mode` | Manual; run once after migration 1 | Yes | Updates existing `agent_runs` rows with derived `controller_style` based on `execution_mode` |
 1726: | (no backfill for `policy_envelope_snapshot`) | NA | NA | Existing rows keep NULL; readers tolerate NULL |
 1727: | (no backfill for `subaccount_agents` governance fields) | NA | NA | Existing rows use the column defaults |
 1728: 
 1729: ### 6.5 Migration risk
 1730: 
 1731: - **Column adds on `agent_runs`**: this table is hot. Migrations should run during a low-traffic window. Postgres `ALTER TABLE ADD COLUMN ... DEFAULT ...` is metadata-only on Postgres 11+ and does not rewrite the table; verify the version on the target environment before running.
 1732: - **JSONB column add**: nullable, no default; metadata-only.
 1733: - **Partial index creation**: small; the partial index on `controller_style = 'operator'` is built fast because the column is fresh.
 1734: 
 1735: Risk mitigation: run migrations on staging first, time the apply, monitor lock duration on production.

 1949: ## 11. Open Decisions
 1950: 
 1951: These are decisions that need to be made before this spec is locked. Each is small enough that the spec-reviewer or architect can resolve them; they do not need a separate spec.
 1952: 
 1953: ### 11.1 Operator default loop iteration limit
 1954: 
 1955: **Decision needed**: should `CONTROLLER_LIMITS.operator.maxLoopIterations` default to 100, or some other value?
 1956: 
 1957: **Discussion**: 100 is 4x the Native default of 25. This roughly matches the difference between "a structured workflow with well-defined steps" (Native) and "an investigative loop that explores and corrects" (Operator). Higher values (200+) risk runaway loops; lower values (50) might prematurely terminate legitimate Operator work.
 1958: 
 1959: **Recommendation**: 100 as default. Configurable per agent via existing `maxToolCallsPerRun` field. Monitor in production after Spec C (Support Agent MVP) ships.
 1960: 
 1961: **Owner**: Architect.
 1962: 
 1963: ### 11.2 Risk Tier 0 to 2 default to `auto`
 1964: 
 1965: **Decision needed**: is the default mapping (Tier 0-2 â†’ auto, Tier 3-5 â†’ review, Tier 6 â†’ block) the right floor for **every** subaccount, or should it be configurable per subaccount?
 1966: 
 1967: **Discussion**: A high-trust subaccount might want Tier 3 actions to default to `auto`. A high-stakes subaccount might want Tier 2 to default to `review`. The existing `subaccountAgents.requireApprovalAtTier` field (Section 5.2.9) handles this per-agent; do we need per-subaccount?
 1968: 
 1969: **Recommendation**: Per-agent is sufficient for Phase 1. Per-subaccount default can be added in Phase 1.5 if customer demand surfaces.
 1970: 
 1971: **Owner**: Product + Architect.
 1972: 
 1973: ### 11.3 Run Trace pagination default size
 1974: 
 1975: **Decision needed**: default `limit` for `GET /api/agent-runs/:runId/trace` queries.
 1976: 
 1977: **Discussion**: 50 is a reasonable scrollable page. 100 risks showing too much at once for non-technical operators. The UI mostly consumes this for the tree view, which can paginate.
 1978: 
 1979: **Recommendation**: default 50, max 200.
 1980: 
 1981: **Owner**: Architect.
 1982: 
 1983: ### 11.4 Policy Envelope snapshot location: column vs separate table
 1984: 
 1985: **Decision needed**: store as JSONB column on `agent_runs`, or as a separate `policy_envelope_snapshots` table keyed by `agent_run_id`?
 1986: 
 1987: **Discussion**: Column is simpler (single read with the run row), but increases `agent_runs` row size. Separate table is more flexible (could share snapshots across related runs in the future) but adds a join.
 1988: 
 1989: **Recommendation**: JSONB column. Single-read semantic is the right trade for Phase 1; row size impact is bounded (2-5KB per run). Future migration to a separate table is straightforward if scale forces it.
 1990: 
 1991: **Owner**: Architect.

 1998: 
 1999: **Recommendation**: separate CSV, per Section 4.2.6. Linked from this spec.

1. **Schema Scope Contradiction**  
Section: 1.4, INV-5, 5.2.9, 6.1, 9.1  
Description: The spec says schema impact is only two new `agent_runs` columns, but later adds four `subaccount_agents` columns and makes them acceptance criteria.  
Severity: critical  
Suggested fix: Update INV-5/Section 1.4 to include `subaccount_agents` governance columns, or remove the Agent Config governance schema from this spec.

2. **Run Trace Dependency Sequenced Too Early**  
Section: 4.4.14, 8.1  
Description: Phase 1A claims Run Trace API is independent, but Section 4.4 says it is blocked by Policy Envelope, which ships in Phase 1B.  
Severity: critical  
Suggested fix: Move Run Trace API to Phase 1B after Policy Envelope, or split the endpoint so the Phase 1A version does not return `envelope`.

3. **Risk Tier Source-of-Truth Conflict**  
Section: G2, INV-8, 4.2.4, 4.2.8, 4.2.10  
Description: The spec says `gateLevel` is derived from `riskTier`, but also says existing `gateLevel` must win unless explicitly overridden; the proposed `deriveGateLevel(riskTier, policyOverride)` has no input for existing `gateLevel`.  
Severity: critical  
Suggested fix: Define precedence explicitly, e.g. `policyOverride > preservedExistingGateLevel > tierDefault`, and update the function signature and tests accordingly.

4. **Policy Envelope “Single Source of Truth” Overclaims**  
Section: 4.5.2, 4.5.7  
Description: The snapshot is called the single source of truth for what was in force, but runtime enforcement explicitly ignores it and reads live sources.  
Severity: important  
Suggested fix: Reword the claim to “audit/replay source of truth at run start” and add a precedence rule: live policy sources win for enforcement; snapshot wins for replay/audit of run-start state.

5. **Policy Envelope Immutability Has No Enforcement Mechanism**  
Section: INV-9, 4.5.6, 4.5.9  
Description: The spec says `policy_envelope_snapshot` is immutable after run start, but only describes a service write path and no DB trigger, update predicate, DTO omission, or static guard.  
Severity: important  
Suggested fix: Name the enforcement mechanism, such as a DB trigger rejecting updates after non-null, or a service-layer predicate plus route/schema exclusion from all update payloads.

6. **Migration Idempotency Is Claimed But Not Implemented**  
Section: 6.3, 4.1.3, 4.5.3, 5.2.9  
Description: The migrations are described as idempotent on re-apply, but the SQL examples use plain `ADD COLUMN` / `CREATE INDEX` / `DROP COLUMN` without `IF NOT EXISTS` or equivalent guards.  
Severity: important  
Suggested fix: Either change the claim or show guarded SQL for every column/index/down migration.

7. **Feature Flag Language Conflicts With Project Posture**  
Section: 4.4.10, 7.3, 8.3, 8.4  
Description: The spec introduces rollout/emergency feature flags even though project context allows feature flags only for behaviour modes, and these are rollout gates.  
Severity: important  
Suggested fix: Remove `RUN_TRACE_API_V1` and `POLICY_ENVELOPE_SNAPSHOT` flag language and use backward-compatible defaults plus normal rollback.

Severity: critical

Severity: important  
Suggested fix: Add a Run Trace lifecycle contract defining terminal event production from `agent_runs`, valid terminal statuses, and how late events are handled.
