- **Critical** — `docs/superpowers/specs/2026-05-11-operator-session-identity-spec.md:256-258` defines `organisation_id` / `user_id` as `NOT NULL ... ON DELETE RESTRICT` and `subaccount_id ... ON DELETE RESTRICT`, but `:282` says those same FKs “may be nulled (via `ON DELETE SET NULL` where applicable)”. Those two statements cannot both be true. The 7-year-retention path is currently mechanically impossible with the DDL as written.

- **Critical** — The default-subscription source of truth is still unresolved, but later sections treat it as settled. `:664` says Default is either an `is_default` flag or `subaccount_default_connection_id`, `:999-1021` specifies `UPDATE integration_connections SET is_default ...`, and `:1147` explicitly says “**Spec author picks in Chunk 1**.” That is a missing verdict on a load-bearing schema choice; it also creates a sequencing bug because Chunks 5, 7, 10, and the acceptance criteria already depend on the decision.

- **Important** — Disclosure-version-bump execution model is internally contradictory. `:785-788` presents Option A vs Option B and says “**Spec author picks one in implementation**,” but `:941` says “**Spec picks "on-read" detection (§11.4 Option A) for V1**.” One section leaves it open; another marks it decided.

- **Important** — The connect idempotency rule conflicts with the multi-subscription/failover design. Goal `:64` locks “**Default first; other allowed subscriptions in alphabetical order on failure**,” but `:996` says connect should return the existing row whenever a usable `(organisation_id, subaccount_id, provider_type = 'openai')` row already exists. That predicate prevents multiple simultaneously usable OpenAI operator-session rows in one subaccount, which breaks both failover and “Make default”.

- **Important** — The spec creates two different connection↔consent links with no declared winner. `:218-231` adds `integration_connections.consent_record_id`; `:259` adds `operator_session_consents.connection_id`; `:657-665` declares source-of-truth precedence for several facts but does not say which FK is canonical if they diverge. This is schema overlap with no source-of-truth rule.

- **Important** — Migration paths are wrong for this repo. The file inventory uses `server/db/migrations/0318_operator_session_columns.sql` and `server/db/migrations/0319_operator_session_consents.sql` at `:404-405`, but the workspace has a project-root `migrations/` directory and no `server/db/migrations` directory. This is a concrete path/numbering accuracy issue.

- **Important** — The refresh-job dedup mechanism is both unbacked and internally inconsistent. `:774` names a unique constraint on `(connection_id, refresh_attempt_at_bucket)`, while `:998` and `:1054` switch to `(connection_id, refresh_bucket)`. No migration, schema file, or pg-boss mechanism is named for where that unique constraint lives. This is a load-bearing idempotency claim without a backing mechanism.

- **Important** — Route/permission semantics overlap for the same mutation surface. `:735` says `PATCH /api/subaccounts/:id/operator-session-connections/:connId` uses `operator_session.connect` “**for availability/default changes**,” but `:740` also introduces `PATCH .../allow-agent-use` guarded by `operator_session.allow_agent_use`. “Availability” is the allow-agent-use capability, so the spec currently assigns two different guards and two different route shapes to the same class of change.

- **Important** — The `/connections` CRUD consolidation has backend file-inventory drift. `:140` says new Govern-surface Add/Edit/Test routes ship under `/api/subaccounts/:subaccountId/`; `:900` says this is either `server/routes/connections.ts` or a new `webLoginConnectionsGovern.ts`; but `§8` only locks `server/routes/connections.ts` for operator-session mounting/list extension (`:461`) and never locks the actual migrated web-login Add/Edit/Test route implementation. The inventory is not complete for the consolidation scope.

- **Minor** — The chunk plan references a client file that is missing from the file inventory lock. `:873` says `03-disclosure-plus.html` is implemented as `DisclosurePlusStep.tsx` inside the wizard, but `§8.12` does not list `DisclosurePlusStep.tsx`. That is straight file-inventory drift against the spec’s own “single source of truth” rule.

- **Minor** — There is stale framing language around provider gating. Goal `:54` says the flow is “**blocked-behind-provider flag** if not verified at time of build,” but `:765`, `:872`, and `:1151` consistently describe a `501 provider_mechanism_not_verified` gate, not a feature flag. Given the repo posture against rollout flags, “flag” is stale/misleading wording.

- **Minor** — Checklist compliance is incomplete on terminal-event semantics. `:1035-1037` names terminal paths for the refresh chain, but it never declares the terminal event `status` field (`success | partial | failed`) required by `docs/spec-authoring-checklist.md §10.4`, and `§16.5` does not define a `status: 'partial'` terminal event for partial-completion cases.
