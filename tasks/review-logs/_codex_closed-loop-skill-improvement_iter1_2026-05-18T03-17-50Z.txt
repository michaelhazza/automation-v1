## 1. Summary

This spec defines Phase 1 of the closed-loop skill improvement system: a human-gated amendment proposal loop that turns scorecard failures into reviewed, typed behavioural overlays on inherited skills. When a scorecard verdict fails, a new `failure_post_mortem` pg-boss job synthesises a root-cause record, drafts a typed amendment, submits it for peer review (GPT-class via OpenAI), and â€” if the peer reviewer confirms it addresses the root cause â€” queues it in the existing morning review surface (Inbox / ReviewQueuePage) for one-click operator approval. Accepted amendments are composed deterministically on top of the inherited skill base without forking it. Rejected amendments become regression test cases. The loop is bounded (5 proposals/skill/week, 20 active/skill lifetime), schema-validated (5 typed amendment kinds with per-kind length ceilings), anti-recursive (evaluator surfaces are resolved through a separate code path that never consults `skill_amendments`), and gated by mandatory human approval.

The framing is: agents propose, humans approve. Nothing in this system activates behaviour autonomously.

## 6. Governance Invariants

These invariants bind every implementation decision and are verified in Â§23. They are taken verbatim from the dev brief and are non-negotiable.

1. **Human approval mandatory.** No amendment activates without an explicit operator accept action. No automated activation, confidence threshold bypass, or silent activation path.
2. **Anti-recursion.** Amendments compose only into agent runtime skill bodies. Scorecard judge prompts, RCA proposer prompts, and peer-review prompts are resolved through a separate code path that never consults `skill_amendments`. Schema validation rejects any `proposed_remedy_body` whose declared target is an evaluator surface.
3. **Amendments are not memory.** Amendments are skill-scoped behaviour overlays; memory is per-entity recall. The two primitives are distinct; code, audit events, and UI copy must not conflate them.
4. **No hidden composition.** Every active amendment affecting runtime behaviour is discoverable from operator-visible surfaces. No invisible runtime-only overlays.
5. **Tenant isolation.** No amendment, RCA record, replay artefact, or proposer context may incorporate behavioural signals from another organisation or subaccount.
6. **Resolver purity.** Resolver output is a pure function of: system-skill snapshot, amendment snapshot set, resolver version, and explicit runtime inputs. No wall-clock time, mutable external services, or live model calls inside the resolver.
7. **Fail-closed truncation.** If composition would exceed 12,000 chars total, the resolver returns the system base text alone and emits an alert. Silent truncation is forbidden.
8. **Retirement is non-destructive.** Retired amendment rows persist with full provenance; the regression linkage survives retirement.
9. **Operator trust posture.** UI copy uses "Proposed amendment from a failed run" not "Fix found"; "Apply" not "Approve"; "Why this was proposed" not "Why this is correct." Multi-paragraph AI-generated rationales are forbidden in the queue UI.

 144: ## 6. Governance Invariants
 145: 
 146: These invariants bind every implementation decision and are verified in Â§23. They are taken verbatim from the dev brief and are non-negotiable.
 147: 
 148: 1. **Human approval mandatory.** No amendment activates without an explicit operator accept action. No automated activation, confidence threshold bypass, or silent activation path.
 149: 2. **Anti-recursion.** Amendments compose only into agent runtime skill bodies. Scorecard judge prompts, RCA proposer prompts, and peer-review prompts are resolved through a separate code path that never consults `skill_amendments`. Schema validation rejects any `proposed_remedy_body` whose declared target is an evaluator surface.
 150: 3. **Amendments are not memory.** Amendments are skill-scoped behaviour overlays; memory is per-entity recall. The two primitives are distinct; code, audit events, and UI copy must not conflate them.
 151: 4. **No hidden composition.** Every active amendment affecting runtime behaviour is discoverable from operator-visible surfaces. No invisible runtime-only overlays.
 152: 5. **Tenant isolation.** No amendment, RCA record, replay artefact, or proposer context may incorporate behavioural signals from another organisation or subaccount.
 153: 6. **Resolver purity.** Resolver output is a pure function of: system-skill snapshot, amendment snapshot set, resolver version, and explicit runtime inputs. No wall-clock time, mutable external services, or live model calls inside the resolver.
 154: 7. **Fail-closed truncation.** If composition would exceed 12,000 chars total, the resolver returns the system base text alone and emits an alert. Silent truncation is forbidden.
 155: 8. **Retirement is non-destructive.** Retired amendment rows persist with full provenance; the regression linkage survives retirement.
 156: 9. **Operator trust posture.** UI copy uses "Proposed amendment from a failed run" not "Fix found"; "Apply" not "Approve"; "Why this was proposed" not "Why this is correct." Multi-paragraph AI-generated rationales are forbidden in the queue UI.

 347: ## 8. Resolver Changes
 348: 
 349: **File:** `server/services/skillService.ts` â€” function `resolveSkillsForAgent()` (~line 115).
 350: 
 351: ### 8.1 Amendment composition step
 352: 
 353: After the existing precedence resolution (subaccount > org > system picks the base skill row), the resolver checks whether the resolved skill is inherited (i.e. `system_skill_id` or `org_skill_id` is set â€” not a custom subaccount skill). If inherited, it runs the amendment composition step:
 354: 
 355: 1. Query `skill_amendments` WHERE `(system_skill_id = $base OR org_skill_id = $base)` AND `org_id = $org` AND `subaccount_id = $subaccount` AND `status = 'accepted'` â€” via `getOrgScopedDb`.
 356: 2. Apply composition ordering (Â§3.6 of the brief, replicated here for self-containment):
 357:    - Bucket order: org guardrails â†’ org instruction_extensions â†’ org examples â†’ org context_facts â†’ org exceptions â†’ subaccount guardrails â†’ subaccount instruction_extensions â†’ subaccount examples â†’ subaccount context_facts â†’ subaccount exceptions.
 358:    - Within a bucket: ascending `activated_at`, then ascending `id` as tiebreak.
 359: 3. Check freeze: if an active `skill_amendment_freezes` row exists for this skill/subaccount/org with `freeze_type = 'amendment_activation'`, return system base text only.
 360: 4. Concatenate base text + amendment bodies in bucket order. If total chars > 12,000: fail-closed â€” return base text alone, emit `composition.degraded` alert, set `review_required` flag on the skill for this scope.
 361: 5. Write a `skill_amendment_run_snapshot` row recording the resolver version, amendment IDs used/excluded, composed body, and hash.
 362: 6. Return composed body.
 363: 
 364: Custom subaccount skills (where the resolved row has `subaccount_id` set in the `skills` table) skip the amendment step entirely.
 365: 
 366: ### 8.2 Anti-recursion code path
 367: 
 368: Scorecard judge prompts, RCA proposer prompts, and peer-review prompts are resolved via a separate function `resolveSkillForEvaluator(slug)` that reads from `system_skills` or `skills` directly, never calls `resolveSkillsForAgent`, and never consults `skill_amendments`. This structural separation is the Â§6 anti-recursion invariant enforcement.
 369: 
 370: ### 8.3 Resolver versioning
 371: 
 372: The resolver version is a semver constant in `server/services/skillService.ts` (e.g. `RESOLVER_VERSION = '1.0.0'`). Increment the patch version on any composition-logic change; increment minor on new amendment kind support; increment major on ordering-invariant changes. The version string is written to `skill_amendment_run_snapshot.resolver_version` on every run.
 373: 
 374: ### 8.4 Cache invalidation
 375: 
 376: Resolver output is cached per `(system_skill_id, subaccount_id, amendment_version_set_hash)`. Any status change on a `skill_amendments` row (accept, reject, retire, rollback) invalidates the cache entry by computing a new hash. Cache is in-process (Map keyed on the triple); not persisted. In-flight runs keep their resolved body for the run lifetime (resolution happens once at agent boot).

 384: ### 9.1 `failure_post_mortem`
 385: 
 386: **Queue name:** `failure:post-mortem`
 387: **TeamSize:** 2
 388: **Trigger:** Subordinate pg-boss `send('failure:post-mortem', payload)` dispatched by `scorecardJudgeJob` inside the same transaction that writes the verdict row, on `verdict = 'fail'` only.
 389: **Payload:** `{ scorecard_judgement_id, run_id, org_id, subaccount_id, skill_slug }`
 390: 
 391: **Job steps:**
 392: 
 393: 1. **Freeze check.** Query `skill_amendment_freezes` for an active row with `freeze_type = 'proposal_generation'` scoped to this skill, subaccount, or org. If found, log `proposal_suppressed: freeze_active` and exit.
 394: 2. **Cap check.** Count `skill_amendments` WHERE `org_id`, `subaccount_id`, `system_skill_id|org_skill_id` match AND `status = 'accepted'`. If â‰¥ 20, set skill to `review_required`, log, and exit without drafting.
 395: 3. **Weekly cap check.** Count amendments created in the last 7 days for this (org, subaccount, skill). If â‰¥ 5, drop and log.
 396: 4. **Inherited-skill detection.** Read the `skill_amendment_run_snapshot` for this run's skill. Determine which FK to set: `system_skill_id` if origin is `system_skills`, `org_skill_id` if origin is `skills` with `org_id` set, abort if custom subaccount skill (log `custom_skill_not_amendable`).
 397: 5. **Context assembly.** Gather the 6 inputs (failed run transcript, rubric snapshot from verdict row, failed check reasoning, entity record, recent operator corrections on this skill+subaccount, current amendment stack on this skill+subaccount). No cross-tenant data.
 398: 6. **RCA synthesis.** Call Claude Opus (frontier-class) with the assembled context. Schema-validate the output: `failure_mode` (string), `contributing_factors` (list â‰¤ 5, each references an input field), `proposed_remedy_kind` (one of 5 kinds or `no_remedy_proposed`), `proposed_remedy_body` (text, within kind ceiling), `confidence` (0.0â€“1.0). If `no_remedy_proposed`, exit cleanly.
 399: 7. **Anti-recursion check.** Reject any proposal whose `proposed_remedy_kind` or `proposed_remedy_body` targets evaluator surfaces (scorecard judge prompts, RCA proposer prompts, peer-review prompts). Log and exit if rejected.
 400: 8. **`context_fact` declarative-only check.** Reject bodies containing imperative modals (`must`, `should`, `never`, `always`, `do`, `do not`). Discard silently; log.
 401: 9. **Deduplication.** Hash `(skill_id, kind, normalised_body)`. Check against active accepted rows (suppress + increment `suppressed_duplicate_count`), pending rows (increment `occurrence_count`), and recently-rejected rows within 14-day freshness window (suppress unless â‰¥ 3 distinct failing runs in 7 days).
 402: 10. **Peer review.** Call GPT-class model via OpenAI API. Input: proposed amendment + RCA context. Output: `{ addresses_root_cause: boolean, reasoning: string }`. If `false`: write to `peer_reviewer_drops`, exit. If `true`: proceed.
 403: 11. **Write amendment row.** Insert into `skill_amendments` with `status = 'draft'`, then immediately transition to `status = 'pending_review'` via a second update (preserving the draftâ†’pending_review state transition for audit). Populate all provenance columns (Â§7.1).
 404: 12. **Write regression case.** Insert into `skill_regression_cases` with `tag = 'unresolved'`, `amendment_id` set.
 405: 
 406: **Idempotency:** keyed on `scorecard_judgement_id` â€” unique constraint `skill_amendments(scorecard_judgement_id)` where `status != 'retired'` prevents duplicate proposals for the same verdict.

 723: ## 18. Execution-Safety Contracts
 724: 
 725: ### 18.1 Idempotency posture
 726: 
 727: | Operation | Posture | Mechanism |
 728: |---|---|---|
 729: | `failure_post_mortem` job execution | key-based | Unique constraint on `skill_amendments(scorecard_judgement_id)` WHERE status != 'retired'. Duplicate dispatch â†’ `23505` caught â†’ no-op. |
 730: | Amendment accept | state-based | `UPDATE skill_amendments SET status='accepted' WHERE id=$id AND status='pending_review'`. 0 rows updated = already actioned â†’ 409. |
 731: | Amendment reject | state-based | Same predicate, `status='pending_review'`. |
 732: | Amendment retire | state-based | `UPDATE ... WHERE status='accepted'`. 0 rows = already retired â†’ 409. |
 733: | Regression replay job | key-based | One replay job per `amendment_id`; pg-boss key deduplication. |
 734: | Freeze create | key-based | Unique partial index on `(org_id, scope, scope_id, freeze_type)` WHERE `thawed_at IS NULL`. Duplicate active freeze â†’ 409. |
 735: 
 736: ### 18.2 Retry classification
 737: 
 738: | Operation | Classification | Notes |
 739: |---|---|---|
 740: | `failure_post_mortem` job | safe | Idempotency key prevents duplicate amendment rows on retry |
 741: | OpenAI peer review call | guarded | Retried up to 3 times with exponential backoff; idempotency key from `scorecard_judgement_id` passed as OpenAI request header |
 742: | Claude Opus RCA synthesis | guarded | Same retry budget; job-level idempotency prevents double insert |
 743: | Amendment accept HTTP | unsafe | Caller must not retry without checking current status first |
 744: | Resolver snapshot write | safe | Write is idempotent: `ON CONFLICT (run_id, system_skill_id, org_skill_id) DO NOTHING` |
 745: 
 746: ### 18.3 Concurrency guards for racing writes
 747: 
 748: - **Double-accept race:** Two concurrent accept requests for the same amendment â†’ one wins (state-based predicate returns 1 row), other gets 0 rows â†’ service returns 409. First caller's audit event is canonical.
 749: - **Accept during regression replay:** Regression replay job checks `status = 'accepted'` before running; if a parallel rollback fired and status is already `retired`, the replay job exits cleanly.
 750: - **Double freeze race:** Unique partial index prevents two concurrent freeze creates for the same scope/type. Loser gets `23505` â†’ 409.
 751: 
 752: ### 18.4 Terminal event guarantee
 753: 
 754: The `failure_post_mortem` job emits exactly one terminal log event per execution:
 755: - `amendment.proposed` â€” amendment row written to `pending_review`
 756: - `amendment.suppressed` â€” dedup match against active accepted or pending row
 757: - `amendment.dropped.no_remedy` â€” proposer returned `no_remedy_proposed`
 758: - `amendment.dropped.peer_review` â€” peer reviewer returned `false`
 759: - `amendment.dropped.schema_invalid` â€” schema validation failed
 760: - `amendment.dropped.cap_exceeded` â€” weekly or lifetime cap hit
 761: - `amendment.dropped.freeze_active` â€” freeze gate blocked
 762: - `amendment.dropped.custom_skill` â€” custom subaccount skill excluded
 763: 
 764: Post-terminal prohibition: no further events with the same `scorecard_judgement_id` after a terminal event from the same job run.
 765: 
 766: ### 18.5 No-silent-partial-success
 767: 
 768: The `failure_post_mortem` job either completes fully (amendment in `pending_review`) or emits a `dropped` terminal event. There is no partial success path â€” if any step fails, the job retries from the start (idempotency key prevents double-insert).
 769: 
 770: ### 18.6 State machine closure
 771: 
 772: **`skill_amendments.status` valid transitions:**
 773: 
 774: | From | To | Trigger |
 775: |---|---|---|
 776: | `draft` | `pending_review` | Proposer job transitions immediately after insert |
 777: | `pending_review` | `accepted` | Operator accept action |
 778: | `pending_review` | `rejected` | Operator reject action |
 779: | `pending_review` | `retired` (`stale`) | Freshness-window job (14-day expiry) |
 780: | `accepted` | `retired` | Operator retire or rollback; or regression replay auto-reject |
 781: 
 782: Forbidden transitions: `rejected` â†’ any; `retired` â†’ any; `accepted` â†’ `rejected` (must go through `retire` with `rollback` reason); `draft` â†’ `accepted` (must pass through `pending_review`).
 783: 
 784: The status enum is closed. Adding a new status value requires a spec amendment.

 858: ## 20. Failure Atomicity Definitions
 859: 
 860: ### Amendment acceptance
 861: 
 862: Sub-steps: DB write (`status = 'accepted'`, `activated_at`) â†’ regression-set tag (`fix_proposed`) â†’ cache invalidation â†’ audit event emission â†’ regression replay dispatch.
 863: 
 864: Atomicity boundary: DB write + regression-set tag are in a single `withOrgTx` transaction. Cache invalidation is synchronous after the transaction commits. Audit event is fire-and-forget via `tryEmitAgentEvent` (best-effort; failure does not roll back the accept). Regression replay dispatch is pg-boss `send` inside the transaction â€” rolls back with the transaction if the transaction fails.
 865: 
 866: If regression replay later fails: the amendment stays `accepted`; the job retries. If the replay job determines a regression: it transitions the amendment to `retired` (`retirement_reason = 'rollback'`) in a new transaction â€” the original accept transaction is not affected.
 867: 
 868: ### Amendment retirement (including rollback)
 869: 
 870: Sub-steps: DB write (`status = 'retired'`, `retired_at`, `retirement_reason`) â†’ cache invalidation â†’ incident alert emission (rollback only) â†’ audit event.
 871: 
 872: Atomicity: DB write in one transaction. Cache invalidation synchronous after commit. Alert and audit event are fire-and-forget. If alert fails: amendment is still retired; operator sees it in the skill detail page.
 873: 
 874: ### Rollback (urgent retirement)
 875: 
 876: Same as retirement but additionally: emits an operational incident event (separate from the morning queue), surfaces at tier 1 in the queue, captures audit trail (operator, telemetry signal, affected runs). Incident emission is fire-and-forget â€” if it fails, the rollback DB write has already committed.
 877: 
 878: ### Regression replay
 879: 
 880: Sub-steps: Load regression cases â†’ replay each case via bench primitives â†’ score â†’ check for regressions â†’ (if regression) auto-retire amendment â†’ write effectiveness sidecar.
 881: 
 882: If any replay sub-step fails: the job retries from the top (idempotent â€” `ON CONFLICT DO NOTHING` on sidecar writes; amendment status check prevents double-retire). If the auto-retire sub-step fails: the amendment stays `accepted`; the job retries; an alert is emitted.
 883: 
 884: ### Freshness-window auto-retirement
 885: 
 886: Sub-steps: Query `pending_review` rows older than 14 days â†’ batch update to `status = 'retired', retirement_reason = 'stale'` â†’ write `amendment_proposer_metrics` update.
 887: 
 888: Atomic per row. If the job crashes mid-batch: already-retired rows have `retirement_reason = 'stale'`; unprocessed rows are retried on the next daily run. No duplicate retirements possible (state-based predicate: `WHERE status = 'pending_review' AND created_at < now() - interval '14 days'`).

 349: **File:** `server/services/skillService.ts` â€” function `resolveSkillsForAgent()` (~line 115).
 350: 
 351: ### 8.1 Amendment composition step
 352: 
 353: After the existing precedence resolution (subaccount > org > system picks the base skill row), the resolver checks whether the resolved skill is inherited (i.e. `system_skill_id` or `org_skill_id` is set â€” not a custom subaccount skill). If inherited, it runs the amendment composition step:
 354: 
 355: 1. Query `skill_amendments` WHERE `(system_skill_id = $base OR org_skill_id = $base)` AND `org_id = $org` AND `subaccount_id = $subaccount` AND `status = 'accepted'` â€” via `getOrgScopedDb`.
 356: 2. Apply composition ordering (Â§3.6 of the brief, replicated here for self-containment):
 357:    - Bucket order: org guardrails â†’ org instruction_extensions â†’ org examples â†’ org context_facts â†’ org exceptions â†’ subaccount guardrails â†’ subaccount instruction_extensions â†’ subaccount examples â†’ subaccount context_facts â†’ subaccount exceptions.
 358:    - Within a bucket: ascending `activated_at`, then ascending `id` as tiebreak.
 359: 3. Check freeze: if an active `skill_amendment_freezes` row exists for this skill/subaccount/org with `freeze_type = 'amendment_activation'`, return system base text only.
 360: 4. Concatenate base text + amendment bodies in bucket order. If total chars > 12,000: fail-closed â€” return base text alone, emit `composition.degraded` alert, set `review_required` flag on the skill for this scope.
 361: 5. Write a `skill_amendment_run_snapshot` row recording the resolver version, amendment IDs used/excluded, composed body, and hash.
 362: 6. Return composed body.
 363: 
 364: Custom subaccount skills (where the resolved row has `subaccount_id` set in the `skills` table) skip the amendment step entirely.
 365: 
 366: ### 8.2 Anti-recursion code path
 367: 
 368: Scorecard judge prompts, RCA proposer prompts, and peer-review prompts are resolved via a separate function `resolveSkillForEvaluator(slug)` that reads from `system_skills` or `skills` directly, never calls `resolveSkillsForAgent`, and never consults `skill_amendments`. This structural separation is the Â§6 anti-recursion invariant enforcement.
 369: 
 370: ### 8.3 Resolver versioning
 371: 
 372: The resolver version is a semver constant in `server/services/skillService.ts` (e.g. `RESOLVER_VERSION = '1.0.0'`). Increment the patch version on any composition-logic change; increment minor on new amendment kind support; increment major on ordering-invariant changes. The version string is written to `skill_amendment_run_snapshot.resolver_version` on every run.
 373: 
 374: ### 8.4 Cache invalidation
 375: 
 376: Resolver output is cached per `(system_skill_id, subaccount_id, amendment_version_set_hash)`. Any status change on a `skill_amendments` row (accept, reject, retire, rollback) invalidates the cache entry by computing a new hash. Cache is in-process (Map keyed on the triple); not persisted. In-flight runs keep their resolved body for the run lifetime (resolution happens once at agent boot).

 381: 
 382: Registered in `server/services/queueService/maintenanceJobs/pgBossRegistrations.ts` alongside existing jobs.
 383: 
 384: ### 9.1 `failure_post_mortem`
 385: 
 386: **Queue name:** `failure:post-mortem`
 387: **TeamSize:** 2
 388: **Trigger:** Subordinate pg-boss `send('failure:post-mortem', payload)` dispatched by `scorecardJudgeJob` inside the same transaction that writes the verdict row, on `verdict = 'fail'` only.
 389: **Payload:** `{ scorecard_judgement_id, run_id, org_id, subaccount_id, skill_slug }`
 390: 
 391: **Job steps:**
 392: 
 393: 1. **Freeze check.** Query `skill_amendment_freezes` for an active row with `freeze_type = 'proposal_generation'` scoped to this skill, subaccount, or org. If found, log `proposal_suppressed: freeze_active` and exit.
 394: 2. **Cap check.** Count `skill_amendments` WHERE `org_id`, `subaccount_id`, `system_skill_id|org_skill_id` match AND `status = 'accepted'`. If â‰¥ 20, set skill to `review_required`, log, and exit without drafting.
 395: 3. **Weekly cap check.** Count amendments created in the last 7 days for this (org, subaccount, skill). If â‰¥ 5, drop and log.
 396: 4. **Inherited-skill detection.** Read the `skill_amendment_run_snapshot` for this run's skill. Determine which FK to set: `system_skill_id` if origin is `system_skills`, `org_skill_id` if origin is `skills` with `org_id` set, abort if custom subaccount skill (log `custom_skill_not_amendable`).
 397: 5. **Context assembly.** Gather the 6 inputs (failed run transcript, rubric snapshot from verdict row, failed check reasoning, entity record, recent operator corrections on this skill+subaccount, current amendment stack on this skill+subaccount). No cross-tenant data.
 398: 6. **RCA synthesis.** Call Claude Opus (frontier-class) with the assembled context. Schema-validate the output: `failure_mode` (string), `contributing_factors` (list â‰¤ 5, each references an input field), `proposed_remedy_kind` (one of 5 kinds or `no_remedy_proposed`), `proposed_remedy_body` (text, within kind ceiling), `confidence` (0.0â€“1.0). If `no_remedy_proposed`, exit cleanly.
 399: 7. **Anti-recursion check.** Reject any proposal whose `proposed_remedy_kind` or `proposed_remedy_body` targets evaluator surfaces (scorecard judge prompts, RCA proposer prompts, peer-review prompts). Log and exit if rejected.
 400: 8. **`context_fact` declarative-only check.** Reject bodies containing imperative modals (`must`, `should`, `never`, `always`, `do`, `do not`). Discard silently; log.
 401: 9. **Deduplication.** Hash `(skill_id, kind, normalised_body)`. Check against active accepted rows (suppress + increment `suppressed_duplicate_count`), pending rows (increment `occurrence_count`), and recently-rejected rows within 14-day freshness window (suppress unless â‰¥ 3 distinct failing runs in 7 days).
 402: 10. **Peer review.** Call GPT-class model via OpenAI API. Input: proposed amendment + RCA context. Output: `{ addresses_root_cause: boolean, reasoning: string }`. If `false`: write to `peer_reviewer_drops`, exit. If `true`: proceed.
 403: 11. **Write amendment row.** Insert into `skill_amendments` with `status = 'draft'`, then immediately transition to `status = 'pending_review'` via a second update (preserving the draftâ†’pending_review state transition for audit). Populate all provenance columns (Â§7.1).
 404: 12. **Write regression case.** Insert into `skill_regression_cases` with `tag = 'unresolved'`, `amendment_id` set.
 405: 
 406: **Idempotency:** keyed on `scorecard_judgement_id` â€” unique constraint `skill_amendments(scorecard_judgement_id)` where `status != 'retired'` prevents duplicate proposals for the same verdict.

 408: ### 9.2 Regression Replay Job
 409: 
 410: **Queue name:** `amendment:regression-replay`
 411: **TeamSize:** 1
 412: **Trigger:** Dispatched by `skillAmendmentService.accept()` after writing the amendment row to `accepted`.
 413: **Payload:** `{ amendment_id, org_id, subaccount_id, system_skill_id, org_skill_id }`
 414: 
 415: **Job steps:**
 416: 
 417: 1. Load `skill_regression_cases` for this skill+subaccount WHERE `tag IN ('fix_proposed', 'fix_wrong')`.
 418: 2. For each case, replay the original scorecard verdict using the `benchExecuteJob` replay primitives against the current composed skill body (includes the newly accepted amendment).
 419: 3. If any regression case flips from `pass` to `fail` under the new composition: reject the just-accepted amendment automatically (status â†’ `rejected`, `reject_reason = 'regression_failure'`), emit alert, write to `amendment_proposer_metrics`.
 420: 4. Write `skill_amendment_effectiveness` row with initial metrics.
 421: 5. Record `replay_judge_version`, `replay_resolver_version`, `replay_model_version`, `replay_timestamp` on each replay verdict.

 725: ### 18.1 Idempotency posture
 726: 
 727: | Operation | Posture | Mechanism |
 728: |---|---|---|
 729: | `failure_post_mortem` job execution | key-based | Unique constraint on `skill_amendments(scorecard_judgement_id)` WHERE status != 'retired'. Duplicate dispatch â†’ `23505` caught â†’ no-op. |

 731: | Amendment reject | state-based | Same predicate, `status='pending_review'`. |

 736: ### 18.2 Retry classification
 737: 
 738: | Operation | Classification | Notes |
 739: |---|---|---|
 740: | `failure_post_mortem` job | safe | Idempotency key prevents duplicate amendment rows on retry |
 741: | OpenAI peer review call | guarded | Retried up to 3 times with exponential backoff; idempotency key from `scorecard_judgement_id` passed as OpenAI request header |
 742: | Claude Opus RCA synthesis | guarded | Same retry budget; job-level idempotency prevents double insert |
 743: | Amendment accept HTTP | unsafe | Caller must not retry without checking current status first |
 744: | Resolver snapshot write | safe | Write is idempotent: `ON CONFLICT (run_id, system_skill_id, org_skill_id) DO NOTHING` |
 745: 
 746: ### 18.3 Concurrency guards for racing writes
 747: 
 748: - **Double-accept race:** Two concurrent accept requests for the same amendment â†’ one wins (state-based predicate returns 1 row), other gets 0 rows â†’ service returns 409. First caller's audit event is canonical.
 749: - **Accept during regression replay:** Regression replay job checks `status = 'accepted'` before running; if a parallel rollback fired and status is already `retired`, the replay job exits cleanly.
 750: - **Double freeze race:** Unique partial index prevents two concurrent freeze creates for the same scope/type. Loser gets `23505` â†’ 409.

  75: This spec defines Phase 1 of the closed-loop skill improvement system: a human-gated amendment proposal loop that turns scorecard failures into reviewed, typed behavioural overlays on inherited skills. When a scorecard verdict fails, a new `failure_post_mortem` pg-boss job synthesises a root-cause record, drafts a typed amendment, submits it for peer review (GPT-class via OpenAI), and — if the peer reviewer confirms it addresses the root cause — queues it in the existing morning review surface (Inbox / ReviewQueuePage) for one-click operator approval.
