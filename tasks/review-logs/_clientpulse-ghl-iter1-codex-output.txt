1. Section: `§§1 Executive summary & Kel requirements matrix, 3 Multi-agency-per-org blocker, 7 Dashboard UI gaps, 9 Canonical taxonomy proposal, 10 Recommended build order`  
Title: `Multi-agency rollback incomplete`  
Problem: Early sections still treat “two GHL backends in one org” as a blocker requiring schema work, but `§3` reverses that and says one GHL backend = one Synthetos org. The drift leaks into `§9.5` (`connectorInstanceLabel`) and `§7.5` (`All | SaaS only | DFY only`), so implementers cannot tell whether to build per-org or per-connector scoping.  
Suggested fix: Make `§3` authoritative. Remove the blocker from `§1.1/§1.2`, delete `connector_configs.connectorInstanceLabel` from `§9.5`, and replace the `§7.5` scope selector with org-level config only; if per-connector overrides remain a future platform need, mark them explicitly out of scope for Kel/V1.  
Severity: `critical`

2. Section: `§§2 GHL data ingestion gaps, 9 Canonical taxonomy proposal, 10 Recommended build order`  
Title: `Nonexistent APIs return later`  
Problem: `§2.0b/§2.0c` correctly say GHL has no login-activity or installed-apps endpoint and propose Staff Activity Pulse plus fingerprint scanning, but `§2.2`, `§9.1`, and Phase 1 regress to `fetchLoginActivity`, `fetchInstalledIntegrations`, `ghl.read.login_activity`, `ghl.read.installed_integrations`, and `subaccount_external_integrations`. That is a direct contradiction and would send implementation toward APIs the spec already ruled out.  
Suggested fix: Delete every login/install-app endpoint reference and make the canonical V1 path explicit: adapter populates mutation/fingerprint-bearing tables, then `compute_staff_activity_pulse` and `scan_integration_fingerprints` derive the signals. Rename all capability/skill/migration entries to the same vocabulary.  
Severity: `critical`

3. Section: `§§4 Health score execution pipeline gaps, 5 Churn risk execution pipeline gaps, 12 Current state of the GHL Agency seed`  
Title: `Current-state inventory is stale`  
Problem: The spec says there is no health snapshot table, no `compute_health_score` executor, and no `compute_churn_risk` implementation, but the repo already has `health_snapshots` (`server/db/schema/canonicalEntities.ts`, migration `0044`), `compute_health_score` and `compute_churn_risk` handlers (`server/services/skillExecutor.ts` and `server/services/intelligenceSkillExecutor.ts`), plus `anomaly_events` and `canonical_metrics`. That means the gap analysis is mixing “missing in code” with “present but not wired/scheduled.”  
Suggested fix: Rewrite `§§4–5` to distinguish existing substrate from missing wiring. Either reuse `health_snapshots`/`anomaly_events` as the source of truth, or explicitly justify replacing them and give a migration plan; do not describe already-shipped tables/handlers as absent.  
Severity: `critical`

4. Section: `§§4 Health score execution pipeline gaps, 12 Current state of the GHL Agency seed`  
Title: `Heartbeat claim contradicts code`  
Problem: `§4.2` says migration 0068 does not enable the Portfolio Health Agent heartbeat, but `§12.1` and `migrations/0068_portfolio_health_agent_seed.sql` show `heartbeat_enabled = true` and `heartbeat_interval_hours = 4`. That makes one of the load-bearing blockers factually wrong.  
Suggested fix: Remove that blocker and replace it with the real missing mechanism, if any, such as how system agents are instantiated/scheduled for entitled orgs or how scans fan out from polling into scoring.  
Severity: `important`

5. Section: `§§6 Intervention workflow gaps, 10 Recommended build order, 11 Cross-reference to existing specs + open questions, 21 V1 vs V2 scope delineation`  
Title: `Auto-proposer scope is unresolved`  
Problem: `§6` and Phase 4 define an automatic proposer job that creates review items after each churn assessment, but `§21.1` says V1 is “manual proposer only” and “no auto-proposer.” Those are materially different products, affecting jobs, quotas, review-queue volume, and ship criteria.  
Suggested fix: Record a single V1 verdict. If V1 is manual-only, remove `proposeClientPulseInterventionsJob` from `§6/§10`; if V1 includes auto-proposal, move “auto-proposer” out of `§21.2` and define the exact trigger/cooldown/quota behavior as in-scope.  
Severity: `critical`

6. Section: `§§6 Intervention workflow gaps, 10 Recommended build order, 15 Action-type primitives`  
Title: `Intervention source of truth duplicates`  
Problem: `§6` says `actions` + `reviewItems` + `interventionOutcomes` are already the right substrate, but Phase 4 introduces a new `client_pulse_interventions` table that appears to store the same proposal/payload/status/outcome lifecycle. Without a source-of-truth statement, implementers will duplicate state and drift the queue from execution records.  
Suggested fix: Choose one authoritative record. The cleaner path is to keep `actions` as the intervention record, keep `reviewItems` as the approval projection, and store any extra ClientPulse metadata in action payload/versioned metadata rather than adding `client_pulse_interventions`.  
Severity: `critical`

7. Section: `§§10 Recommended build order, 15 Action-type primitives, 21 V1 vs V2 scope delineation`  
Title: `Primitive slugs collide with existing actions`  
Problem: The spec repurposes `send_email` and `create_task` as CRM-dispatched intervention primitives, but those slugs already exist with different schemas and semantics in `server/config/actionRegistry.ts`: `send_email` is generic provider email (`to/subject/body/provider`) and `create_task` creates an internal board task. Reusing the same names would break existing callers or require a silent contract change.  
Suggested fix: Do not overload existing action types. Either keep `trigger_account_intervention` as the top-level action and put `primitive: 'fire_automation' | 'send_email' | ...` inside its payload, or define distinct CRM-specific slugs with new schemas; in either case, leave existing global `send_email`/`create_task` behavior unchanged.  
Severity: `critical`

8. Section: `§§2 OAuth scope audit, 10 Recommended build order, 17 Configuration Agent integration`  
Title: `Scope source of truth drifts`  
Problem: `§2.3` treats `server/config/oauthProviders.ts` as the OAuth scope source of truth, but `server/routes/ghl.ts` hardcodes a different scope list for the actual onboarding redirect. If implementers update only one of those places, the spec’s “extend scopes before next connect flow” instruction will still leave the live flow wrong.  
Suggested fix: Name one authority for GHL scopes and make the other consume it. The concrete change should be “`/api/ghl/oauth-url` builds scopes from `OAUTH_PROVIDERS.ghl.scopes`,” then list the required additions once in the spec.  
Severity: `important`

9. Section: `§§12 Subscription tiering, 11.3 Decisions recorded from Kel's follow-up`  
Title: `Tier source of truth overlaps`  
Problem: `§12.4` proposes adding `org_subscriptions.tier = monitor|operate`, but the repo already models product access through `subscriptions`, `org_subscriptions.subscription_id`, and `modules`. Adding a separate tier column creates two entitlement systems unless the precedence and sync rules are specified.  
Suggested fix: Use the existing subscription catalog as the single source of truth: add Monitor/Operate subscription rows or an explicit subscription-to-capability mapping, and derive ClientPulse behavior from the active subscription rather than introducing a parallel `tier` field.  
Severity: `important`

10. Section: `§§13 Intelligence Briefing + Weekly Digest integration, 14 Plain-English delivery summary, 19 Onboarding flows`  
Title: `Pattern C wording still drifts`  
Problem: `§13.3` chooses explicit org scope (Pattern C), but later text still says the org-subaccount is the “canonical home” for org-level work, `§13.6` labels the new files as “Pattern A new files,” and `§19.1` says create the org-subaccount “(Pattern C).” That leaves the reader unclear whether org-level playbooks run on `scope='org'` or on a synthetic subaccount.  
Suggested fix: Replace all residual Pattern A language. State explicitly: org-scoped playbooks execute with `scope='org'`; org-subaccount may still exist for portal/inbox concerns, but it is not the execution target or query key for org-level playbook runs.  
Severity: `important`

11. Section: `§§7 Dashboard UI gaps, 13 Intelligence Briefing + Weekly Digest integration, 19 Onboarding flows`  
Title: `Timezone field is unspecified`  
Problem: The Monday email requirements reference `organisations.timezone`, but `server/db/schema/organisations.ts` has no such column. The scheduling system already has timezone fields elsewhere, so the spec currently points at a field that does not exist and does not name the actual source of truth.  
Suggested fix: Either add `organisations.timezone` explicitly to the schema plan and use it consistently, or rewrite the spec to use the existing schedule-level timezone fields. Do not leave report delivery tied to a non-existent column.  
Severity: `important`

12. Section: `§§17 Configuration Agent integration, 18 Template editor + system-admin governance model`  
Title: `Audit table duplicates existing history`  
Problem: `§17` introduces a new `config_changes` table and new audit flow, but the repo already has `config_history`, `configHistoryService`, and config-history routes. A second audit store would split change history across two tables and complicate rollback/history UI.  
Suggested fix: Reuse `config_history` as the audit source of truth. If ClientPulse needs per-path diffs or extra metadata, extend `config_history`/`configHistoryService` for hierarchy-template changes instead of creating `config_changes`.  
Severity: `important`
