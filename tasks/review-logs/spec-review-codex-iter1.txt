1. **Section 11. Open items for the implementation plan**
   **Authoritative source unresolved**
   The spec treats `system_skills` as the write target in sections 1, 4, 5, 6, and 8, but section 11 reopens that as an undecided question. In the current codebase, `server/services/systemSkillService.ts` explicitly says `server/skills/*.md` is the source of truth and exposes no DB CRUD for system skills, so implementation cannot safely proceed as written.
   **Suggested fix:** Remove this from “open items” and make a hard decision in the spec. If DB-backed, add a prerequisite phase that converts `systemSkillService` to read/write `system_skills` and defines migration/backfill from markdown. If markdown-backed, rewrite every `system_skills` read/write reference in the spec to a filesystem-based flow and drop the DB-write assumptions.
   **Severity:** critical

2. **Section 5.3 `skill_analyzer_results.matchedSkillId` semantics**
   **Match identity drift**
   This section says `matchedSkillId` now points at `system_skills.id`, but the current analyzer still distinguishes org matches via `matchedSkillId` and system matches via `matchedSystemSkillSlug` in `server/db/schema/skillAnalyzerResults.ts`, `server/jobs/skillAnalyzerJob.ts`, and the client types in `client/src/components/skill-analyzer/SkillAnalyzerWizard.tsx`. The spec never states whether `matchedSystemSkillSlug` is removed, retained for compatibility, or how old rows are interpreted.
   **Suggested fix:** Declare one canonical match identifier model. Either keep both fields and document exact semantics for each, or replace them with a single system-skill ID field plus an explicit migration/response-contract update that removes `matchedSystemSkillSlug` everywhere.
   **Severity:** important

3. **Section 5.2 New columns on `skill_analyzer_results`**
   **Agent proposals use unstable keys**
   `agentProposals` stores only `{ slug, score, selected }`, and the new endpoints in 7.3 also mutate by `slug`. In the current code, `systemAgentService.updateAgent()` rewrites the slug when an agent name changes, so a proposal captured at analysis time can drift before execute even if the agent still exists.
   **Suggested fix:** Store agent identity by immutable `systemAgentId`, with `slug`/`name` as display-only snapshots. Change the proposal schema and the `/agents` endpoint to use `systemAgentId`, and resolve current slug/name when reading.
   **Severity:** important

4. **Section 6.2 Agent-propose edge cases**
   **Selection rules contradict each other**
   The pipeline says “take top 3 with score `>= 0.50`, write to `agentProposals` with `selected: true`,” but the UI example in 7.1 shows an unchecked `52%` chip, and 6.2 also says a single below-threshold agent should still appear unchecked. The manual-add flow is also underspecified because it appends `{ slug, score, selected: false }` without saying how `score` is obtained.
   **Suggested fix:** Choose one consistent rule and state it explicitly: either persist top-K regardless of threshold and set `selected = score >= threshold`, or persist only threshold-passing proposals and allow manual additions with `score: null`. Update the example and endpoint contract to match.
   **Severity:** important

5. **Section 7.2 "Approve all new" behaviour**
   **Bulk partial-overlap flow undefined**
   The section says bulk-approving partial overlaps skips rows with `proposedMergedContent === null` and shows a count, but there is no defined bulk action for partial overlaps in the current UI, and the spec does not say whether skipped rows become `actionTaken = 'skipped'` or are merely omitted from the request. That ambiguity will leak into both client behavior and execute semantics.
   **Suggested fix:** Add an explicit bulk-approve-partial-overlaps contract: which button triggers it, which endpoint it calls, what rows are included, and whether unavailable rows are persisted as `skipped` or left unchanged.
   **Severity:** important

6. **Section 3. Current state**
   **Classifier behavior misstated**
   The spec says the classify stage calls Claude only for the ambiguous band `0.60–0.92` and that DISTINCT and DUPLICATE are assigned by band without an LLM call. The current job code in `server/jobs/skillAnalyzerJob.ts` actually sends both `likely_duplicate` and `ambiguous` bands to the LLM, and only `distinct` bypasses classification.
   **Suggested fix:** Rewrite section 3 to match the actual pipeline, including the `likely_duplicate` LLM path, or explicitly scope a code change that removes that path before relying on the new description elsewhere in the spec.
   **Severity:** important

7. **Section 3. Current state**
   **Route list drifts from implementation**
   The listed current routes omit the job-scoped path segments and do not match `server/routes/skillAnalyzer.ts`. For example, the current patch route is `PATCH /api/system/skill-analyser/jobs/:jobId/results/:resultId`, not `PATCH /results/:id`.
   **Suggested fix:** Replace the shorthand list with the full current paths exactly as implemented, or label the current list as suffix-only shorthand. Keep section 7.3 aligned to the same style.
   **Severity:** minor

8. **Section 10. Build phases**
   **Test plan conflicts with repo posture**
   Phase 2 calls for a “full pipeline” integration test, and phases 4 and 5 call for component tests. That conflicts with the codebase’s stated testing envelope here: static gates plus pure-function unit tests, with no frontend tests and no composition/integration-style tests.
   **Suggested fix:** Rewrite the phase gates to use pure-function unit tests only for ranking/parser/merge helpers, plus `lint`, `typecheck`, and existing static gates. Remove the frontend/component and full-pipeline integration-test requirements from the spec.
   **Severity:** important
