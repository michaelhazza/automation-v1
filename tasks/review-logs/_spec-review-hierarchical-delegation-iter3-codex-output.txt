Not implementation-ready yet; the biggest blockers are phantom persistence artifacts, contradictory phase statements, and a few spec contracts that do not line up with the current repo surface.

1. The spec routes on `task.triggerContext.scope`, but `tasks` has no `triggerContext` column and no schema/file-inventory item adds one, so Phase 2 cannot work as written. Section: §6.7, §11 Phase 2, §14.2. Fix: either add a concrete `tasks.trigger_context` JSONB migration plus schema/inventory entries, or pass `scope` through the orchestrator job payload instead of the task row. Severity: critical.

2. The delegation graph design depends on `agent_runs.handoffSourceRunId` as if it already exists, but the current schema only has `tasks.handoffSourceRunId`, and the spec never adds the missing run column. Section: §1 item 9, §2.1 Run trace, §7.2, §14.4. Fix: either add `agent_runs.handoff_source_run_id` as an explicit schema change, or redesign the graph around the columns that actually exist today. Severity: critical.

3. Root-only `subaccount` delegation is gated on `caller.hierarchy.parentId === null`, but §4.1 also makes `parentId` null in rootless subaccounts, so non-root agents can become root-authorized by accident. Section: §4.1, §4.2, §6.4. Fix: add an explicit `isRoot` flag or define root-only checks as `rootId === agentId && rootId !== null`, then use that invariant everywhere. Severity: critical.

4. Phase 1 is described as “telemetry writes” and “records what’s happening under the flat model,” but later the same phase says `delegation_outcomes` ships empty and writes only start in Phase 4. Section: §1 item 1, §4.3, §11 Phase 1, §12.4. Fix: decide whether outcome writes start in Phase 1 or Phase 4, then rewrite all affected sections to match. Severity: critical.

5. The `agent_runs` telemetry migration is internally inconsistent: §5.3 defines 0204 as adding `delegation_scope`, `hierarchy_depth`, and `delegation_direction`, but the overview, Phase 1 schema list, and file inventory repeatedly omit `delegation_direction`. Section: §5.3, §5.5, §11 Phase 1, §14.1. Fix: make 0204 consistently two columns or three columns everywhere, including exit criteria and inventory. Severity: important.

6. The spec claims Phase 2 removes the hardcoded Orchestrator slug, but §6.6 and §13 still rely on the slug for `scope === 'org'`, so slug removal is only partial. Section: §1 items 3 and 7, §6.6, §6.7, §13. Fix: either narrow the claim to subaccount-scope routing only, or fully specify the org-scope replacement so the slug can truly disappear. Severity: important.

7. The new RLS policy uses `current_setting('app.current_organisation_id')`, which is stale relative to the codebase’s shipped RLS contract on `app.organisation_id`. Section: §5.4, §9.1. Fix: change the policy and prose to `app.organisation_id` unless the spec also introduces a coordinated global RLS-variable rename. Severity: important.

8. The proposed graph endpoint uses a route/helper shape that does not match the current run API surface: the repo uses `/api/agent-runs/:id`, while the spec introduces `/api/runs/:runId/delegation-graph` and `requireRunAccess('view')` without inventorying the new access primitive. Section: §7.2, §9.3, §14.4. Fix: choose the actual route family and access mechanism up front and make the spec consistent end-to-end. Severity: important.

9. The client file inventory is wrong for subaccount creation: `client/src/pages/SubaccountCreatePage.tsx` does not exist, and current creation happens in `AdminSubaccountsPage.tsx` plus a quick-create path in `Layout.tsx`. Section: §8.1, §14.2. Fix: name the real create surface(s) and give each an explicit ship/defer verdict. Severity: important.

10. The spec assigns the `config_list_*` tool-definition changes to `server/config/actionRegistry.ts`, but those slugs are not defined in that registry in the current codebase. Section: §6.2, §14.3. Fix: point the contract update at the real source of truth for these tool definitions and update the inventory accordingly. Severity: important.

11. The background section is already factually stale: the current `automation-os` manifest has `portfolio-health-agent` with `reportsTo: null`, so “Orchestrator root and 15 direct reports” is no longer true. Section: §2.1. Fix: update the seeded-company description, or explicitly scope the statement to the subaccount hierarchy only. Severity: important.

12. The spec removes the “sub-agents cannot spawn sub-agents” block and then says recursion is capped by `MAX_HANDOFF_DEPTH = 5`, but it never states how spawn-created runs increment or enforce that limit. Section: §6.3, §7.2, §10.2. Fix: explicitly reuse and increment an existing depth field for spawned runs and name the write path that enforces it. Severity: important.

13. `briefErrorArtefactService.emit` is used as a load-bearing fallback for unsupported system-scope routing, but that service is not otherwise defined, reused, or inventoried anywhere in the spec. Section: §6.6, §6.7, §14. Fix: either reuse the existing brief conversation/artefact writer primitives already in the repo or add a concrete new service/file to the inventory. Severity: important.

14. The shared `DelegationScope` contract says `subaccount` is valid across the execution layer, but §6.3 later makes `spawn_sub_agents` reject `subaccount` for every caller, so the shared contract is false. Section: §4.2, §6.3. Fix: split the semantics by skill in the contract table or explicitly carve spawn out as an exception. Severity: important.

15. The spec gives zero-root subaccounts three different statuses at once: valid fallback state, post-Phase-2 critical health finding, and operator-opt-in steady state. Section: §5.1, §6.9, §11 Phase 2, §16.3. Fix: declare one explicit steady-state verdict for zero-root subaccounts and align detector severity with that decision. Severity: important.

16. Success criteria use `delegation_outcomes` for zero-count and threshold metrics even though the same spec defines those writes as best-effort and lossy under DB pressure. Section: §4.3, §10.3, §17.2-§17.3. Fix: use `delegation_outcomes` for advisory trends only, and anchor correctness invariants to non-lossy validation paths or immutable run/task state. Severity: important.
