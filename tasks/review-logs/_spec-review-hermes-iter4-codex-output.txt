**Findings**

- `P1` §7.3/§7.4.1/§8.3, lines `124-125`, `530-534`, `594`, `833`: Phase C promises an `llm_requests`-backed breaker read for `llmRouter`, but the router pseudocode still calls unchanged `assertWithinRunBudget(...)` and the contracts keep that export’s shape unchanged, so the source-selection mechanism is not actually pinned. Recommendation: choose one concrete mechanism and use it everywhere in the spec, e.g. add a router-only `assertWithinRunBudgetFromLedger(ctx)` sibling helper and have `llmRouter` call that, or explicitly allow an internal source selector on the called helper.

- `P1` §9, lines `891-914`: The spec explicitly introduces `RunCostPanel.test.tsx` and `llmUsage.test.ts` even though `docs/spec-context.md` says `frontend_tests: none_for_now` and `api_contract_tests: none_for_now`, but it does not include the HITL/spec-level override that the context file requires for intentional framing departures. Recommendation: either remove those two test surfaces from this spec or add an explicit framing-override checkpoint section before implementation authorizing those exact exceptions.

- `P1` §8.3/§9.2, lines `755-815`, `930-932`: `options.overrides` is defined only on `extractRunInsights(..., options?)`, but the test plan places override coverage in `workspaceMemoryServicePure.test.ts` even though the pure exports do not accept `options` or emit row-level `isUnverified` / `provenanceConfidence`, so that test plan is impossible as written. Recommendation: move override coverage to a `workspaceMemoryService`-level test, or extract a pure override-application helper into `workspaceMemoryServicePure.ts` and add that helper to the file inventory and contracts.

- `P2` §5.1/§5.3/§11.5, lines `158`, `205`, `1069`: After dropping `PlaybookRunDetailPage.tsx` from Phase A, stale prose still says “agent or playbook run,” “three other run-detail surfaces,” and “all four host pages,” which contradicts the now-scoped set of three direct agent-run surfaces. Recommendation: normalize all Phase A prose and verification text to “three direct agent-run surfaces” only: `AgentRunHistoryPage`, `RunTraceViewerPage`, and `AdminAgentEditPage`.

- `P2` §1/§4.5/§6.4/§11.4, lines `30`, `146`, `355`, `1059`: High-level Phase B prose still says the spec threads or consumes `TrajectoryDiff.pass`, while the detailed contract says `trajectoryPassed` is always `null` in Phase B because no verdict is persisted. Recommendation: rewrite the summary/inventory language to say `trajectoryPassed` is a reserved forward-compatible slot, not an active Phase B signal.

- `P2` §6.3.1, line `299`: The write-once invariant claims a read-then-write guard in `finaliseAgentRun` or equivalent with `runResultStatus.write_skipped`, but no concrete function/file is pinned there and no such symbol appears elsewhere in the spec inventory, so the guarantee lacks a named mechanism. Recommendation: pin the exact helper(s) and file(s) that perform the guard at each write site, or drop the guard claim and rely only on the three explicitly enumerated atomic terminal writes.

- `P2` §4/§9.6, lines `81`, `996`: The inventory lock says any needed file must be listed up front, but the review gate requires writing `tasks/pr-review-log-hermes-tier-1-<ISO-timestamp>.md`, which is not in any inventory table. Recommendation: either add a non-source artifact entry for that review-log path or explicitly exempt generated review artifacts from the §4 inventory lock.
