## 1. Goals, non-goals, framing assumptions

### 1.1 Goals

- Replace per-document **eager / lazy** loading with one of three modes: **Auto** (default), **Always available**, **Reference only**. The user-facing eager / lazy concept disappears.
- Add a **shared retrieval engine** that ranks both `memory_blocks` and `reference_documents` against the current task context, applying threshold filtering and budget capping.
- Add the missing **org** and **recurring-task** scopes for documents; add the **recurring-task** scope for memory blocks. Five tiers for documents, four tiers for memory blocks.
- Ship a **chunked-embedding model** for documents (whole-document embeddings are not used).
- Surface **retrieval observability** as an operator product (not internal-only telemetry): "why was this loaded / not loaded" tooltips, per-document relevance signals, per-run trace.
- Surface **Files** as a Knowledge tab and ship the **Add to Knowledge** promotion flow.
- Hard tenant isolation: authorization filtering occurs **before** semantic retrieval.

### 1.2 Non-goals (explicit)

- No new storage backend. Continue with Postgres + pgvector + OpenAI `text-embedding-3-small`.
- No cross-encoder re-ranking. Pure cosine for v1.
- No version-aware retrieval (latest version only).
- No document-level ad-hoc semantic search UI.
- No per-tenant custom embedding provider.
- No backfill of historical documents (dev environment, no data).
- No promotion-request workflow for sub-account admins (org admins are the sole org-pinning authority in v1).
- No new storage layer for files; the Files tab surfaces existing `execution_files` rows.
- No bundle-driven retrieval. **Bundles remain organisational only**, they do not modify ranking, threshold, or budget behaviour. Mode is set on the document, not on the bundle (enforced).

### 1.3 Framing assumptions (cross-checked against `docs/spec-context.md`)

- `pre_production: yes`, `live_users: no`, `feature_stability: low`. Breaking schema changes acceptable.
- `testing_posture: static_gates_primary`, `runtime_tests: pure_function_only`. The retrieval ranker, threshold logic, scoping logic, and re-embedding decision are pure-function tested. No frontend / API-contract / E2E tests against the app.
- `feature_flags: only_for_behaviour_modes`. No rollout-gate flags introduced. The legacy `loadingMode: 'eager' | 'lazy'` column on `agent_data_sources` is **retired** as part of this build (see Â§15).
- `staged_rollout: never_for_this_codebase_yet`. Direct cutover.
- Existing primitives prevail: `memoryBlockRetrievalServicePure`, `referenceDocumentService`, `executionFiles`, the bundle stack, `withOrgTx`, `RLS_PROTECTED_TABLES`, `withBackoff`, `createWorker`. No new service layers without a "why-not-reuse" paragraph.

### 3.3 Invented (genuinely new primitives)

Three new primitives. Each has a one-paragraph justification.

#### 3.3.1 `reference_document_chunks` (new table)

**Why:** documents are chunked for retrieval; each chunk gets its own embedding and is independently retrievable (brief Â§8). Memory blocks embed the whole block because blocks are short-by-design; documents can be 50+ pages and require chunk-level granularity. Reusing `memory_blocks` for chunks would conflate two distinct lifecycles (a chunk is derived; a block is authored).

#### 3.3.2 `reference_document_data_sources` (new join table, link / scope / mode)

**Why:** the existing bundle-attachment table `document_bundle_attachments` only supports bundle-level attachment with `attachment_mode: 'always_load' | 'available_on_demand'`. The brief requires direct (non-bundle) attachment of a document to a scope (org / sub-account / agent / recurring task / task instance) with the **document's own mode** (Auto / Always available / Reference only). Reusing `agent_data_sources` would force documents through the HTTP-source ingestion model; reusing `documentBundleAttachments` would couple every document to a synthetic single-document bundle and contradict Â§5 of the brief ("bundles are organisational, not retrieval-altering"). A new join table is correct.

#### 3.3.3 `retrievalServicePure` (new pure-function service)

**Why:** the existing `memoryBlockRetrievalServicePure` is memory-block-specific (priority enum, divergence flags, ownerAgent semantics). The brief Â§8 requires a single ranker that takes "a set of candidate items with embeddings" and returns "items to load, in priority order". The right move is to **extract the generic part** of the memory-block ranker into a new pure module with a polymorphic input shape (`RetrievalCandidate`), and have `memoryBlockRetrievalServicePure` plus the new `documentRetrievalServicePure` both consume it. That preserves the memory-block-specific logic where it belongs (block-side filters before ranking) while concentrating the ranking algorithm in one place. Future cross-encoder re-ranking, learned thresholds, and new knowledge primitives all benefit.

## 4. Domain model (data layer)

### 4.1 The five tiers (documents)

A document attaches at exactly one of these scopes per link row:

| Tier | Tier ID | Scope key columns on `reference_document_data_sources` |
|---|---|---|
| 1. Task instance | `task_instance` | `task_instance_id` (FK to `agent_runs.id` or successor) |
| 2. Recurring task | `recurring_task` | `scheduled_task_id` |
| 3. Agent | `agent` | `agent_id` |
| 4. Sub-account | `subaccount` | `subaccount_id` |
| 5. Organisation | `organisation` | (org-only; all scope keys above are null) |

A single document can have multiple link rows (one per scope). Mode is set on the document, not on the link, so all attachments of the same document share the same mode (see Â§6.4 for the source-of-truth rule).

### 4.2 The four tiers (memory blocks)

Memory blocks gain `scheduled_task_id`. They do **not** gain `task_instance_id` (brief Â§7, persistent memory at an ephemeral scope is a category error).

| Tier | Memory blocks | Documents |
|---|---|---|
| Org | yes (existing) | yes (new via `reference_document_data_sources`) |
| Sub-account | yes (existing) | yes (existing path now goes via the new link table) |
| Agent | yes (existing, `owner_agent_id`) | yes (new) |
| Recurring task | **yes (new, `scheduled_task_id`)** | yes (new) |
| Task instance | no | yes (new) |

### 4.3 Modes

Three discriminated values on `reference_documents.mode`:

- `auto` (default), eligible for retrieval; participates in threshold + ranking + budget.
- `always_available`, bypasses threshold and ranking; **does still participate in overall context budgeting** (brief Â§3 hard invariant). If always-available alone exceeds context budget, the system surfaces operator guidance and degrades gracefully (see Â§10.5).
- `reference_only`, never auto-loaded. Only its title + 1-2 sentence summary appears in the prompt manifest. The agent fetches full content via tool call.

The status set is **closed**: adding a new mode value requires a spec amendment (Â§10.7).

### 4.4 Source provenance

`reference_documents.source` tracks where the document came from. Closed enum:

- `manual` (default, user-authored on the Documents tab)
- `from_file` (promoted from `execution_files` via Add to Knowledge)
- `auto_memory_approved` (promoted from auto-memory pending review)
- `synthesised_by_agent` (reserved for future, brief Â§6 "system-generated documents")
- `external` (existing, Google Drive etc., for compatibility)

Brief Â§17 rule: only non-default cases get a UI badge. The default `manual` shows no badge.

### 4.5 Chunks

`reference_document_chunks` rows are derived from `reference_document_versions`. When a new version lands, chunk rows are written by the chunking job and the prior version's chunks are retained until the new version is ready, then atomically swapped (atomic-swap source of truth: Â§6.4 / Â§13).

Chunk size, overlap, and boundary detection are tuned in implementation; the spec mandates only:

- Chunks are semantically coherent (paragraph- or section-aligned, not byte-windowed).
- Each chunk has its own embedding row.
- Each chunk references the parent `document_version_id` so version pinning is preserved.
- Whole-document embeddings are NOT stored on `reference_documents`. Retrieval operates at chunk granularity (brief Â§8).

### 4.6 Files vs Documents

`execution_files` is **not** modified by this build. The Files tab is a read view of existing rows filtered by sub-account / agent / scope. The "Add to Knowledge" action transforms an `execution_files` row into a new `reference_documents` row plus link rows; the file row itself is untouched (the brief mandates the source file becomes durable, `execution_files.expiresAt` is updated to `NULL` or far-future as part of the promotion transaction).

## 6. Contracts

Every cross-boundary data shape is pinned here with at least one concrete example. Source-of-truth precedence is declared where multiple representations exist.

### 6.1 `RetrievalCandidate`

```ts
type RetrievalCandidate = {
  id: string;                    // chunk id OR memory_block id
  kind: 'document_chunk' | 'memory_block';
  organisationId: string;
  subaccountId: string | null;
  agentId: string | null;
  scheduledTaskId: string | null;
  taskInstanceId: string | null;       // documents only; always null for memory blocks
  embedding: number[];                  // vector(1536)
  tokenCount: number;
  // For documents: the chunk's parent document id, version id, and mode.
  parentDocumentId?: string;
  parentDocumentVersionId?: string;
  mode?: 'auto' | 'always_available' | 'reference_only';
  // For memory blocks: priority + isAuthoritative + ownerAgent (already in the existing ranker).
  priority?: 'low' | 'medium' | 'high';
  isAuthoritative?: boolean;
};
```

### 6.2 `RetrievalResult`

```ts
type RetrievalResult = {
  loaded: Array<{
    candidateId: string;
    finalScore: number;
    relevanceScore: number;
    scopeBonus: number;
    recencyBonus: number;
    operatorPinBonus: number;
    tier: 'task_instance' | 'recurring_task' | 'agent' | 'subaccount' | 'organisation';
    tokenCost: number;
  }>;
  rejected: {
    aboveThreshold: Array<{ candidateId: string; finalScore: number; reason: 'budget_exhausted' | 'lower_score_in_tie' }>;
    belowThreshold: { count: number };  // Counts only, not full list.
  };
  alwaysAvailable: Array<{ candidateId: string; tokenCost: number; degraded: boolean }>;
  referenceOnlyManifest: Array<{ candidateId: string; title: string; summary: string }>;
  poolSize: { taskInstance: number; recurringTask: number; agent: number; subaccount: number; organisation: number };
};
```

### 6.3 `RetrievalRejectionReason` (closed)

`'budget_exhausted' | 'lower_score_in_tie' | 'mode_excluded' | 'authorization_filtered' | 'below_threshold'`. Authorization-filtered candidates never appear in the result; the count is a property of the candidate-pool builder, not the ranker.

### 6.4 Source-of-truth precedence

Multiple representations exist for the same fact. The precedence is **declared, not implicit**:

| Fact | Source of truth | Read path |
|---|---|---|
| Document content | `reference_document_versions` (the row pointed to by `reference_documents.current_version_id`) | always read via `referenceDocumentService.getCurrent(documentId)` |
| Document mode | `reference_documents.mode` | one column, no other source |
| Link / scope | `reference_document_data_sources` rows for `(document_id, scope_columns)` | `documentDataSourceService.findByDocument(documentId)` |
| Document chunks | `reference_document_chunks` joined to `reference_document_versions` via `version_id` | always pinned by version id; never "latest by timestamp" |
| `summary_stale` | `reference_documents.summary_stale` (boolean), set true on version write, set false when `document:summarise` job completes successfully | UI must surface stale state when true |

The atomic-swap rule for chunks (Â§4.5) is: on new version write, the new chunks are inserted with the new `version_id`; the active version pointer flips only after all chunks for the new version are embedded; old chunks are not deleted, they age out via `version_id != current_version_id`.

### 6.5 Add-to-Knowledge promotion contract

### 7.1 Authorization-before-retrieval invariant (brief Â§10)

The candidate pool is built **before** ranking. The pool builder (`retrievalService.buildCandidatePool`) is the only path that touches the DB; it runs inside `withOrgTx` so RLS is in effect. The ranker (`retrievalServicePure`) is pure and never sees rows it shouldn't have. The rule is: **no path constructs a candidate from a row that didn't survive the RLS-scoped query**. Tested at the pure-function boundary by passing mixed-org candidates and asserting the ranker filters by org id (defence-in-depth, the ranker treats wrong-org candidates as a programmer error, not normal flow).

## 8. Execution model

Per checklist Â§5: every behaviour that crosses a transactional or latency boundary picks one of inline / queued / cached.

| Behaviour | Model | Why |
|---|---|---|
| Per-run retrieval (Â§4 of brief) | **Inline / synchronous.** | The agent execution path needs the result before assembling the prompt. Latency budget is < 100ms added per run (brief Â§16). |
| Document version write | **Inline.** | The version row IS the durable artefact; the user expects "save" to commit. Returns the new version id. |
| Summarisation on version write | **Queued (`document:summarise`).** | LLM call, network-bound, retryable. |
| Chunking + embedding on version write | **Queued (`document:chunk-embed`).** | Chunking is CPU; embedding is network-bound. The new version's chunks are committed inside this job; the active-version pointer flips at the end. |
| Re-embedding on model upgrade | **Queued sweep (`document:reembed`).** | Walks every document; idempotent per chunk via `(version_id, embedding_model)` unique index. |
| Add-to-Knowledge promotion | **Inline transaction + queued finalise.** | The user expects the file to appear as a document immediately. Summarisation + embedding queue afterward. |
| Retrieval observability event emission | **Inline at terminal (run end).** | Same boundary as the existing `agent_execution_events` emitter. |

No prompt-partition / cached-context interaction in v1. The brief explicitly defers cross-tier cache-token optimisation; documents go into the dynamic suffix per existing prompt-assembly conventions. Future work (Phase 8+) can graduate Always-available documents into a stable-prefix partition.

### 8.1 Idempotency across queued boundaries

| Job | Idempotency key | Posture |
|---|---|---|
| `document:summarise` | `(document_id, version_id)` | key-based, DB unique index on the pair; if job replays, no new write |
| `document:chunk-embed` | `(version_id, chunk_index)` | key-based, DB unique index on chunks; if a chunk already exists, the job no-ops it |
| `document:reembed` | `(version_id, chunk_index, embedding_model)` | key-based, supports embedding-model migrations |
| `document:promotion-finalise` | `(document_id, file_id)` | key-based |

## 10. Execution-safety contracts

Per checklist Â§10.

### 10.1 Idempotency posture (every external-trigger write)

| Write | Posture | Mechanism |
|---|---|---|
| Add document version | state-based (`version = current_version + 1` predicate; UNIQUE on `(document_id, version)`) | `referenceDocumentService.writeVersion` |
| Insert chunk | key-based (`UNIQUE (version_id, chunk_index)`) | `documentEmbeddingService.embedChunk` |
| Insert link row | key-based (`UNIQUE (document_id, scope_columns) WHERE deleted_at IS NULL`) | `documentDataSourceService.link` |
| Mode change | state-based (`UPDATE ... WHERE mode <> :new_mode`) | `documentDataSourceService.changeMode` |
| Add-to-Knowledge promote | key-based (`UNIQUE (file_id, document_id) WHERE deleted_at IS NULL` on a new `document_promotion_audit` row inside the transaction) | `documentPromotionService.promoteFile` |
| Retrieval event | non-idempotent (intentional; per-run terminal event, single writer) | `retrievalObservabilityService.emit` |

### 10.2 Retry classification

| Operation | Class | Boundary |
|---|---|---|
| Add version | guarded | predicate above |
| Insert chunk | safe | dedup'd by unique index |
| Insert link row | guarded | unique index |
| Mode change | safe | optimistic predicate |
| Promote file | guarded | inside `withOrgTx`; unique index on the audit row |
| Re-embed sweep | safe | per-chunk idempotency |

### 10.3 Concurrency guard for racing writes

| Race | Guard | Losing-caller response |
|---|---|---|
| Two callers promote the same file simultaneously | UNIQUE on `(file_id) WHERE ...` on the audit row; catch `23505` | 409 Conflict + "Already promoted to document {id}" + the existing document id in the body |
| Two callers change document mode at the same time | Last-write-wins is acceptable (mode is a property, not a state machine); no guard needed | the second response carries the final mode |
| Two callers add the same document at the same scope | UNIQUE on `(document_id, scope_columns) WHERE deleted_at IS NULL`; catch `23505` | 409 Conflict + "Already linked at this scope" |
| Two embedders run on the same chunk | UNIQUE on `(version_id, chunk_index)` | safe, one wins, the other no-ops |
| Two re-embed jobs touch the same chunk | same UNIQUE | same |

### 10.4 Terminal event guarantee

Every retrieval has exactly one terminal observability event (`retrieval.summary`) per run, emitted once at the end of the run regardless of success / partial / failed status. No intermediate retrieval events.

### 10.5 No-silent-partial-success

The "always-available bypass-but-still-budget-bound" rule (brief Â§3) is the trickiest place. The contract is:

- If `sum(alwaysAvailable token_cost) > context_budget`, the system flags the run as `degraded` in the retrieval result and emits operator guidance via `RetrievalResult.alwaysAvailable[i].degraded = true`. It does NOT silently truncate; the agent execution continues but the run-summary surface marks it. Operator-facing copy in the UI reads "Some always-available documents could not be loaded due to context limits."
- Threshold-passing documents that don't fit the remaining budget are tracked in `rejected.aboveThreshold[i].reason = 'budget_exhausted'` so the "why wasn't this loaded?" drill-in (Â§11) can surface the truth.

### 10.6 Unique-constraint-to-HTTP mapping

| Constraint | HTTP status | Body |
|---|---|---|
| `reference_documents_org_name_uq` violation | 409 Conflict | `{ error: "DOCUMENT_NAME_EXISTS", existingDocumentId: "..." }` |
| `reference_document_data_sources_unique_per_scope` | 409 Conflict | `{ error: "DOCUMENT_ALREADY_LINKED", existingLinkId: "..." }` |
| `reference_document_chunks_unique_per_chunk_index` | 200 (idempotent hit) | `{ noop: true, chunkId: "..." }` (job context only; never bubbled to operators) |
| `document_promotion_audit_unique_per_file` | 409 Conflict | `{ error: "FILE_ALREADY_PROMOTED", existingDocumentId: "..." }` |

### 10.7 State machine closure

The `mode` enum is closed: `auto | always_available | reference_only`. Adding a new mode requires a spec amendment. The `source` enum is also closed (Â§4.4).

`summary_stale` is a boolean (`true` after version write, `false` after summarisation completes). Forbidden transitions: none; the column is a flag, not a multi-state machine.

`reference_document_data_sources` rows have a soft-delete model (`deleted_at IS NULL` filter). Forbidden transitions: re-creating a deleted link must use a fresh row, not un-deleting (audit-friendly). The unique index applies only to `WHERE deleted_at IS NULL`.

## 12. Tenant isolation invariants

Brief Â§10. The hard rules:

1. **Cross-org boundary is absolute.** `withOrgTx` is the only entry to retrieval reads. The `RLS_PROTECTED_TABLES` manifest enforces FORCE RLS on every new table. The CI gate `verify-rls-coverage.sh` blocks merges that miss a policy.
2. **Sub-account isolation is enforced at candidate-pool construction.** The pool builder runs `WHERE subaccount_id = :ctx_subaccount_id OR subaccount_id IS NULL` for sub-account-scoped reads.
3. **Authorization filtering occurs BEFORE semantic retrieval.** No path runs a similarity search and then filters; the candidate pool IS the result of an authorization-scoped query.
4. **Embeddings are tenant-scoped.** `reference_document_chunks.organisation_id` is on every row (denormalised from the parent document for RLS-policy locality).

Tested by:

- Existing `rls.context-propagation.test.ts` harness expanded with the new tables (Phase 1).
- Pure-function ranker test that asserts mixed-org candidates are filtered by `organisationId` (defence-in-depth, Phase 2).
- A new manual test in the spec-conformance step at Phase 4: spin up two orgs, attach the same-named document to each, verify run isolation.

## 13. Lifecycle, versioning, re-embedding

Brief Â§11. Knowledge artefacts are versioned and re-indexable; embedding generations are replaceable infrastructure.

### 13.1 Document edit

1. New version row written (`reference_document_versions`); `current_version_id` flipped at the end.
2. `summary_stale = true` immediately on version write.
3. `document:summarise` job enqueued.
4. `document:chunk-embed` job enqueued.
5. Retrieval continues using the prior version's chunks until the new version's chunks are committed.
6. Atomic swap: when the chunking job completes, it updates `reference_documents.last_chunked_at`. Until then, retrieval queries chunk rows `WHERE version_id = current_version_id`. Old version's chunks are retained (history) but excluded.

### 13.2 Source-file change (e.g. underlying CSV updated)

Same as document edit, triggered by the existing version-write hook in `referenceDocumentService`.

### 13.3 Embedding model upgrade

When OpenAI ships a new model (or operator chooses to migrate):

1. New unique index on `(version_id, chunk_index, embedding_model)` ensures parallel embedding generations don't collide.
2. `document:reembed` job sweeps all chunks where the embedding model is the old version.
3. Reads continue from the old embedding until the new one is ready, swap is per-chunk-row; no operator action.

### 13.4 Bundle changes

Bundle composition affects which documents are pulled into a scope. The link rows are in `reference_document_data_sources` (direct attachment) and `document_bundle_attachments` (bundle attachment); neither requires re-embedding. The retrieval engine treats both as candidate-pool sources and deduplicates by `documentId`.

### 13.5 Document deletion

Soft delete on `reference_documents.deleted_at`. Cascade soft-delete to `reference_document_data_sources` (link rows). Chunks and version rows are retained for audit. Affected agents see a notice on their Data Sources tab (UI surface in Phase 6).

## 18. Open questions

For `spec-reviewer` and `chatgpt-spec-review` to triage. Each is a question, not a hidden assumption:

1. **Chunk size and overlap defaults.** Tuning belongs in the implementation plan, not here, but the spec author leaves the default-value decision to the Phase 2 builder. Reasonable starting point: 512 tokens with 64-token overlap. Spec-reviewer may push back if a different default is conventional in this repo.
2. **Recency bonus shape.** The brief mentions "recency_bonus" in the ranking formula (Â§3) but does not pin the decay. The spec defers this to Phase 2 with a default of "linear decay over 30 days, max +0.05 added to the cosine score". Open to alternative defaults.
3. **Operator-pin bonus shape.** Same as recency. Default +0.10, bounded so it can never invert a relevance gap > 0.10.
4. **Reference-only manifest token cost.** Each manifest entry is title + 1-2 sentence summary; rough budget is < 50 tokens per entry. Should the spec pin a hard cap? Recommendation: yes, 80 tokens per manifest entry, configurable per organisation.
5. **Files tab scope filter.** Should the Files tab default to "current sub-account scope" or "all scopes the user can see"? Recommendation: current sub-account, with a scope selector for org admins.
6. **Add-to-Knowledge transaction shape.** Inline transaction for the row inserts + queued for summarisation/embedding (Â§8). Should the modal block until the summary is generated, or return immediately and show a "summarising..." indicator? Recommendation: return immediately, show indicator, refresh on background completion (existing pattern in this repo).
7. **Phase 7 observability storage option.** A vs B (Â§6.7). Recommendation: A (extend `agent_execution_events`). Spec-reviewer / ChatGPT-spec-review may push back if event volume is a concern.
8. **`agent_data_sources.loadingMode` retirement.** The spec retires the column in Phase 4. Should it instead be DEFAULT NULL'd and ignored, leaving the column for one cycle? Recommendation: drop in Phase 4 cutover. Open to "soft-retire" instead.
9. **Recurring-task scope: which table holds the FK?** `memory_blocks.scheduled_task_id` is straightforward. For `reference_document_data_sources`, the column is part of the scope-key tuple. CHECK constraints enforce mutual exclusion (only one scope key non-null per row except for the org case). Open to alternative shape (e.g. polymorphic `scope_type` + `scope_id`), recommendation: stick with the scope-key columns since the join cardinality is asymmetric and policy-checking is easier with named columns.
10. **What permission key does Files-tab read use?** Recommendation: `knowledge:read` covers it; the existing `files:read` (if any) is not extended. Open if the existing convention prefers per-resource keys.

These are starting points; the author expects them to be tightened or closed by the review pass.

## 6. Contracts

Every cross-boundary data shape is pinned here with at least one concrete example. Source-of-truth precedence is declared where multiple representations exist.

### 6.1 `RetrievalCandidate`

```ts
type RetrievalCandidate = {
  id: string;                    // chunk id OR memory_block id
  kind: 'document_chunk' | 'memory_block';
  organisationId: string;
  subaccountId: string | null;
  agentId: string | null;
  scheduledTaskId: string | null;
  taskInstanceId: string | null;       // documents only; always null for memory blocks
  embedding: number[];                  // vector(1536)
  tokenCount: number;
  // For documents: the chunk's parent document id, version id, and mode.
  parentDocumentId?: string;
  parentDocumentVersionId?: string;
  mode?: 'auto' | 'always_available' | 'reference_only';
  // For memory blocks: priority + isAuthoritative + ownerAgent (already in the existing ranker).
  priority?: 'low' | 'medium' | 'high';
  isAuthoritative?: boolean;
};
```

### 6.2 `RetrievalResult`

```ts
type RetrievalResult = {
  loaded: Array<{
    candidateId: string;
    finalScore: number;
    relevanceScore: number;
    scopeBonus: number;
    recencyBonus: number;
    operatorPinBonus: number;
    tier: 'task_instance' | 'recurring_task' | 'agent' | 'subaccount' | 'organisation';
    tokenCost: number;
  }>;
  rejected: {
    aboveThreshold: Array<{ candidateId: string; finalScore: number; reason: 'budget_exhausted' | 'lower_score_in_tie' }>;
    belowThreshold: { count: number };  // Counts only, not full list.
  };
  alwaysAvailable: Array<{ candidateId: string; tokenCost: number; degraded: boolean }>;
  referenceOnlyManifest: Array<{ candidateId: string; title: string; summary: string }>;
  poolSize: { taskInstance: number; recurringTask: number; agent: number; subaccount: number; organisation: number };
};
```

### 6.3 `RetrievalRejectionReason` (closed)

`'budget_exhausted' | 'lower_score_in_tie' | 'mode_excluded' | 'authorization_filtered' | 'below_threshold'`. Authorization-filtered candidates never appear in the result; the count is a property of the candidate-pool builder, not the ranker.

### 6.4 Source-of-truth precedence

Multiple representations exist for the same fact. The precedence is **declared, not implicit**:

| Fact | Source of truth | Read path |
|---|---|---|
| Document content | `reference_document_versions` (the row pointed to by `reference_documents.current_version_id`) | always read via `referenceDocumentService.getCurrent(documentId)` |
| Document mode | `reference_documents.mode` | one column, no other source |
| Link / scope | `reference_document_data_sources` rows for `(document_id, scope_columns)` | `documentDataSourceService.findByDocument(documentId)` |
| Document chunks | `reference_document_chunks` joined to `reference_document_versions` via `version_id` | always pinned by version id; never "latest by timestamp" |
| `summary_stale` | `reference_documents.summary_stale` (boolean), set true on version write, set false when `document:summarise` job completes successfully | UI must surface stale state when true |

The atomic-swap rule for chunks (Â§4.5) is: on new version write, the new chunks are inserted with the new `version_id`; the active version pointer flips only after all chunks for the new version are embedded; old chunks are not deleted, they age out via `version_id != current_version_id`.

### 6.5 Add-to-Knowledge promotion contract

```ts
type AddToKnowledgeRequest = {
  fileId: string;                       // execution_files.id
  title: string;                        // pre-filled from filename, editable
  scope: 'organisation' | 'subaccount' | 'agent';
  agentIds?: string[];                  // when scope='agent'; otherwise applies to all agents in the chosen scope
  mode?: 'auto' | 'always_available' | 'reference_only';  // default 'auto' if omitted
};
```

Producer: `client/src/api/knowledgeApi.ts -> addToKnowledge`. Consumer: `documentPromotionService.promoteFile(req, principal)`. The transaction is: insert `reference_documents` + insert initial `reference_document_versions` + insert `reference_document_data_sources` rows + update `execution_files.expiresAt` to durable + emit `document:summarise` job + emit `document:chunk-embed` job. All in `withOrgTx`.

### 6.6 Mode-change contract

```ts
type ChangeDocumentModeRequest = {
  documentId: string;
  mode: 'auto' | 'always_available' | 'reference_only';
};
```

Idempotent (state-based: predicate `WHERE mode <> :mode`). Returns the new mode. UI surfaces the change inline. Mode is a property of the document, not the link, so changing mode applies to every scope the document is linked at.

### 6.7 Retrieval observability event

Two implementation options for the spec author and Phase 7 builder:

**Option A (preferred):** write a new event type to the existing `agent_execution_events` ledger (`shared/types/agentExecutionLog.ts -> AGENT_EXECUTION_EVENT_CRITICALITY`). New `criticality` entry: `'observability'`. New event-shape `'retrieval.summary'` carrying the `RetrievalResult` shape from Â§6.2 plus a per-document chunk fan-out under `loaded[i].chunkIds: string[]`.

**Option B:** dedicated `retrieval_events` table (migration `0292`). Use only if event volume justifies a separate table per `architecture.md` event-volume guidance.

**Default:** Option A. Switch to Option B only if Phase 7 telemetry shows event volume swamping the agent-execution ledger. The choice is recorded in the Phase 2 implementation plan, not re-litigated here.

### 6.8 Bundle attachment contract (no change)

`document_bundle_attachments` continues to drive bundle-level attachment. Per brief Â§5: bundles do not modify retrieval. The new per-document `reference_document_data_sources` rows are independent of bundle attachments. A document can be both attached directly and through a bundle; retrieval treats it as a single candidate (deduplicated by `(documentId, chunkId)`).

### 7.1 Authorization-before-retrieval invariant (brief Â§10)

The candidate pool is built **before** ranking. The pool builder (`retrievalService.buildCandidatePool`) is the only path that touches the DB; it runs inside `withOrgTx` so RLS is in effect. The ranker (`retrievalServicePure`) is pure and never sees rows it shouldn't have. The rule is: **no path constructs a candidate from a row that didn't survive the RLS-scoped query**. Tested at the pure-function boundary by passing mixed-org candidates and asserting the ranker filters by org id (defence-in-depth, the ranker treats wrong-org candidates as a programmer error, not normal flow).

## 8. Execution model

Per checklist Â§5: every behaviour that crosses a transactional or latency boundary picks one of inline / queued / cached.

| Behaviour | Model | Why |
|---|---|---|
| Per-run retrieval (Â§4 of brief) | **Inline / synchronous.** | The agent execution path needs the result before assembling the prompt. Latency budget is < 100ms added per run (brief Â§16). |
| Document version write | **Inline.** | The version row IS the durable artefact; the user expects "save" to commit. Returns the new version id. |
| Summarisation on version write | **Queued (`document:summarise`).** | LLM call, network-bound, retryable. |
| Chunking + embedding on version write | **Queued (`document:chunk-embed`).** | Chunking is CPU; embedding is network-bound. The new version's chunks are committed inside this job; the active-version pointer flips at the end. |
| Re-embedding on model upgrade | **Queued sweep (`document:reembed`).** | Walks every document; idempotent per chunk via `(version_id, embedding_model)` unique index. |
| Add-to-Knowledge promotion | **Inline transaction + queued finalise.** | The user expects the file to appear as a document immediately. Summarisation + embedding queue afterward. |
| Retrieval observability event emission | **Inline at terminal (run end).** | Same boundary as the existing `agent_execution_events` emitter. |

No prompt-partition / cached-context interaction in v1. The brief explicitly defers cross-tier cache-token optimisation; documents go into the dynamic suffix per existing prompt-assembly conventions. Future work (Phase 8+) can graduate Always-available documents into a stable-prefix partition.

### 8.1 Idempotency across queued boundaries

| Job | Idempotency key | Posture |
|---|---|---|
| `document:summarise` | `(document_id, version_id)` | key-based, DB unique index on the pair; if job replays, no new write |
| `document:chunk-embed` | `(version_id, chunk_index)` | key-based, DB unique index on chunks; if a chunk already exists, the job no-ops it |
| `document:reembed` | `(version_id, chunk_index, embedding_model)` | key-based, supports embedding-model migrations |
| `document:promotion-finalise` | `(document_id, file_id)` | key-based |

## 10. Execution-safety contracts

Per checklist Â§10.

### 10.1 Idempotency posture (every external-trigger write)

| Write | Posture | Mechanism |
|---|---|---|
| Add document version | state-based (`version = current_version + 1` predicate; UNIQUE on `(document_id, version)`) | `referenceDocumentService.writeVersion` |
| Insert chunk | key-based (`UNIQUE (version_id, chunk_index)`) | `documentEmbeddingService.embedChunk` |
| Insert link row | key-based (`UNIQUE (document_id, scope_columns) WHERE deleted_at IS NULL`) | `documentDataSourceService.link` |
| Mode change | state-based (`UPDATE ... WHERE mode <> :new_mode`) | `documentDataSourceService.changeMode` |
| Add-to-Knowledge promote | key-based (`UNIQUE (file_id, document_id) WHERE deleted_at IS NULL` on a new `document_promotion_audit` row inside the transaction) | `documentPromotionService.promoteFile` |
| Retrieval event | non-idempotent (intentional; per-run terminal event, single writer) | `retrievalObservabilityService.emit` |

### 10.2 Retry classification

| Operation | Class | Boundary |
|---|---|---|
| Add version | guarded | predicate above |
| Insert chunk | safe | dedup'd by unique index |
| Insert link row | guarded | unique index |
| Mode change | safe | optimistic predicate |
| Promote file | guarded | inside `withOrgTx`; unique index on the audit row |
| Re-embed sweep | safe | per-chunk idempotency |

### 10.3 Concurrency guard for racing writes

| Race | Guard | Losing-caller response |
|---|---|---|
| Two callers promote the same file simultaneously | UNIQUE on `(file_id) WHERE ...` on the audit row; catch `23505` | 409 Conflict + "Already promoted to document {id}" + the existing document id in the body |
| Two callers change document mode at the same time | Last-write-wins is acceptable (mode is a property, not a state machine); no guard needed | the second response carries the final mode |
| Two callers add the same document at the same scope | UNIQUE on `(document_id, scope_columns) WHERE deleted_at IS NULL`; catch `23505` | 409 Conflict + "Already linked at this scope" |
| Two embedders run on the same chunk | UNIQUE on `(version_id, chunk_index)` | safe, one wins, the other no-ops |
| Two re-embed jobs touch the same chunk | same UNIQUE | same |

### 10.4 Terminal event guarantee

Every retrieval has exactly one terminal observability event (`retrieval.summary`) per run, emitted once at the end of the run regardless of success / partial / failed status. No intermediate retrieval events.

### 10.5 No-silent-partial-success

The "always-available bypass-but-still-budget-bound" rule (brief Â§3) is the trickiest place. The contract is:

- If `sum(alwaysAvailable token_cost) > context_budget`, the system flags the run as `degraded` in the retrieval result and emits operator guidance via `RetrievalResult.alwaysAvailable[i].degraded = true`. It does NOT silently truncate; the agent execution continues but the run-summary surface marks it. Operator-facing copy in the UI reads "Some always-available documents could not be loaded due to context limits."
- Threshold-passing documents that don't fit the remaining budget are tracked in `rejected.aboveThreshold[i].reason = 'budget_exhausted'` so the "why wasn't this loaded?" drill-in (Â§11) can surface the truth.

| Constraint | HTTP status | Body |
|---|---|---|
| `reference_documents_org_name_uq` violation | 409 Conflict | `{ error: "DOCUMENT_NAME_EXISTS", existingDocumentId: "..." }` |
| `reference_document_data_sources_unique_per_scope` | 409 Conflict | `{ error: "DOCUMENT_ALREADY_LINKED", existingLinkId: "..." }` |

| `document_promotion_audit_unique_per_file` | 409 Conflict | `{ error: "FILE_ALREADY_PROMOTED", existingDocumentId: "..." }` |

### 10.7 State machine closure

The `mode` enum is closed: `auto | always_available | reference_only`. Adding a new mode requires a spec amendment. The `source` enum is also closed (Â§4.4).

`summary_stale` is a boolean (`true` after version write, `false` after summarisation completes). Forbidden transitions: none; the column is a flag, not a multi-state machine.

`reference_document_data_sources` rows have a soft-delete model (`deleted_at IS NULL` filter). Forbidden transitions: re-creating a deleted link must use a fresh row, not un-deleting (audit-friendly). The unique index applies only to `WHERE deleted_at IS NULL`.

## 11. Retrieval observability

Brief Â§9, operator-facing principle, not internal-only telemetry. Phase 7.

### 11.1 What gets recorded

Per the `RetrievalResult` contract (Â§6.2). Every per-run retrieval emits one `retrieval.summary` event. No intermediate events.

### 11.2 Operator-facing surfaces

Each one consumes the same event stream:

1. **"Why was this loaded?" tooltip** on document rows in Agent Data Sources and Documents tabs. Reads the last N runs that used this document; renders plain-language summary ("Loaded for CRM enrichment and lead scoring tasks").
2. **"Why wasn't this loaded?" drill-in** on a per-run trace (existing run-trace surface). Renders `rejected.aboveThreshold` with reasons.
3. **"Loaded in N of last 30 runs" relevance bar** on document rows. Backed by an aggregate query over the event stream.
4. **Token usage per workspace and per agent** on the spending surface (already exists; this build adds the per-document attribution).

### 11.3 Internal telemetry surfaces

Engineering-only:

- Relevance threshold tuning histograms (per organisation).
- Embedding model evaluation (RAG quality benchmarking).
- Retrieval drift detection (week-over-week candidate-pool size and rejection-reason distribution).

These surfaces are not gated behind the new `knowledge` permission; they're admin-only.

## 12. Tenant isolation invariants

Brief Â§10. The hard rules:

1. **Cross-org boundary is absolute.** `withOrgTx` is the only entry to retrieval reads. The `RLS_PROTECTED_TABLES` manifest enforces FORCE RLS on every new table. The CI gate `verify-rls-coverage.sh` blocks merges that miss a policy.
2. **Sub-account isolation is enforced at candidate-pool construction.** The pool builder runs `WHERE subaccount_id = :ctx_subaccount_id OR subaccount_id IS NULL` for sub-account-scoped reads.
3. **Authorization filtering occurs BEFORE semantic retrieval.** No path runs a similarity search and then filters; the candidate pool IS the result of an authorization-scoped query.
4. **Embeddings are tenant-scoped.** `reference_document_chunks.organisation_id` is on every row (denormalised from the parent document for RLS-policy locality).

## 1. What this brief is and the question it answers

Synthetos agents need access to the right knowledge at the right time. Today, an operator who wants their agent to "know" something has to upload a document, link it to the agent, decide whether it loads into every prompt or only on demand, and manage that linkage forever as documents come and go. This is the *eager-vs-lazy* maintenance burden. It scales badly â€” at 50 documents per agent, no human can sensibly decide which ones load into which task.

The question this brief answers: **how does Synthetos automatically deliver the right knowledge to the right agent at the right moment, without forcing the operator to configure it?**

The answer, in one sentence: **the operator links a document to a scope; the system handles the rest** â€” generating a summary, embedding it, retrieving only the documents semantically relevant to each task at run time, while honouring sensible overrides for the rare cases where the system should always or never load a document.

This is a brief, not a spec. It captures the design decisions reached through reviewer iteration. Engineering detail (schema, API contracts, migration, etc.) belongs in the spec that follows.

## 2. The problem we're solving

The current model has three failure modes:

1. **The maintenance tax.** Every document linked to an agent has a "loading mode" (eager or lazy). Choosing wrong wastes tokens (eager docs that aren't relevant) or wastes runs (lazy docs the agent doesn't know to fetch). At small scale this is annoying; at scale it's unworkable.
2. **The relevance gap.** Memory blocks already use semantic retrieval â€” the system loads only the memory entries relevant to the current task. Documents don't. They're loaded by linkage alone, regardless of whether they have anything to do with what the agent is currently doing. Two parallel knowledge primitives, two different mental models, one of which is markedly smarter.
3. **The hierarchy gap.** Documents can be linked to a specific agent or scheduled task, but there's no clean way to pin a document to *all agents in a sub-account* or *all agents across an entire organisation.* An agency that wants every client's agents to have access to a brand briefing has to manually link it to each agent. That doesn't scale.

The fix is to extend the proven memory-block retrieval pattern to documents, add the missing scopes, and remove the user-facing eager/lazy concept entirely.

## 3. The new model: Auto, Always available, Reference only

Three modes. The default is Auto. The other two are escape hatches for the rare cases.

### Auto (default)
The document is in the **candidate pool** for the scope it's linked to. When the agent runs a task, the system embeds the task description and runs vector similarity against all candidate documents. Documents above a minimum **relevance threshold** are loaded in score order until either all relevant documents are loaded or the token budget cap is reached.

Critical: **the budget is a cap, not a target.** The system never loads documents to fill the budget. If only one document is relevant, only one loads. If none are relevant, none load. Most tasks load zero to three documents. The cap is a backstop for the rare case where many documents are genuinely relevant.

### Retrieval ordering and invariants

Retrieval is deterministic, in this order:

1. **Authorization filter.** Build the candidate pool from documents the agent is authorized to see (see Â§10). Documents outside that pool never enter retrieval, regardless of similarity.
2. **Threshold filter.** Drop any document whose relevance score is below the minimum threshold.
3. **Rank.** Compute the final ranking score: `final_score = (relevance_score Ã— relevance_weight) + scope_bonus + recency_bonus + operator_pin_bonus`.
4. **Truncate to budget.** Load documents in `final_score` order until the token budget is reached.

**Hard invariants** the implementation must preserve:

- **Scope is a tiebreaker, not a multiplier.** Scope bonuses are bounded and small. They MUST NEVER cause an irrelevant document to outrank a materially more relevant one. A highly-relevant agent-scope document beats a marginally-relevant org-pinned document; an org-pinned document only "wins ties" against documents at similar relevance scores.
- **Always-available documents bypass threshold and ranking.** They load every run. They are the only category that ignores relevance. **They do, however, still participate in overall context budgeting.** If the always-available set alone exceeds safe context limits, the system must fail gracefully and surface operator guidance (rather than silently truncating or unbounded expansion). Always-available is a relevance bypass, not a budget bypass.
- **Reference-only documents never enter the candidate pool.** Their manifest is added to the prompt; their content is fetched by the agent on demand via tool call.
- **Operator pins are bounded too.** A user-pinned document gets a bonus, but the same hard rule applies: the bonus must never cause irrelevant content to outrank relevant content.

### Platform principle: retrieval quality over retrieval volume

This system is intentionally not optimised for "how many documents can we fit." It is optimised for "how few documents can we load while still answering well."

> **Retrieval quality is prioritised over retrieval volume.** The system always prefers loading fewer highly-relevant documents over many marginally-relevant documents.

This shapes every threshold-tuning decision, every UI surface that shows the model's behaviour, and every default that ships. Future improvements (better re-ranking, learned thresholds, cross-encoder rescoring) all serve this principle. Reviewers and spec authors should weight any proposed change against it.

### Always available
The document is loaded into every run regardless of relevance. Use for compliance rules, brand voice guidelines, or any context the agent must always have. Replaces the old "eager" mode for cases where the operator genuinely needs the guarantee.

### Reference only
The document is never auto-loaded. Only its title and a 1-2 sentence summary appear in the prompt as a manifest entry. The agent calls a tool to fetch the full content when it explicitly decides it needs to. Use for huge manuals, codebases, or reference material the agent should look up rather than memorise.

The UI shows all three with descriptive helper text inline, and defaults Auto on every document. 99% of documents stay in Auto and the operator never thinks about loading mechanics again.

## 4. The five-tier hierarchy

Documents can be linked at five scopes, ordered from most specific to least specific. The retrieval engine searches across the union of all scopes the agent has access to.

| Tier | Scope | Use case |
|---|---|---|
| 1. Task instance | A single task run | One-off attachment for a specific question |
| 2. Recurring task | A scheduled task definition (e.g. nightly CRM hygiene) | Knowledge specific to a recurring job |
| 3. Agent | A specific agent | Knowledge this agent always uses |
| 4. Sub-account | All agents in a client workspace | Client-specific knowledge (CRM context, brand, voice) |
| 5. Organisation | All agents across all sub-accounts | Agency-level knowledge (master playbook, agency-wide rules) |

When an agent runs, the candidate pool is the union of documents at every tier the agent is exposed to. The retrieval engine picks the most relevant from across the union.

**Scope is determined by where the document is added.** Adding from the Knowledge page in a sub-account view defaults to sub-account scope; adding from the org view defaults to org-wide; adding from the Agent edit Data Sources tab defaults to agent scope; adding from a Manage Task page defaults to task instance. Operators don't pick a scope from a menu in the common case â€” the surface they're on tells the system. Org-admin operators can promote upward.

When relevant documents come from multiple tiers and the budget is tight, **more-specific scopes win ties** â€” task instance beats recurring task beats agent beats sub-account beats org. Specific knowledge beats general when relevance scores are close.

## 5. Files vs Documents, how they relate

Files and Documents are kept as **separate primitives** because they have fundamentally different roles in how the system works:

- **Documents** are *what an agent knows.* They're loaded into agent context (eager via Always available, smartly via Auto, on-demand via Reference only). They have token costs, scopes, and retrieval rules. They live in the database as durable text content.
- **Files** are *what an agent has produced or used.* They're artifacts â€” CSVs, images, generated reports â€” tied to specific runs. They're not in the agent's prompt unless explicitly cited. They live in object storage (S3/R2) with metadata in the database.

Different lifecycles, different costs, different mental models. Forcing them into one storage layer would either bloat the database with binaries or destroy query power on knowledge content. Keeping them separate is the right architectural call.

But the user mental model should be **"stuff in my workspace,"** not *"where is my stuff."* So both surfaces live on the **Knowledge page** as sibling tabs, and the bridge between them is a one-click promotion flow (Â§6).

The Knowledge page tab strip becomes:

- **Authored memory** â€” manually written facts, rules, preferences (existing)
- **Auto-memory** â€” agent-extracted memory pending review (existing)
- **Documents** â€” durable knowledge documents with scope and mode (existing, refreshed for new model)
- **Files** â€” agent-produced artifacts across all runs (new tab, surfaces what already exists in the database)

A file becomes a document via the **Add to Knowledge** action (Â§6). This is the only direction we support; promoting an artifact to durable knowledge. The reverse (document to file) is uncommon and unsupported in v1.

### What bundles are, and what they aren't

The Knowledge â†’ Documents tab has a *Bundles* sub-tab (existing primitive). To prevent ambiguity:

> **Bundles are organisational and operational groupings only. Retrieval still occurs at the document level. Bundles do not alter semantic ranking behaviour.**

A bundle is a curatorial collection of documents you can attach to an agent or task in one action, or clone across sub-accounts. A document inside a bundle is retrieved per its own mode and scope; the bundle does not modify relevance scoring, threshold behaviour, or token budget rules. Mode is set on the document, not on the bundle. This is enforced in the UI (Bundle Edit modal shows mode chips read-only) and called out in the brief here so it does not drift during spec.

## 6. The Add to Knowledge flow

The user clicks the three-dot menu on a file row â†’ **Add to Knowledge.** A modal opens.

**The modal asks for:**

1. **Title** â€” pre-filled from filename, editable.
2. **Content preview** â€” auto-extracted text from the file (when extractable; e.g. PDF text, Markdown, transcript).
3. **Scope** â€” defaults to the surface the file came from (sub-account / org / agent). Org-admin sees "Promote to org-wide" if applicable.
4. **Apply to** â€” defaults to "All agents in scope." Power users can pick specific agents from a multi-select.
5. *(Advanced expander, hidden by default)* â€” option to override mode to *Always available* or *Reference only.* 99% of users skip this; the default Auto handles them.

**On confirm:**

- A new Reference Document is created with the extracted content.
- Document Data Source links are created at the chosen scope.
- The system runs the document through the cheap-LLM summariser (one-time, async) and generates an embedding.
- The source file gets a **"Linked: [Document name]"** indicator on its row.
- The source file is automatically marked durable (no TTL expiry) so the link remains valid.

**Auto-suggestion path** (parallel feature): when an agent produces a text-heavy output (markdown summary, transcript, structured report), the agent emits a *"candidate for knowledge"* hint. The file appears in the **Auto-memory tab** alongside agent-extracted memory entries. The user clicks Approve and it's promoted via the same flow with sensible defaults. No new UI surface; this reuses the existing Auto-memory pending-review pattern.

This means most knowledge accumulation happens via review-and-approve, not manual addition. The operator stays in control without doing the curation work.

### System-generated documents (future-native concept)

The promotion flow above covers *agent-produced files become durable documents*. The architecture should reserve conceptual space for a stronger version: **agents synthesizing reusable knowledge directly**, without going through a file step.

Example future flow:
- An agent completes a research task across many runs.
- The agent identifies a recurring pattern worth keeping ("Acme Corp's primary buyer always asks about SOC 2 first").
- The agent emits a *knowledge candidate* directly (no intermediate file).
- The candidate appears in the Auto-memory tab for operator approval.
- On approval, it becomes a durable Reference Document with the standard scope and mode controls.

We are NOT building this in v1. The brief reserves the concept so the schema, the source-provenance taxonomy, and the Auto-memory tab UX can accommodate it without rework when it ships. Concretely: the document source taxonomy includes a future *"synthesised by agent"* badge alongside the current *"From file"*, *"Approved from auto-memory"*, *"Manually authored"*, *"Uploaded"* badges.

## 7. Memory blocks at additional tiers

Memory blocks today support three scopes (org, sub-account, agent). Documents in this brief use five scopes (org, sub-account, agent, recurring task, task instance).

When the shared retrieval engine (Â§8) is built, **memory blocks gain the recurring-task scope** as a free side-effect. This is genuinely valuable: a recurring task running daily for months accumulates patterns specific to that scheduled job (e.g. *"the inbox cleanup agent learned that emails from this vendor are always promotional"*). That knowledge is task-bound, not agent-bound, and there's nowhere clean to put it today.

**Memory blocks do not gain the task-instance scope.** Memory is persistent by definition; task instances are ephemeral. Putting persistent memory at an ephemeral scope creates a mental-model contradiction. What you'd actually want at task-instance level is *"facts the agent learned during this run"* â€” which is exactly what Auto-memory already captures.

Final tier matrix:

| Tier | Memory blocks | Documents |
|---|---|---|
| Org | Yes (existing) | Yes (new) |
| Sub-account | Yes (existing) | Yes (existing) |
| Agent | Yes (existing) | Yes (existing) |
| Recurring task | **Yes (new)** | Yes (existing) |
| Task instance | No | Yes (existing) |

Documents support five tiers because they can be one-off attachments to a single run; memory blocks support four because persistent knowledge at an ephemeral scope doesn't make sense.

## 8. The shared retrieval engine

Today, memory blocks have a relevance ranker buried in the memory-block service. Documents have nothing equivalent; they're loaded by linkage alone.

The build extracts the existing ranker into a small shared **RetrievalService** that:

- Takes a set of candidate items (memory blocks or documents) with embeddings already computed.
- Applies the relevance threshold and ranks by similarity to the current task context.
- Honours the token budget cap as a backstop.
- Returns the items to load, in priority order.

Memory blocks and documents each have their own data-source layer (different tables, different scoping rules) but plug into the same ranker. The infrastructure that already exists; OpenAI `text-embedding-3-small`, pgvector storage, cosine similarity, stays exactly as it is. Documents adopt it.

This abstraction is small but high-leverage:

- Single source of truth for relevance ranking, threshold tuning, and budget enforcement.
- Future improvements (better re-ranking, cross-encoder rescoring, learned threshold tuning) ship to both primitives at once.
- New knowledge primitives in the future (e.g. structured CRM entities) can plug into the same engine.

Cost per task at scale: under one cent of embedding overhead per run. End-to-end retrieval latency: under 100ms added per run. Already proven by memory blocks in production today.

### What gets embedded (chunking model)

Critical decision the spec must lock: **retrieval operates on chunked semantic units, not whole-document embeddings.**

- During ingestion, documents are split into semantically coherent chunks. Chunk size and boundaries are tuned for the embedding model in use; the spec author owns the tuning.
- Each chunk gets its own embedding and is independently retrievable.
- **Retrieval operates at chunk granularity. The system may inject only the relevant portions of a document, not necessarily the entire document.** A chunk match identifies the relevant content; the system decides whether to inject just that chunk (with optional surrounding chunks for coherence) or the full document, based on size and relevance. The default is *inject the relevant portions, not the whole document* â€” full-document injection is the exception, not the rule.
- Whole-document embeddings are NOT used. They scale poorly (embedding quality drops as documents get longer) and produce worse retrieval.

This decision determines whether the system handles a 50-page playbook gracefully or chokes. The spec author should treat chunking as a first-class engineering concern with explicit tests for boundary conditions (very short documents, very long documents, mixed-content documents).

**For the *Reference only* mode** (large manuals, codebases): chunks are still indexed for the manifest preview, but the full content is fetched on-demand by the agent rather than auto-loaded. This preserves the manifest-only behaviour while still benefiting from chunked semantic indexing.

## 9. Retrieval observability

Without this, debugging is guesswork and operators won't trust the system. **First-class concept, not an afterthought.**

> **Operator-facing principle:** *Retrieval behaviour should be inspectable by operators. The system should explain why documents were or were not loaded.* Observability is not just internal telemetry; it is a product surface.

For every agent run, the system must record:

- **Candidate pool size** for each tier (org / sub-account / agent / recurring task / task instance).
- **Retrieved documents** with their final score, mode, and tier.
- **Rejected documents that were above threshold** but didn't make it into context, with the reason: budget exhausted, lower-scoring at tie, etc.
- **Rejected documents below threshold** (counts only, not full list): how many candidates were filtered before ranking.
- **Token contribution per document** loaded.
- **Retrieval score per document** loaded.
- **Reference-only fetches** the agent invoked during the run via tool call.

This data powers four operator-facing surfaces and one internal one:

- **"Why was this loaded?" tooltip** on each document row in Agent Data Sources and Documents tabs. Hover shows the recent task contexts where this document scored above threshold, in plain language ("Frequently retrieved for CRM enrichment and lead scoring tasks").
- **"Why wasn't this loaded?"** drill-in on a specific run trace, showing which documents were considered, scored, and rejected, with reasons.
- **"Loaded in N of last 30 runs" relevance bar** (already in mockups) backed by this data.
- **Token usage per workspace and per agent**, for the spending / budget surface.
- **Internal telemetry** for the engineering team: relevance threshold tuning, retrieval drift detection, embedding model evaluation, RAG quality benchmarking.

The data feed is structured (per-run JSON in the LLM observability ledger that already exists in the codebase). Spec author should choose between: (a) write to existing ledger with new event types, or (b) dedicated `retrieval_events` table. Recommend (a) for v1, can split later if volume justifies.

Without retrieval observability, four things go wrong: operators don't trust retrieval, debugging is impossible, token optimisation is guesswork, support burden balloons. This section is non-negotiable for v1.

## 10. Tenant isolation invariants

This is a security concern, not just an architectural concern.

**Hard invariants** the implementation must preserve:

> **Authorization filtering occurs BEFORE semantic retrieval.** A document outside the agent's authorized scope MUST NEVER enter the retrieval candidate pool, regardless of semantic similarity.

Concretely:

- **Cross-org boundary is absolute.** Retrieval never crosses organisations. An agent in Org A cannot see Org B's documents under any condition.
- **Sub-account isolation is enforced at the candidate-pool level.** An agent in sub-account X with sub-account-scoped documents available cannot see sub-account Y's documents.
- **Org-pinned documents flow downward only.** An org-pinned document is visible to all sub-accounts in that org. The reverse is impossible.
- **Embeddings are tenant-scoped.** The vector store schema must enforce org_id (and sub-account_id where relevant) on every embedding row. RLS policies apply before similarity search.
- **The candidate pool is constructed from authorization first, then filtered by relevance.** The order matters: never retrieve, then filter; always filter, then retrieve.

This applies equally to memory blocks and to documents (both run through the shared RetrievalService). The spec must include explicit RLS / authorization tests that verify cross-tenant queries return empty pools, never partial results.

## 11. Lifecycle, versioning, and re-embedding

Knowledge artifacts are not static. The system must handle change over time without requiring operator intervention.

**Lifecycle philosophy** the spec must build to:

> **Knowledge artifacts are versioned and re-indexable. Embedding generations are replaceable infrastructure, not permanent truth. The system supports background re-embedding and retrieval regeneration without operator intervention.**

Concretely, the system handles:

- **Document edited.** Re-summarise (cheap LLM) and re-embed in the background. Retrieval continues using the prior version until the new version is ready, then swaps atomically. UI shows a `summary_stale` flag during the brief catch-up window.
- **Source file changes** (e.g. underlying CSV updated). Same as document edit.
- **Bundle changes.** Bundle composition affects which documents are pulled into a scope; the affected documents' linkage is updated, no re-embedding needed.
- **Embedding model upgrades.** When OpenAI ships text-embedding-4-small or we choose to migrate, the system supports a background re-embedding job that walks every document and replaces the vectors. No operator intervention. Old embeddings remain valid until the new ones are ready, then swapped.
- **Extraction quality improvements.** As text extraction from PDFs / images improves, documents derived from those sources should support being re-extracted and re-embedded without losing their linkage history or scope settings.

What the spec must include:

- A `summary_stale` flag (or equivalent) that surfaces transitional states.
- A background job that re-embeds documents whose source has changed, on a configurable cadence.
- A migration path for embedding-model upgrades.
- Clear ownership: re-embedding is the system's job, not the operator's. The UI never asks the user to "click here to update embeddings."

This is the difference between v1 working at scale and v1 becoming a pile of operational debt the team has to manually maintain.

## 12. What this enables (Phase 2, agent workspace)

This work ships as Phase 1 of the broader agent workspace strategy described in `docs/agent-cloud-compute-dev-brief.md`. It is valuable in its own right â€” smarter knowledge retrieval improves every agent run regardless of whether the workspace UI ever ships. But it is also a prerequisite for several Phase 2 surfaces:

- The **per-agent Data Sources tab** becomes a clean view of what an agent knows, with relevance signals and easy linking. Today it's complicated by eager/lazy decisions; once those are gone, it simplifies dramatically.
- The **Add to Knowledge flow on artifacts** lets an agent's run output flow back into its persistent knowledge in one click â€” directly supporting the "agent embodiment" surface the workspace UI delivers.
- **Org-wide and sub-account-wide pinning** unlocks agency-tier value: brand briefs, agency playbooks, and shared rules flow down to every client without per-agent linkage. This is a structural fit for the multi-tenant model that differentiates Synthetos from VM-per-agent competitors.
- **Memory blocks at the recurring-task tier** lets long-running scheduled work accumulate task-bound learning, which surfaces naturally in the agent workspace UI later.

In short: shipping Phase 1 first means Phase 2 mockups are simpler, the agent workspace surface has better content to display, and several Phase 2 decisions get easier.

## 13. Decisions made

|---|---|---|
| 1 | Default mode for documents | **Auto** â€” embedding-based retrieval, relevance-thresholded, budget-capped. |

| 4 | Hierarchy tiers (documents) | Five tiers: org, sub-account, agent, recurring task, task instance. |
| 5 | Hierarchy tiers (memory blocks) | Four tiers: gain recurring task; skip task instance. |

| 7 | Embedding provider | OpenAI `text-embedding-3-small` (existing, already in production for memory). No change. |
| 8 | Files vs Documents | Stay separate primitives. Unified UX on the Knowledge page (sibling tabs). One-click promotion via Add to Knowledge. |
| 9 | Auto-suggestion of files-to-knowledge | Yes â€” reuses the existing Auto-memory pending-review pattern. |

## 14. Spec-risk areas to watch

The spec author and implementer should treat these as the messy areas. Each one is more likely than average to require iteration, careful design, and explicit testing.

| Area | Risk |
|---|---|
| Chunking strategy (Â§8) | Huge downstream impact on retrieval quality. Wrong choice scales poorly. |
| Relevance threshold tuning (Â§3) | Threshold drift over time as document corpora grow. Needs telemetry-driven tuning. |
| Cross-tier ranking (Â§3) | Non-deterministic behaviour if scope bonus is unbounded. Hard invariant in Â§3 must be tested. |
| Reference-only mode UX | Risk that agents ignore the manifest and don't fetch when they should. Tool-call behaviour needs prompt-level reinforcement. |
| Massive documents (>50K tokens) | Potential token explosions if Reference-only mode isn't selected correctly. Default behaviour for large uploads needs care. |
| Bundle semantics (Â§5) | User confusion: are bundles for retrieval, organisation, security, or all? Brief locks the answer (organisational only); spec must enforce. |
| Re-embedding lifecycle (Â§11) | Operational debt if not built in from v1. Stale embeddings produce silent retrieval drift. |
| Retrieval observability (Â§9) | Without it, debugging is impossible. Build it day one, not as a v1.1 follow-on. |
| Tenant isolation (Â§10) | Security risk if RLS / authorization order is wrong. Must be explicitly tested with cross-tenant query attempts. |
| "Always available" abuse | Cost blowout if users mark many documents as Always available. v1 has soft warning only; revisit hard caps if real abuse emerges. |
| Reference-only as second retrieval system | Long term, "Reference only" tool-fetched docs and Auto retrieved docs may drift apart in quality, ranking, and observability. The spec should keep both paths on shared infrastructure: shared chunk store, shared ranking logic, shared authorization, shared observability. Otherwise operators eventually ask *"why can the tool find it but auto retrieval can't?"* |

The spec author should produce explicit test cases for each of these. Where ambiguity remains after the spec, flag back to the brief author rather than guessing.

## 15. Out of scope for v1

- **Document version-aware retrieval.** A document can be revised; the system uses the latest version. Diff-based retrieval (load the old version when relevant context is older) is not in v1.
- **Cross-encoder re-ranking.** Pure cosine similarity for v1; future re-ranking with a small LLM if telemetry shows we need it.
- **Per-agent or per-user Always-available token budget caps.** Soft warning in v1; hard caps come later if abuse emerges.
- **Document-level search UI.** Operators will be able to see what's linked but not run ad-hoc semantic search. That's a future feature if demand surfaces.
- **Document â†’ File reverse promotion.** Uncommon, not supported.
- **Promotion-request workflow** for sub-account admins to request org pinning. v1 is org-admin-only; promotion flow comes later if needed.
- **Multilingual retrieval.** Embeddings work across languages but retrieval quality is best in English. Future improvement.
- **Custom embedding provider per workspace.** All workspaces use OpenAI text-embedding-3-small for v1.

## 16. Success criteria for v1

A non-technical operator can:

- Add a document to a sub-account (or org-wide if they're an org admin) without choosing a loading mode. The system handles retrieval automatically.
- Click "Add to Knowledge" on an agent-produced file and have it promoted to a durable document in three clicks or fewer.
- See, on the Agent edit Data Sources tab, what documents the agent has access to â€” with a relevance signal indicating how often each is actually used.
- Pin a document to the entire organisation and have every client's agents pick it up automatically.
- Add 50 documents to a sub-account without any of them blocking or bloating their agents' runs. The system loads only the relevant ones per task.

A reasonable internal observer can:

- Verify that the document and memory-block retrieval engines share a single ranker (no duplicated logic).
- Verify that retrieval latency stays under 100ms added per run.
- Verify that telemetry captures relevance scores, loaded documents, and token costs per run for tuning.
- Verify that documents flagged Always available always load and Reference only never auto-load.

The competitive frame writes itself: *"Your agents see the right knowledge at the right moment, automatically. You curate; we route."*

## 17. UI patterns established through mockups

Three rounds of mockup feedback (May 2026) established a set of UI decisions for this feature. They are captured here so the spec author and the implementer don't re-litigate them, and so future surfaces in the product can adopt the same patterns. Many of these have been promoted into `docs/frontend-design-principles.md` as recurring patterns.

**Decisions about controls and surfaces:**

1. **Mode is changed via the three-dots menu, not via an inline dropdown.** Mode rarely changes after the initial choice. Inline dropdowns invited fiddling and added visual noise. The menu has a *"Change mode â€º"* item that opens a flyout with Auto / Always available / Reference only.
2. **Three-dots menus stay short** (6 to 8 items max). Grouped sub-options (mode, scope) collapse into single items with chevron flyouts, not expanded inline sections.
3. **"Open document" and "Edit details" merge into one action.** Row-click opens the detail modal; the menu has *Edit*, never both.
4. **Source / origin badges only appear for non-default cases.** "Manually authored" is the default and gets no badge. *"From file"*, *"Approved from auto-memory"*, *"Uploaded PDF"* are non-default and do get badges.

**Decisions about modals:**

5. **The Add to Knowledge modal is minimal by default.** Title, content preview, and (for org admins only) scope. Available-to and mode override live behind a collapsed *Advanced* expander. Most users complete the action with default everything.
6. **Scope picker is hidden for non-org-admins.** The control is absent from the DOM, not disabled. Org admins see it with a small *"Org admin only"* pill on the field label.
7. **Document Detail modal is a two-column layout.** Main column has editable fields (title, content, mode, scope, available to). Side panel has read-only metadata (tier, size widget, created info, linked agents with usage bars).
8. **Bundle Edit modal shows per-document mode chips as read-only.** Mode is set on each document, not on the bundle. This is enforced by the UI and explicitly called out (dismissable) in the modal.

**Decisions about token / cost / size information:**

9. **Token counts are never shown by default.** No per-document token counts, no per-run cost tiles, no embedding-size detail in user-facing surfaces.
10. **Size is shown only as a warning chip when over recommended.** A "âš  Large document" chip appears on the doc name when a document exceeds the recommended size threshold. The chip carries the action: *something here is unusual, you might want to look*.
11. **The Document Detail modal has one expanded size widget** (visual bar + qualitative label like "Small"). Acceptable because users opening the modal are doing so deliberately. Still no raw numbers as the primary information.

**Decisions about page-level chrome:**

12. **Stat tiles capped at 2 per list page.** Each one must be something the operator would act on. Operational metrics (avg per run, cost MTD) do not qualify.
13. **Explainer banners are dismissable.** All info banners ship with a `Ã—` close button that persists per-user. Permanent help copy is not shipped.
14. **Filter pills appear when collection size justifies them** (rough threshold: more than 8 items). Below that, search alone is enough.
15. **Footer notes are avoided** when a banner above already covers the same content.

**Decisions about copy and content:**

16. **No em-dashes in any UI copy or sample data.** Use commas, colons, or rewrite. Applies to mockup data (sample document names) too.
17. **Row sub-text is trimmed to one most-actionable fact.** Mime types, run identifiers, and verbose author / version metadata do not appear in default rows.
18. **"Used by" copy drops the contextual scope when the breadcrumb already provides it.** *"9 agents"* not *"9 agents in Acme Corp"*.

These decisions are visible in the mockups linked in the brief header. The mockups should be considered the authoritative reference for any disagreement between this list and the implementation.

## 1. Goals, non-goals, framing assumptions

### 1.1 Goals

- Replace per-document **eager / lazy** loading with one of three modes: **Auto** (default), **Always available**, **Reference only**. The user-facing eager / lazy concept disappears.
- Add a **shared retrieval engine** that ranks both `memory_blocks` and `reference_documents` against the current task context, applying threshold filtering and budget capping.
- Add the missing **org** and **recurring-task** scopes for documents; add the **recurring-task** scope for memory blocks. Five tiers for documents, four tiers for memory blocks.
- Ship a **chunked-embedding model** for documents (whole-document embeddings are not used).
- Surface **retrieval observability** as an operator product (not internal-only telemetry): "why was this loaded / not loaded" tooltips, per-document relevance signals, per-run trace.
- Surface **Files** as a Knowledge tab and ship the **Add to Knowledge** promotion flow.
- Hard tenant isolation: authorization filtering occurs **before** semantic retrieval.

### 1.2 Non-goals (explicit)

- No new storage backend. Continue with Postgres + pgvector + OpenAI `text-embedding-3-small`.
- No cross-encoder re-ranking. Pure cosine for v1.
- No version-aware retrieval (latest version only).
- No document-level ad-hoc semantic search UI.
- No per-tenant custom embedding provider.
- No backfill of historical documents (dev environment, no data).
- No promotion-request workflow for sub-account admins (org admins are the sole org-pinning authority in v1).
- No new storage layer for files; the Files tab surfaces existing `execution_files` rows.
- No bundle-driven retrieval. **Bundles remain organisational only**, they do not modify ranking, threshold, or budget behaviour. Mode is set on the document, not on the bundle (enforced).

### 1.3 Framing assumptions (cross-checked against `docs/spec-context.md`)

- `pre_production: yes`, `live_users: no`, `feature_stability: low`. Breaking schema changes acceptable.
- `testing_posture: static_gates_primary`, `runtime_tests: pure_function_only`. The retrieval ranker, threshold logic, scoping logic, and re-embedding decision are pure-function tested. No frontend / API-contract / E2E tests against the app.
- `feature_flags: only_for_behaviour_modes`. No rollout-gate flags introduced. The legacy `loadingMode: 'eager' | 'lazy'` column on `agent_data_sources` is **retired** as part of this build (see Â§15).
- `staged_rollout: never_for_this_codebase_yet`. Direct cutover.
- Existing primitives prevail: `memoryBlockRetrievalServicePure`, `referenceDocumentService`, `executionFiles`, the bundle stack, `withOrgTx`, `RLS_PROTECTED_TABLES`, `withBackoff`, `createWorker`. No new service layers without a "why-not-reuse" paragraph.

### 3.3 Invented (genuinely new primitives)

Three new primitives. Each has a one-paragraph justification.

#### 3.3.1 `reference_document_chunks` (new table)

**Why:** documents are chunked for retrieval; each chunk gets its own embedding and is independently retrievable (brief Â§8). Memory blocks embed the whole block because blocks are short-by-design; documents can be 50+ pages and require chunk-level granularity. Reusing `memory_blocks` for chunks would conflate two distinct lifecycles (a chunk is derived; a block is authored).

#### 3.3.2 `reference_document_data_sources` (new join table, link / scope / mode)

**Why:** the existing bundle-attachment table `document_bundle_attachments` only supports bundle-level attachment with `attachment_mode: 'always_load' | 'available_on_demand'`. The brief requires direct (non-bundle) attachment of a document to a scope (org / sub-account / agent / recurring task / task instance) with the **document's own mode** (Auto / Always available / Reference only). Reusing `agent_data_sources` would force documents through the HTTP-source ingestion model; reusing `documentBundleAttachments` would couple every document to a synthetic single-document bundle and contradict Â§5 of the brief ("bundles are organisational, not retrieval-altering"). A new join table is correct.

#### 3.3.3 `retrievalServicePure` (new pure-function service)

**Why:** the existing `memoryBlockRetrievalServicePure` is memory-block-specific (priority enum, divergence flags, ownerAgent semantics). The brief Â§8 requires a single ranker that takes "a set of candidate items with embeddings" and returns "items to load, in priority order". The right move is to **extract the generic part** of the memory-block ranker into a new pure module with a polymorphic input shape (`RetrievalCandidate`), and have `memoryBlockRetrievalServicePure` plus the new `documentRetrievalServicePure` both consume it. That preserves the memory-block-specific logic where it belongs (block-side filters before ranking) while concentrating the ranking algorithm in one place. Future cross-encoder re-ranking, learned thresholds, and new knowledge primitives all benefit.

Every retrieval has exactly one terminal observability event (`retrieval.summary`) per run, emitted once at the end of the run regardless of success / partial / failed status. No intermediate retrieval events.

### 10.5 No-silent-partial-success

The "always-available bypass-but-still-budget-bound" rule (brief Â§3) is the trickiest place. The contract is:

- If `sum(alwaysAvailable token_cost) > context_budget`, the system flags the run as `degraded` in the retrieval result and emits operator guidance via `RetrievalResult.alwaysAvailable[i].degraded = true`. - Threshold-passing documents that don't fit the remaining budget are tracked in `rejected.aboveThreshold[i].reason = 'budget_exhausted'` so the "why wasn't this loaded?" drill-in (Â§11) can surface the truth.

## 13. Lifecycle, versioning, re-embedding

Brief Â§11. Knowledge artefacts are versioned and re-indexable; embedding generations are replaceable infrastructure.

### 13.1 Document edit

1. New version row written (`reference_document_versions`); `current_version_id` flipped at the end.
2. `summary_stale = true` immediately on version write.
3. `document:summarise` job enqueued.
4. `document:chunk-embed` job enqueued.
5. Retrieval continues using the prior version's chunks until the new version's chunks are committed.
6. Atomic swap: when the chunking job completes, it updates `reference_documents.last_chunked_at`. Until then, retrieval queries chunk rows `WHERE version_id = current_version_id`. Old version's chunks are retained (history) but excluded.

### 13.2 Source-file change (e.g. underlying CSV updated)

Same as document edit, triggered by the existing version-write hook in `referenceDocumentService`.

### 13.3 Embedding model upgrade

When OpenAI ships a new model (or operator chooses to migrate):

1. New unique index on `(version_id, chunk_index, embedding_model)` ensures parallel embedding generations don't collide.
2. `document:reembed` job sweeps all chunks where the embedding model is the old version.
3. Reads continue from the old embedding until the new one is ready, swap is per-chunk-row; no operator action.

### 13.4 Bundle changes

Bundle composition affects which documents are pulled into a scope. The link rows are in `reference_document_data_sources` (direct attachment) and `document_bundle_attachments` (bundle attachment); neither requires re-embedding. The retrieval engine treats both as candidate-pool sources and deduplicates by `documentId`.

### 13.5 Document deletion

Soft delete on `reference_documents.deleted_at`. Cascade soft-delete to `reference_document_data_sources` (link rows). Chunks and version rows are retained for audit. Affected agents see a notice on their Data Sources tab (UI surface in Phase 6).

The spec has multiple load-bearing inconsistencies: missing scope coverage in the creation contract, incompatible re-embedding uniqueness rules, sequencing that conflicts with the source brief, and undefined enforcement for key invariants. These issues are material enough that the architecture should not be considered internally consistent yet.

Full review comments:

- [P1] Extend Add-to-Knowledge to express recurring-task and task-instance scope — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:371-376
  The spec’s data model advertises five document tiers, and the brief says adding from Manage Task should default to task-instance scope, but this contract only allows `organisation | subaccount | agent`. As written, the main file-to-document creation flow cannot create recurring-task or one-off task-instance attachments, so two of the five promised scopes have no API surface.

- [P1] Reconcile chunk uniqueness with parallel embedding generations — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:627-629
  Earlier sections make chunk writes/idempotency depend on `UNIQUE (version_id, chunk_index)`, but this section requires old and new embedding generations to coexist during re-embedding and introduces `UNIQUE (version_id, chunk_index, embedding_model)`. Those designs are incompatible on one table: the original unique key prevents inserting the replacement generation, so the promised “read old until new is ready” swap cannot work as specified.

- [P1] Do not sequence retrieval observability after the v1 feature cutover — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:83-83
  The brief makes retrieval observability a day-one, non-negotiable part of v1, and its success criteria depend on per-run telemetry. By placing all observability surfaces in Phase 7 while earlier phases are described as independently shippable, the spec allows the feature to land without the trust/debugging capability the source brief requires.

- [P2] Pick one transaction boundary for Add-to-Knowledge side effects — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:380-380
  This section says promotion atomically inserts the document, updates `execution_files.expiresAt`, and enqueues summarise/embed jobs inside `withOrgTx`, but §5.5/§8 define promotion as `inline transaction + queued finalise`, with `document:promotion-finalise` marking the file durable after commit. Those are materially different failure modes, so the spec currently leaves durability, retry, and recovery semantics undefined for the promotion path.

- [P2] Specify authorization filters for agent, recurring-task, and run scope — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:595-598
  The tenant-isolation rules only spell out org and sub-account predicates. For the newly introduced same-org scopes, there is no named mechanism equivalent to `agent_id = currentAgent`, `scheduled_task_id = currentScheduledTask`, or `task_instance_id = currentRun`, so a broad candidate-pool query could still leak another agent’s task-local attachments while satisfying every invariant listed here.

- [P2] Remove the server feature-mode fallback from phase sequencing — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:483-483
  The framing assumptions explicitly say this codebase does direct cutovers and does not introduce rollout-gate flags, but this paragraph recommends shipping Phase 5 behind a `server feature mode` before Phase 4. In this repo’s stated testing posture, that creates an extra partial-product branch the spec has already ruled out.

- [P2] Add an enforcement mechanism for the one-terminal-event invariant — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:538-538
  `Exactly one terminal observability event per run` is load-bearing for the aggregates in §11, but §10.1 also declares event emission non-idempotent and relies only on a `single writer`. If run finalisation retries after emitting, the same run will be counted twice, so this invariant needs a concrete dedupe key or upsert rule instead of a convention.

- [P3] Add `document_promotion_audit` to the file inventory lock — C:\files\Claude\automation-v1\tasks\builds\auto-knowledge-retrieval\spec.md:512-512
  §5 claims to be the single source of truth for every touched table and migration, but the idempotency/concurrency design here depends on a new `document_promotion_audit` table that is not listed anywhere in the inventory. That drift is risky because an implementation following §5 can miss the very table the rest of §10 assumes exists.
