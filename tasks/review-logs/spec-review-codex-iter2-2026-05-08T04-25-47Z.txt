## 1. Goals, non-goals, framing assumptions

### 1.1 Goals

- Replace per-document **eager / lazy** loading with one of three modes: **Auto** (default), **Always available**, **Reference only**. The user-facing eager / lazy concept disappears.
- Add a **shared retrieval engine** that ranks both `memory_blocks` and `reference_documents` against the current task context, applying threshold filtering and budget capping.
- Add the missing **org** and **recurring-task** scopes for documents; add the **recurring-task** scope for memory blocks. Five tiers for documents, four tiers for memory blocks.
- Ship a **chunked-embedding model** for documents (whole-document embeddings are not used).
- Surface **retrieval observability** as an operator product (not internal-only telemetry): "why was this loaded / not loaded" tooltips, per-document relevance signals, per-run trace.
- Surface **Files** as a Knowledge tab and ship the **Add to Knowledge** promotion flow.
- Hard tenant isolation: authorization filtering occurs **before** semantic retrieval.

### 1.2 Non-goals (explicit)

- No new storage backend. Continue with Postgres + pgvector + OpenAI `text-embedding-3-small`.
- No cross-encoder re-ranking. Pure cosine for v1.
- No version-aware retrieval (latest version only).
- No document-level ad-hoc semantic search UI.
- No per-tenant custom embedding provider.
- No backfill of historical documents (dev environment, no data).
- No promotion-request workflow for sub-account admins (org admins are the sole org-pinning authority in v1).
- No new storage layer for files; the Files tab surfaces existing `execution_files` rows.
- No bundle-driven retrieval. **Bundles remain organisational only**, they do not modify ranking, threshold, or budget behaviour. Mode is set on the document, not on the bundle (enforced).

### 1.3 Framing assumptions (cross-checked against `docs/spec-context.md`)

- `pre_production: yes`, `live_users: no`, `feature_stability: low`. Breaking schema changes acceptable.
- `testing_posture: static_gates_primary`, `runtime_tests: pure_function_only`. The retrieval ranker, threshold logic, scoping logic, and re-embedding decision are pure-function tested. No frontend / API-contract / E2E tests against the app.
- `feature_flags: only_for_behaviour_modes`. No rollout-gate flags introduced. The legacy `loadingMode: 'eager' | 'lazy'` column on `agent_data_sources` is **retired** as part of this build (see Â§15).
- `staged_rollout: never_for_this_codebase_yet`. Direct cutover.
- Existing primitives prevail: `memoryBlockRetrievalServicePure`, `referenceDocumentService`, `executionFiles`, the bundle stack, `withOrgTx`, `RLS_PROTECTED_TABLES`, `withBackoff`, `createWorker`. No new service layers without a "why-not-reuse" paragraph.

### 3.3 Invented (genuinely new primitives)

Three new primitives. Each has a one-paragraph justification.

#### 3.3.1 `reference_document_chunks` (new table)

**Why:** documents are chunked for retrieval; each chunk gets its own embedding and is independently retrievable (brief Â§8). Memory blocks embed the whole block because blocks are short-by-design; documents can be 50+ pages and require chunk-level granularity. Reusing `memory_blocks` for chunks would conflate two distinct lifecycles (a chunk is derived; a block is authored).

#### 3.3.2 `reference_document_data_sources` (new join table, link / scope / mode)

**Why:** the existing bundle-attachment table `document_bundle_attachments` only supports bundle-level attachment with `attachment_mode: 'always_load' | 'available_on_demand'`. The brief requires direct (non-bundle) attachment of a document to a scope (org / sub-account / agent / recurring task / task instance) with the **document's own mode** (Auto / Always available / Reference only). Reusing `agent_data_sources` would force documents through the HTTP-source ingestion model; reusing `documentBundleAttachments` would couple every document to a synthetic single-document bundle and contradict Â§5 of the brief ("bundles are organisational, not retrieval-altering"). A new join table is correct.

#### 3.3.3 `retrievalServicePure` (new pure-function service)

**Why:** the existing `memoryBlockRetrievalServicePure` is memory-block-specific (priority enum, divergence flags, ownerAgent semantics). The brief Â§8 requires a single ranker that takes "a set of candidate items with embeddings" and returns "items to load, in priority order". The right move is to **extract the generic part** of the memory-block ranker into a new pure module with a polymorphic input shape (`RetrievalCandidate`), and have `memoryBlockRetrievalServicePure` plus the new `documentRetrievalServicePure` both consume it. That preserves the memory-block-specific logic where it belongs (block-side filters before ranking) while concentrating the ranking algorithm in one place. Future cross-encoder re-ranking, learned thresholds, and new knowledge primitives all benefit.

## 4. Domain model (data layer)

### 4.1 The five tiers (documents)

A document attaches at exactly one of these scopes per link row:

| Tier | Tier ID | Scope key columns on `reference_document_data_sources` |
|---|---|---|
| 1. Task instance | `task_instance` | `task_instance_id` (FK to `agent_runs.id` or successor) |
| 2. Recurring task | `recurring_task` | `scheduled_task_id` |
| 3. Agent | `agent` | `agent_id` |
| 4. Sub-account | `subaccount` | `subaccount_id` |
| 5. Organisation | `organisation` | (org-only; all scope keys above are null) |

A single document can have multiple link rows (one per scope). Mode is set on the document, not on the link, so all attachments of the same document share the same mode (see Â§6.4 for the source-of-truth rule).

### 4.2 The four tiers (memory blocks)

Memory blocks gain `scheduled_task_id`. They do **not** gain `task_instance_id` (brief Â§7, persistent memory at an ephemeral scope is a category error).

| Tier | Memory blocks | Documents |
|---|---|---|
| Org | yes (existing) | yes (new via `reference_document_data_sources`) |
| Sub-account | yes (existing) | yes (existing path now goes via the new link table) |
| Agent | yes (existing, `owner_agent_id`) | yes (new) |
| Recurring task | **yes (new, `scheduled_task_id`)** | yes (new) |
| Task instance | no | yes (new) |

### 4.3 Modes

Three discriminated values on `reference_documents.mode`:

- `auto` (default), eligible for retrieval; participates in threshold + ranking + budget.
- `always_available`, bypasses threshold and ranking; **does still participate in overall context budgeting** (brief Â§3 hard invariant). If always-available alone exceeds context budget, the system surfaces operator guidance and degrades gracefully (see Â§10.5).
- `reference_only`, never auto-loaded. Only its title + 1-2 sentence summary appears in the prompt manifest. The agent fetches full content via tool call.

The status set is **closed**: adding a new mode value requires a spec amendment (Â§10.7).

### 4.4 Source provenance

`reference_documents.source` tracks where the document came from. Closed enum:

- `manual` (default, user-authored on the Documents tab)
- `from_file` (promoted from `execution_files` via Add to Knowledge)
- `auto_memory_approved` (promoted from auto-memory pending review)
- `synthesised_by_agent` (reserved for future, brief Â§6 "system-generated documents")
- `external` (existing, Google Drive etc., for compatibility)

Brief Â§17 rule: only non-default cases get a UI badge. The default `manual` shows no badge.

### 4.5 Chunks

`reference_document_chunks` rows are derived from `reference_document_versions`. When a new version lands, chunk rows are written by the chunking job and the prior version's chunks are retained until the new version is ready, then atomically swapped (atomic-swap source of truth: Â§6.4 / Â§13).

Chunk size, overlap, and boundary detection are tuned in implementation; the spec mandates only:

- Chunks are semantically coherent (paragraph- or section-aligned, not byte-windowed).
- Each chunk has its own embedding row.
- Each chunk references the parent `document_version_id` so version pinning is preserved.
- Whole-document embeddings are NOT stored on `reference_documents`. Retrieval operates at chunk granularity (brief Â§8).

### 4.6 Files vs Documents

`execution_files` is **not** modified by this build. The Files tab is a read view of existing rows filtered by sub-account / agent / scope. The "Add to Knowledge" action transforms an `execution_files` row into a new `reference_documents` row plus link rows; the file row itself is untouched (the brief mandates the source file becomes durable, `execution_files.expiresAt` is updated to `NULL` or far-future as part of the promotion transaction).

## 6. Contracts

Every cross-boundary data shape is pinned here with at least one concrete example. Source-of-truth precedence is declared where multiple representations exist.

### 6.1 `RetrievalCandidate`

```ts
type RetrievalCandidate = {
  id: string;                    // chunk id OR memory_block id
  kind: 'document_chunk' | 'memory_block';
  organisationId: string;
  subaccountId: string | null;
  agentId: string | null;
  scheduledTaskId: string | null;
  taskInstanceId: string | null;       // documents only; always null for memory blocks
  embedding: number[];                  // vector(1536)
  tokenCount: number;
  // For documents: the chunk's parent document id, version id, and mode.
  parentDocumentId?: string;
  parentDocumentVersionId?: string;
  mode?: 'auto' | 'always_available' | 'reference_only';
  // For memory blocks: priority + isAuthoritative + ownerAgent (already in the existing ranker).
  priority?: 'low' | 'medium' | 'high';
  isAuthoritative?: boolean;
};
```

### 6.2 `RetrievalResult`

```ts
type RetrievalResult = {
  loaded: Array<{
    candidateId: string;
    finalScore: number;
    relevanceScore: number;
    scopeBonus: number;
    recencyBonus: number;
    operatorPinBonus: number;
    tier: 'task_instance' | 'recurring_task' | 'agent' | 'subaccount' | 'organisation';
    tokenCost: number;
  }>;
  rejected: {
    aboveThreshold: Array<{ candidateId: string; finalScore: number; reason: 'budget_exhausted' | 'lower_score_in_tie' }>;
    belowThreshold: { count: number };  // Counts only, not full list.
  };
  alwaysAvailable: Array<{ candidateId: string; tokenCost: number; degraded: boolean }>;
  referenceOnlyManifest: Array<{ candidateId: string; title: string; summary: string }>;
  poolSize: { taskInstance: number; recurringTask: number; agent: number; subaccount: number; organisation: number };
};
```

### 6.3 `RetrievalRejectionReason` (closed)

`'budget_exhausted' | 'lower_score_in_tie' | 'mode_excluded' | 'authorization_filtered' | 'below_threshold'`. Authorization-filtered candidates never appear in the result; the count is a property of the candidate-pool builder, not the ranker.

### 6.4 Source-of-truth precedence

Multiple representations exist for the same fact. The precedence is **declared, not implicit**:

| Fact | Source of truth | Read path |
|---|---|---|
| Document content | `reference_document_versions` (the row pointed to by `reference_documents.current_version_id`) | always read via `referenceDocumentService.getCurrent(documentId)` |
| Document mode | `reference_documents.mode` | one column, no other source |
| Link / scope | `reference_document_data_sources` rows for `(document_id, scope_columns)` | `documentDataSourceService.findByDocument(documentId)` |
| Document chunks | `reference_document_chunks` joined to `reference_document_versions` via `version_id` | always pinned by version id; never "latest by timestamp" |
| `summary_stale` | `reference_documents.summary_stale` (boolean), set true on version write, set false when `document:summarise` job completes successfully | UI must surface stale state when true |

The atomic-swap rule for chunks (Â§4.5) is: on new version write, the new chunks are inserted with the new `version_id`; the active version pointer flips only after all chunks for the new version are embedded; old chunks are not deleted, they age out via `version_id != current_version_id`.

### 6.5 Add-to-Knowledge promotion contract

```ts
type AddToKnowledgeRequest = {
  fileId: string;                       // execution_files.id
  title: string;                        // pre-filled from filename, editable
  scope: 'organisation' | 'subaccount' | 'agent' | 'recurring_task' | 'task_instance';
  agentIds?: string[];                  // when scope='agent'; otherwise applies to all agents in the chosen scope
  scheduledTaskId?: string;             // required when scope='recurring_task'
  taskInstanceId?: string;              // required when scope='task_instance'
  mode?: 'auto' | 'always_available' | 'reference_only';  // default 'auto' if omitted
};
```

Producer: `client/src/api/knowledgeApi.ts -> addToKnowledge`. Consumer: `documentPromotionService.promoteFile(req, principal)`. The default scope is **inherited from the surface that originated the request** (brief Â§6) â€” the Manage-Task page defaults to `task_instance`, the recurring-task surface defaults to `recurring_task`, the Agent edit surface defaults to `agent`, the Knowledge page in a sub-account view defaults to `subaccount`, and the Knowledge page in an org view defaults to `organisation`. Org admins may promote upward; sub-account admins may not.

The promotion transaction (inline, in `withOrgTx`) inserts `reference_documents` + inserts initial `reference_document_versions` + inserts `reference_document_data_sources` rows + writes a `document_promotion_audit` row (the idempotency anchor) + enqueues `document:summarise` + `document:chunk-embed` via pg-boss. The `execution_files.expiresAt` durability flip is **deferred to `document:promotion-finalise`** which runs after the chunking job has committed the new version's chunks. Until that completes, the file remains expirable; if the user races deletion, the audit row stops the promotion path from re-running. See Â§8 for the rationale.

### 6.6 Mode-change contract

```ts

  documentId: string;
  mode: 'auto' | 'always_available' | 'reference_only';
```

Idempotent (state-based: predicate `WHERE mode <> :mode`). Returns the new mode. UI surfaces the change inline. Mode is a property of the document, not the link, so changing mode applies to every scope the document is linked at.

## 8. Execution model

Per checklist Â§5: every behaviour that crosses a transactional or latency boundary picks one of inline / queued / cached.

| Behaviour | Model | Why |
|---|---|---|
| Per-run retrieval (Â§4 of brief) | **Inline / synchronous.** | The agent execution path needs the result before assembling the prompt. Latency budget is < 100ms added per run (brief Â§16). |
| Document version write | **Inline.** | The version row IS the durable artefact; the user expects "save" to commit. Returns the new version id. |
| Summarisation on version write | **Queued (`document:summarise`).** | LLM call, network-bound, retryable. |
| Chunking + embedding on version write | **Queued (`document:chunk-embed`).** | Chunking is CPU; embedding is network-bound. The new version's chunks are committed inside this job; the active-version pointer flips at the end. |
| Re-embedding on model upgrade | **Queued sweep (`document:reembed`).** | Walks every document; idempotent per chunk via `(version_id, embedding_model)` unique index. |
| Add-to-Knowledge promotion | **Inline transaction + queued finalise.** | The user expects the file to appear as a document immediately. The inline transaction (in `withOrgTx`) inserts `reference_documents`, `reference_document_versions`, `reference_document_data_sources`, and `document_promotion_audit` rows, and enqueues `document:summarise` + `document:chunk-embed`. The `execution_files.expiresAt` durability flip and any post-commit telemetry happen in `document:promotion-finalise`, which is enqueued after the chunking job has committed (chained on the chunking job's success). The audit row is the idempotency anchor â€” see Â§10.1. |
| Retrieval observability event emission | **Inline at terminal (run end).** | Same boundary as the existing `agent_execution_events` emitter. |

No prompt-partition / cached-context interaction in v1. The brief explicitly defers cross-tier cache-token optimisation; documents go into the dynamic suffix per existing prompt-assembly conventions. Future work (Phase 8+) can graduate Always-available documents into a stable-prefix partition.

### 8.1 Idempotency across queued boundaries

| Job | Idempotency key | Posture |
|---|---|---|
| `document:summarise` | `(document_id, version_id)` | key-based, DB unique index on the pair; if job replays, no new write |
| `document:chunk-embed` | `(version_id, chunk_index)` | key-based, DB unique index on chunks; if a chunk already exists, the job no-ops it |
| `document:reembed` | `(version_id, chunk_index, embedding_model)` | key-based, supports embedding-model migrations |
| `document:promotion-finalise` | `(document_id, file_id)` | key-based |

## 10. Execution-safety contracts

Per checklist Â§10.

### 10.1 Idempotency posture (every external-trigger write)

| Write | Posture | Mechanism |
|---|---|---|
| Add document version | state-based (`version = current_version + 1` predicate; UNIQUE on `(document_id, version)`) | `referenceDocumentService.writeVersion` |
| Insert chunk | key-based (`UNIQUE (version_id, chunk_index, embedding_model)`, the same key used by re-embedding in Â§13.3 so a model upgrade can write a new generation alongside the old one without violating the index) | `documentEmbeddingService.embedChunk` |
| Insert link row | key-based (`UNIQUE (document_id, scope_columns) WHERE deleted_at IS NULL`) | `documentDataSourceService.link` |
| Mode change | state-based (`UPDATE ... WHERE mode <> :new_mode`) | `documentDataSourceService.changeMode` |
| Add-to-Knowledge promote | key-based (`UNIQUE (file_id, document_id) WHERE deleted_at IS NULL` on a new `document_promotion_audit` row inside the transaction) | `documentPromotionService.promoteFile` |
| Retrieval event | key-based (`UNIQUE (run_id, event_type='retrieval.summary')` partial unique index on `agent_execution_events`; replay catches `23505` and treats it as an idempotent hit) | `retrievalObservabilityService.emit` |

### 10.2 Retry classification

| Operation | Class | Boundary |
|---|---|---|
| Add version | guarded | predicate above |
| Insert chunk | safe | dedup'd by unique index |
| Insert link row | guarded | unique index |
| Mode change | safe | optimistic predicate |
| Promote file | guarded | inside `withOrgTx`; unique index on the audit row |
| Re-embed sweep | safe | per-chunk idempotency |

### 10.3 Concurrency guard for racing writes

| Race | Guard | Losing-caller response |
|---|---|---|
| Two callers promote the same file simultaneously | UNIQUE on `(file_id) WHERE ...` on the audit row; catch `23505` | 409 Conflict + "Already promoted to document {id}" + the existing document id in the body |
| Two callers change document mode at the same time | Last-write-wins is acceptable (mode is a property, not a state machine); no guard needed | the second response carries the final mode |
| Two callers add the same document at the same scope | UNIQUE on `(document_id, scope_columns) WHERE deleted_at IS NULL`; catch `23505` | 409 Conflict + "Already linked at this scope" |
| Two embedders run on the same chunk | UNIQUE on `(version_id, chunk_index)` | safe, one wins, the other no-ops |
| Two re-embed jobs touch the same chunk | same UNIQUE | same |

### 10.4 Terminal event guarantee

Every retrieval has exactly one terminal observability event (`retrieval.summary`) per run, emitted once at the end of the run regardless of success / partial / failed status. No intermediate retrieval events. **Enforcement mechanism:** a partial unique index `UNIQUE (run_id) WHERE event_type = 'retrieval.summary'` on `agent_execution_events`. If run finalisation retries after a successful emit, the second insert violates the index and is treated as an idempotent hit (no-op). Single-writer-per-run is preserved; double-counting in Â§11 aggregates is impossible.

### 10.5 No-silent-partial-success

The "always-available bypass-but-still-budget-bound" rule (brief Â§3) is the trickiest place. The contract is:

- If `sum(alwaysAvailable token_cost) > context_budget`, the system flags the run as `degraded` in the retrieval result and emits operator guidance via `RetrievalResult.alwaysAvailable[i].degraded = true`. It does NOT silently truncate; the agent execution continues but the run-summary surface marks it. Operator-facing copy in the UI reads "Some always-available documents could not be loaded due to context limits."
- Threshold-passing documents that don't fit the remaining budget are tracked in `rejected.aboveThreshold[i].reason = 'budget_exhausted'` so the "why wasn't this loaded?" drill-in (Â§11) can surface the truth.

### 10.6 Unique-constraint-to-HTTP mapping

| Constraint | HTTP status | Body |
|---|---|---|
| `reference_documents_org_name_uq` violation | 409 Conflict | `{ error: "DOCUMENT_NAME_EXISTS", existingDocumentId: "..." }` |
| `reference_document_data_sources_unique_per_scope` | 409 Conflict | `{ error: "DOCUMENT_ALREADY_LINKED", existingLinkId: "..." }` |
| `reference_document_chunks_unique_per_version_chunk_model` (i.e. `(version_id, chunk_index, embedding_model)`) | 200 (idempotent hit) | `{ noop: true, chunkId: "..." }` (job context only; never bubbled to operators) |
| `document_promotion_audit_unique_per_file` | 409 Conflict | `{ error: "FILE_ALREADY_PROMOTED", existingDocumentId: "..." }` |

### 10.7 State machine closure

The `mode` enum is closed: `auto | always_available | reference_only`. Adding a new mode requires a spec amendment. The `source` enum is also closed (Â§4.4).

`summary_stale` is a boolean (`true` after version write, `false` after summarisation completes). Forbidden transitions: none; the column is a flag, not a multi-state machine.

`reference_document_data_sources` rows have a soft-delete model (`deleted_at IS NULL` filter). Forbidden transitions: re-creating a deleted link must use a fresh row, not un-deleting (audit-friendly). The unique index applies only to `WHERE deleted_at IS NULL`.

## 12. Tenant isolation invariants

Brief Â§10. The hard rules:

1. **Cross-org boundary is absolute.** `withOrgTx` is the only entry to retrieval reads. The `RLS_PROTECTED_TABLES` manifest enforces FORCE RLS on every new table. The CI gate `verify-rls-coverage.sh` blocks merges that miss a policy.
2. **Per-scope filters are enforced at candidate-pool construction**, not after ranking. The pool builder constructs a UNION of five authorization-scoped queries, one per tier, each with its own filter:
   - Org tier: `organisation_id = :ctx_org_id AND subaccount_id IS NULL AND agent_id IS NULL AND scheduled_task_id IS NULL AND task_instance_id IS NULL`
   - Sub-account tier: `subaccount_id = :ctx_subaccount_id` (with org guard already in scope via `withOrgTx`).
   - Agent tier: `agent_id = :current_run.agent_id`.
   - Recurring-task tier: `scheduled_task_id = :current_run.scheduled_task_id` (null when the run is not on a schedule).
   - Task-instance tier: `task_instance_id = :current_run.id` (null when not applicable).
3. **Authorization filtering occurs BEFORE semantic retrieval.** No path runs a similarity search and then filters; the candidate pool IS the result of an authorization-scoped query (the five-tier UNION above).
4. **Embeddings are tenant-scoped.** `reference_document_chunks.organisation_id` is on every row (denormalised from the parent document for RLS-policy locality).
5. **Defence-in-depth at the ranker.** `retrievalServicePure` re-asserts org-id equality on every candidate; mixed-org candidates are filtered out as a programmer-error case (the existing pattern in `memoryBlockRetrievalServicePure`).

Tested by:

- Existing `rls.context-propagation.test.ts` harness expanded with the new tables (Phase 1).
- Pure-function ranker test that asserts mixed-org candidates are filtered by `organisationId` (defence-in-depth, Phase 2).
- A new manual test in the spec-conformance step at Phase 4: spin up two orgs, attach the same-named document to each, verify run isolation.

## 13. Lifecycle, versioning, re-embedding

Brief Â§11. Knowledge artefacts are versioned and re-indexable; embedding generations are replaceable infrastructure.

### 13.1 Document edit

1. New version row written (`reference_document_versions`); `current_version_id` flipped at the end.
2. `summary_stale = true` immediately on version write.
3. `document:summarise` job enqueued.
4. `document:chunk-embed` job enqueued.
5. Retrieval continues using the prior version's chunks until the new version's chunks are committed.
6. Atomic swap: when the chunking job completes, it updates `reference_documents.last_chunked_at`. Until then, retrieval queries chunk rows `WHERE version_id = current_version_id`. Old version's chunks are retained (history) but excluded.

### 13.2 Source-file change (e.g. underlying CSV updated)

Same as document edit, triggered by the existing version-write hook in `referenceDocumentService`.

### 13.3 Embedding model upgrade

When OpenAI ships a new model (or operator chooses to migrate):

1. New unique index on `(version_id, chunk_index, embedding_model)` ensures parallel embedding generations don't collide.
2. `document:reembed` job sweeps all chunks where the embedding model is the old version.
3. Reads continue from the old embedding until the new one is ready, swap is per-chunk-row; no operator action.

### 13.4 Bundle changes

Bundle composition affects which documents are pulled into a scope. The link rows are in `reference_document_data_sources` (direct attachment) and `document_bundle_attachments` (bundle attachment); neither requires re-embedding. The retrieval engine treats both as candidate-pool sources and deduplicates by `documentId`.

### 13.5 Document deletion

Soft delete on `reference_documents.deleted_at`. Cascade soft-delete to `reference_document_data_sources` (link rows). Chunks and version rows are retained for audit. Affected agents see a notice on their Data Sources tab (UI surface in Phase 6).

## 18. Open questions

### 6.4 Source-of-truth precedence

Multiple representations exist for the same fact. The precedence is **declared, not implicit**:

| Fact | Source of truth | Read path |
|---|---|---|
| Document content | `reference_document_versions` (the row pointed to by `reference_documents.current_version_id`) | always read via `referenceDocumentService.getCurrent(documentId)` |
| Document mode | `reference_documents.mode` | one column, no other source |
| Link / scope | `reference_document_data_sources` rows for `(document_id, scope_columns)` | `documentDataSourceService.findByDocument(documentId)` |
| Document chunks | `reference_document_chunks` joined to `reference_document_versions` via `version_id` | always pinned by version id; never "latest by timestamp" |
| `summary_stale` | `reference_documents.summary_stale` (boolean), set true on version write, set false when `document:summarise` job completes successfully | UI must surface stale state when true |

The atomic-swap rule for chunks (Â§4.5) is: on new version write, the new chunks are inserted with the new `version_id`; the active version pointer flips only after all chunks for the new version are embedded; old chunks are not deleted, they age out via `version_id != current_version_id`.

## 8. Execution model

Per checklist Â§5: every behaviour that crosses a transactional or latency boundary picks one of inline / queued / cached.

| Behaviour | Model | Why |
|---|---|---|
| Per-run retrieval (Â§4 of brief) | **Inline / synchronous.** | The agent execution path needs the result before assembling the prompt. Latency budget is < 100ms added per run (brief Â§16). |
| Document version write | **Inline.** | The version row IS the durable artefact; the user expects "save" to commit. Returns the new version id. |
| Summarisation on version write | **Queued (`document:summarise`).** | LLM call, network-bound, retryable. |
| Chunking + embedding on version write | **Queued (`document:chunk-embed`).** | Chunking is CPU; embedding is network-bound. The new version's chunks are committed inside this job; the active-version pointer flips at the end. |
| Re-embedding on model upgrade | **Queued sweep (`document:reembed`).** | Walks every document; idempotent per chunk via `(version_id, embedding_model)` unique index. |
| Add-to-Knowledge promotion | **Inline transaction + queued finalise.** | The user expects the file to appear as a document immediately. The inline transaction (in `withOrgTx`) inserts `reference_documents`, `reference_document_versions`, `reference_document_data_sources`, and `document_promotion_audit` rows, and enqueues `document:summarise` + `document:chunk-embed`. The `execution_files.expiresAt` durability flip and any post-commit telemetry happen in `document:promotion-finalise`, which is enqueued after the chunking job has committed (chained on the chunking job's success). The audit row is the idempotency anchor â€” see Â§10.1. |
| Retrieval observability event emission | **Inline at terminal (run end).** | Same boundary as the existing `agent_execution_events` emitter. |

No prompt-partition / cached-context interaction in v1. The brief explicitly defers cross-tier cache-token optimisation; documents go into the dynamic suffix per existing prompt-assembly conventions. Future work (Phase 8+) can graduate Always-available documents into a stable-prefix partition.

### 8.1 Idempotency across queued boundaries

| Job | Idempotency key | Posture |
|---|---|---|
| `document:summarise` | `(document_id, version_id)` | key-based, DB unique index on the pair; if job replays, no new write |
| `document:chunk-embed` | `(version_id, chunk_index)` | key-based, DB unique index on chunks; if a chunk already exists, the job no-ops it |
| `document:reembed` | `(version_id, chunk_index, embedding_model)` | key-based, supports embedding-model migrations |

### 13.3 Embedding model upgrade

When OpenAI ships a new model (or operator chooses to migrate):

1. New unique index on `(version_id, chunk_index, embedding_model)` ensures parallel embedding generations don't collide.
2. `document:reembed` job sweeps all chunks where the embedding model is the old version.
3. Reads continue from the old embedding until the new one is ready, swap is per-chunk-row; no operator action.
