Not implementation-ready yet; the critical contract, phasing, and data-model gaps below need to be closed first.

**Critical**
- `critical` — §§1, 4.5, 6.4, 16.1: The spec both promises upward reassignment for non-root agents and states that the current scope model makes it impossible in practice. Fix: resolve this before Phase 4, then update the overview, contract tables, validator flow, and success criteria to match one chosen rule.

- `critical` — §§4.3, 4.4, 5.3, 11 Phase 1, 12.4: Phase 1 expects `delegation_outcomes` rows and meaningful `agent_runs.delegation_scope` data before the only producers of those values are modified in later phases. Fix: either move the minimal write hooks/scope propagation into Phase 1 and add the touched files to the inventory, or explicitly state that the table/columns ship empty until Phase 4.

- `critical` — §§4.4, 4.5, 7.2, 8.2: The delegation-graph API promises full spawn plus handoff trees with `delegationDirection`, but the proposed service only walks `agent_runs.parentRunId` and there is no canonical join from a child run to `tasks.delegation_direction` or an accepted `delegation_outcomes` row. Fix: declare one source of truth for graph edges and direction, ideally on `agent_runs`, or fully specify the join algorithm using `parentRunId` and `handoffSourceRunId`.

- `critical` — §6.3: `spawn_sub_agents` rejects the whole call if any target is out of scope, but the spec still logs per-target `accepted` outcomes for targets that individually passed validation even though no sub-agent was actually created. Fix: decide outcomes after whole-call resolution and either reject all rows on atomic failure or add a separate validation-only state.

- `critical` — §§5.4, 9.1: `delegation_outcomes` gets only an RLS `USING` policy, so inserts are not protected by a `WITH CHECK` clause despite the spec claiming full tenant-scoped coverage. Fix: add `WITH CHECK (organisation_id = current_setting('app.current_organisation_id')::uuid)` and state whether any write path intentionally bypasses RLS.

**Important**
- `important` — §§4.1, 4.3, 6.2: The hierarchy-missing behavior is contradictory because §4 says skills that require hierarchy must fail closed, while §6.2 says the list skills silently fall back to `subaccount`. Fix: pick one rule and make the consumer list and error-code contract match it.

- `important` — §§1 item 6, 6.5, 6.9: The spec says delegation authority is derived solely from graph position with “no attachment, no drift,” but later preserves explicit attachment for leaf agents and then flags those attachments as drift. Fix: either ban explicit attachment of the delegation trio or keep it as a supported escape hatch and rename/rescope the detector.

- `important` — §§6.9, 11 Phase 1, 15.7: Two Phase 1 detectors are knowingly noisy or wrong in the phases where they ship: `subaccountNoRoot` is `critical` while zero-root is the expected pre-Phase-2 state, and `managerWithoutDerivedSkills` ships before derived skills exist. Fix: move those detectors to the phases where their invariants become real, or gate/lower their severity until then.

- `important` — §§4.4, 6.3: `spawn_sub_agents` hardcodes every logged direction as `down`, but the spec also allows root callers to use `subaccount` scope, which creates accepted lateral delegations across branches. Fix: compute spawn direction the same way as `reassign_task`, or explicitly forbid non-descendant spawn targets even for roots.

- `important` — §§3.1, 6.2: `children`/`descendants`/`subaccount` are well-defined for `config_list_agents` but not concretely defined for `config_list_subaccounts` or `config_list_links`, so implementers have to guess what those scopes mean. Fix: add per-skill filter semantics and examples, or drop the new scope parameter from the skills where the mapping is not natural.

- `important` — §§3.4, 6.6, 6.7, 10.4, 11 Phase 2: The resolver call path is inconsistent because some sections say `briefCreationService` also calls `resolveRootForScope()`, while the phase plan says it only passes `scope` into the job and the job resolves it. Fix: choose one resolution point and align the rationale, execution model, and phase inventory to that single path.

- `important` — §6.6: `resolveRootForScope()` is typed as returning `ResolveRootResult`, but the same section says `scope === 'system'` returns `null`, and the fallback enum includes `hardcoded_slug` without a defined runtime path that returns it. Fix: change the contract to `ResolveRootResult | null` if null is real, and either remove `hardcoded_slug` or define exactly when it is emitted.

- `important` — §§4.3, 4.4, 10.3, 17: The spec makes `delegation_outcomes` the sole source of delegation telemetry, but also makes writes best-effort and droppable, which makes the Phase 4 success metrics non-authoritative. Fix: either make outcome writes synchronous for accepted/rejected decisions or soften the “sole source” language and mark the metrics as advisory.

- `important` — §§4.4, 5.4: `delegation_outcomes.subaccount_id` is nullable for “org-scoped runs,” but both actor FKs point at `subaccount_agents`, which always imply a concrete subaccount link. Fix: make `subaccount_id` non-null and record the actual link subaccount, or broaden the actor model if true non-subaccount delegation must exist.

- `important` — §§7.2, 9.3: The graph route says every returned run is access-checked, but the service signature only receives `runId` and `orgId`, not the caller principal or access scope needed to enforce that claim. Fix: thread principal/access context into `delegationGraphService.buildForRun`, or explicitly state that root-run access grants subtree access and remove the stronger per-node claim.

- `important` — §§8.2, 12.3, 14.4: The spec includes `DelegationGraphView.test.tsx` even though the testing posture explicitly forbids frontend/component tests for `DelegationGraphView` and the picker. Fix: remove that test artifact from the plan/inventory, or replace it with a pure helper test if there is non-React tree-shaping logic worth testing.

**Minor**
- `minor` — §§8.3, 9.2, 14: The file inventory is not actually complete because it does not name the permission-seed file or the client router/sidebar files needed for the new `/admin/delegation-outcomes` page. Fix: add the exact file paths or drop the “single source of truth” claim.

- `minor` — §§1, 6.7, 13: There is stale editorial language, including the overview placing the delegation-graph UI in Phase 1 even though later sections place it in Phase 4, plus broken cross-references like `briefCreationService, §6.8` and `§5.9`. Fix: do one editorial consistency pass before implementation so the phase plan and section refs are unambiguous.

The critical items are blocking; the important items are the next layer of implementation-readiness cleanup once those are resolved.
