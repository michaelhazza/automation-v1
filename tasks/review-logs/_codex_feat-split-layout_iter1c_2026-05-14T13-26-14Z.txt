## 1. Goals

- Decompose `client/src/components/Layout.tsx` (1,325 LOC) into a thin layout host plus one file per logically distinct chrome region (icon rail, sidebar, topbar, alerts, modals), matching the established `client/src/components/<feature>/` convention.
- Lift side-effect-heavy state (identity, permissions, badges, projects/agents, budget, socket, keyboard) into focused hooks under `client/src/hooks/` so each hook tests one slice of orchestration logic and the host stays declarative.
- Preserve every user-visible behaviour: icon-rail order, org picker, client avatars + active-marker bar, sidebar groups, breadcrumb derivation, budget banner thresholds (ƒ%¾75% / ƒ%¾90% / ƒ%¾95% colour bands), trial countdown copy thresholds, Cmd/Ctrl+K palette, four create-modal flows (Project, Agent, Client, Brief), and the cross-tenant safety logic in the New Brief submit handler (lines 563ƒ?"612).
- Preserve the caller contract. `App.tsx` continues to import `Layout` as a default export with `{ user, children }` props from the same path.

## 2. Non-goals

- Visual change. All Tailwind classes, SVG paths, colour tokens, dimensions, animation timings stay byte-for-byte.
- WebSocket protocol change. `useSocketRoom('subaccount', activeClientId, ƒ?Ý)` keeps the same event handlers and `resyncBadges` semantics.
- Auth / localStorage change. The `auth.ts` getters / setters (`getActiveOrgId`, `setActiveOrg`, etc.) remain the source of truth; React mirror state still tracks them.
- Permission-resolution change. `hasOrgPerm` / `hasClientPerm` keep the same `__system_admin__` / `__org_admin__` sentinels and `'/api/my-permissions'` shape.
- Nav-config change. `buildNavItems(navCtx)` and the `NavItemSpec` / `NavContext` types in `client/src/config/sidebar.ts` are NOT touched ƒ?" the host keeps producing `navCtx` and passing it through.
- New tests beyond those allowed by `docs/spec-context.md` (`runtime_tests: pure_function_only`). Tests for the two pure helpers (`avatarColor`, `buildBreadcrumbs`) are in scope and small.

- Description: The goals say extracted hooks should each be testable, but the non-goals and helper section limit new tests to pure helpers only. That makes hook testing a stale/load-bearing claim in a pure refactor spec.
- Quote: “so each hook tests one slice of orchestration logic” vs “Tests for the two pure helpers (`avatarColor`, `buildBreadcrumbs`) are in scope”
- Suggested fix: Change the goal to “so each hook owns one slice of orchestration logic” and remove the implication that hooks get tests.

## Finding 2 - important

- Section: 5 Target structure vs 10 Chunk 4 vs 13 Acceptance criteria / 14 Open questions
- Description: `OrgPicker.tsx` is simultaneously listed as part of the target file inventory, described as optional/in-chunk, and then required by acceptance through “file listing in §5.”
- Quote: “`OrgPicker.tsx` optional split out of IconRail” and “New folders ... exist with the file listing in ¶õ5” and “keep `OrgPicker` inline ... or extract to its own file”

## Finding 3 - critical

- Section: 7.8 Org list vs 8.1 / 8.5 Prop contracts vs 10 Chunk 4
- Description: Org-list ownership is internally inconsistent. §7.8 says org state may live inside `IconRail.tsx` and never cross the chrome/host boundary, but §8 requires both `IconRail` and `NewBriefModal` to receive `orgs` as props. - Suggested fix: Declare one source of truth. Most mechanically: add `useOrgList(isSystemAdmin)` to the hook inventory, have the host own/pass `orgs`, and remove the “kept inline” option.
