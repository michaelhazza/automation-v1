1. **Section 2f / 7b**: `ScrapeOptions` is missing `orgId` and `subaccountId`, but `executeScrapeUrl()` passes both into `scrapingEngine.scrape()`, so the engine contract is internally inconsistent.  
   **Fix:** In section 2f, add `orgId: string;` and `subaccountId?: string;` to the `ScrapeOptions` interface and state that these are required for org-scoped rate limits, settings, and selector/cache persistence.

2. **Section 9a / 10**: The Drizzle schema for `scraping_selectors_upsert_key` omits `NULLS NOT DISTINCT`, while the SQL migration includes it, so the schema and migration define different uniqueness behavior for nullable keys.  
   **Fix:** In section 9a, change the Drizzle index definition to `uniqueIndex('scraping_selectors_upsert_key').on(...).nullsNotDistinct()` so it matches section 10 exactly.

3. **Section 3d / 5b / 9a**: Selector persistence is keyed only by `urlPattern` in the documented `load()` API, but the table schema and skills also use `selectorGroup`, so the read/write contract for learned selectors is under-specified and can collide across different extraction intents on the same URL pattern.  
   **Fix:** In section 3d, change `load(orgId, subaccountId, urlPattern)` to `load(orgId, subaccountId, urlPattern, selectorGroup)` and add a sentence that `selectorGroup` is the source of truth for separating independent structured extractions on the same site.

4. **Section 4b / 6b**: `scrape_structured` exposes `remember`, but the extraction flow always saves selectors after the first LLM scrape, so the spec claims a toggle without defining how it is enforced.  
   **Fix:** In section 4b step 5, change the text to “Engine saves the selectors as adaptive fingerprints for next time **only when `remember !== false`**”; also add the same condition to section 6c’s implementation note.

5. **Section 5c / 9b**: `monitor_webpage.md` says “Only call `monitor_webpage` once per URL,” but the cache schema explicitly supports multiple monitors on the same URL via `scheduled_task_id`, which is a direct contradiction.  
   **Fix:** In section 5c, replace that sentence with “Do not call `monitor_webpage` repeatedly for the same monitoring intent; distinct monitors on the same URL are allowed when they track different criteria or schedules.”

6. **Section 7d / 12a**: The monitor flow says `executeMonitorWebpage` calls `scrapingEngine.scrape()` to establish a baseline, but the example then depends on structured JSON extraction and selector learning, which `scrapingEngine.scrape()` alone does not define.  
   **Fix:** In section 7d step 1, change the text to “If `fields` is provided, call `executeScrapeStructured` (or the equivalent structured extraction path) to establish the baseline; otherwise call `scrapingEngine.scrape()` for unstructured content,” and update section 12a step 3 to match.

7. **Section 7d / 5c / 6c**: Scheduled-task metadata stores `selectorGroup`, but `monitor_webpage` has no `selector_group` parameter and no rule for deriving one, so later scheduled runs depend on data that is never defined.  
   **Fix:** In section 7d step 3, add “When `fields` is provided, derive `selectorGroup` deterministically as `<normalized-host>:<hash(fields)>` unless explicitly added as a future parameter”; mirror that sentence in section 5c under Parameters or Instructions.

8. **Section 11a**: The Strategic Intelligence Agent prompt tells the agent to call `update_memory_block`, but that skill is not present in the YAML skills list shown in the same section.  
   **Fix:** Either add `update_memory_block` to the YAML `skills:` list in section 11a, or change the prompt text to use one of the listed capabilities instead; the simplest text change is to replace “`update_memory_block` with the latest data snapshot” with “update workspace memory with the latest data snapshot.”

9. **Section 12b / 4a**: The “one-off research” example uses `scrape_structured`, but section 4a’s decision boundary says one-off structured extraction should use `scrape_url(output_format='json')` and reserve `scrape_structured` for recurring extraction.  
   **Fix:** In section 12b, change the example action from `scrape_structured({...})` to `scrape_url({ url: ..., extract: "plan name, price, features, limits", output_format: "json" })`, or rewrite section 4a’s decision boundary if `scrape_structured` is intentionally allowed for one-off use.

10. **Section 13b / 13c**: The prose says robots.txt opt-out can be set “for specific domains,” but the typed settings contract only defines a single org-wide boolean `respectRobotsTxt`, so the enforcement mechanism described does not exist.  
   **Fix:** In section 13b, replace “for specific domains” with “for the org,” or extend section 13c’s `OrgScrapingSettings` to include a domain-scoped override map and state that this map is the source of truth.

11. **Section 14b / 7d**: The integration test `monitorWebpageSkill.test.ts` is described as covering “change detection on re-run,” but section 7d says subsequent monitoring runs call `scrape_structured`, not `monitor_webpage`, so the test target is stale.  
   **Fix:** In section 14b, change that line to “`monitorWebpageSkill.test.ts` — baseline creation and scheduled task creation”; add a separate test entry for the scheduled-run agent flow that reads task metadata, re-scrapes, compares baseline, and updates `scraping_cache`.

12. **Summary of Files Changed / Section 7d**: The spec assigns recurring-run comparison logic to “the Strategic Intelligence Agent” but the file inventory names no runtime file where scheduled-task execution or baseline comparison logic is implemented, leaving the implementation location incomplete.  
   **Fix:** In section 7d, name the concrete file that will perform scheduled-run reconstruction and baseline comparison, and add that file to the “Modified files” table; if the intent is to implement this inside an existing scheduled-task execution path, state that file explicitly.
