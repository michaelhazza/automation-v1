Skill & Agent Ring Rollout

Interactive mockup — walking through how a system-agent or skill change travels from dev to prod without breaking every client at once.

1. Concept 2. Author edit 3. Rollout pipeline 4. Canary health 5. Agency view 6. Rollback 7. Skills 8. Summary

1. The concept in one sentence

A change to a system agent or a skill never goes straight to every client. It moves through four rings, gathering evidence at each step that it's safe.

Dev

Author's sandbox

Runs only on the author's test subaccount. Zero client impact.
Test

Internal + early access

5–10 internal orgs. Real runs, side-by-side with prod.
Canary

5–10% of prod

Live for a slice of paying orgs. Auto-pauses if metrics regress.
Prod

Everyone

Full rollout. Previous version kept warm for rollback.
The mental model: same as software releases. You already trust this pattern when GitHub rolls out a feature to 5% of users first. We're applying it to the thing that actually drives every client's business — the agent prompts and skills.

What moves through the rings

ArtifactWhat versioning already existsWhat rings add
System agent promptEvery save creates a row in agent_prompt_revisionsA ring column per revision + which orgs are on which ring
System agent skill-linksVersioned per agentSame ring stages
System skill bodyFile-based today (server/skills/*.md)Versioned table + ring column (new)
Org agents / skillsOrg-owned, org's riskNot gated — agencies control their own rollouts

Only system-managed artifacts (the ones every org inherits) go through rings. Org-level customisation stays direct-edit.

The readiness checks are the meat of the system. They're the automated version of "is this safe to ship to more people yet?" If any row is red, the promote button is disabled with the specific reason shown. Every ring transition has its own thresholds (Dev→Test is loose, Canary→Prod is strict).

Worked example — the auto-pause in action: imagine rev 47 ships with a subtle prompt bug that makes the agent occasionally hallucinate URLs. HITL escalation rate jumps from 4.2% to 7.1% in the canary cohort. Auto-pause trips within minutes. 38 orgs revert. The other 504 orgs never noticed a thing. You wake up to a Slack alert, not a support firestorm.

6. Rollback — one click, any ring

Auto-pause handles the 3am failure. Rollback handles the "we shipped to prod, it passed canary, but a real-world edge case emerged on day 3" case. Both are the same motion: point the ring at a previous revision.

Modal — Rollback Prod: rev 44 → rev 43
Confirm rollback
Current Prodrev 44shipped 18d ago
Rolling back torev 43previous Prod, shipped 41d ago
Orgs affected504everyone currently on rev 44
In-flight runs12finish on rev 44 · next run uses rev 43
Regression in tone handling for regulated-industry clients — surfaced by BigLaw Agency ticket #4821.
Roll back Prod to rev 43 Cancel

Rollback freezes rev 44 from being re-promoted until the author marks it as "fixed and ready to re-ship." Prevents accidental re-promotion of a known-bad revision.

Worst-case recovery time: ~30 seconds from "I see the problem" to "all 504 orgs are back on rev 43." In-flight runs finish on the old version; the next run picks up the rollback. No data migration, no background job, no restart.

8. Summary — usability and what you need to decide

What's simple about this

  • You already version agent prompts. agent_prompt_revisions exists. Rings add one column.
  • You already have skill versioning partially via the Skill Studio. The move is file → table.
  • Ring membership is one table. (ring, revision_id, organisation_id). Every run reads "which revision should this org use for this agent?" and gets one answer.
  • The UI is additive. Every screen shown here sits on top of pages that already exist — agent detail, skill detail, agency admin. Nothing is replaced.

What's genuinely complex

  • Metric comparison infrastructure. Comparing canary-cohort runs to prod-cohort runs needs a stable metrics pipeline. We have the raw data (llm_requests, agent_runs) but no rolling comparator. This is the biggest new build.
  • Canary org selection. Must be deterministic (same orgs stay in canary across promotions, unless explicitly excluded), must respect tier (white-glove orgs excluded by default), must be fair (not the same orgs every time for every rollout).
  • Auto-pause trigger. A job that samples metrics every N minutes and flips the ring pointer back if any threshold trips. Low complexity, but has to be rock-solid.
  • Org-level overrides. The "pinned" option for white-glove tier is a UX simplicity win, but adds a second ring-membership path to reason about.

Open design decisions you need to make

  1. Canary percentage default. 5%? 10%? Per-agent configurable, or platform-wide? Start with one number, let the data tell us.
  2. Who can promote? Currently the author can promote through every ring. Do we want a second pair of eyes required for Canary→Prod on high-blast-radius agents?
  3. Early-access opt-in. Do agencies self-serve opt into the Test ring, or is it invite-only?
  4. What about org-level agent customisation? If an agency has edited their copy of a system agent's additionalPrompt, how do we handle upgrading the base masterPrompt? Probably: base rev moves, their override stays. Worth confirming.
  5. Skill ring = agent ring, or independent? If the Outreach Agent calls draft_followup, and the skill is on canary v13 while the agent is on prod v44 — is that OK? (My read: yes, they're independent, that's the point.)
  6. Rollback authority. Who can rollback Prod? Just the author? Any system admin? Should it trigger a Slack alert?

Proposed build order (if you greenlight)

  1. Phase 1 — agent rings only (2 weeks). Add ring column to agent_prompt_revisions, ring-membership table, the pipeline screen, the promotion modal. Manual promotion, no auto-pause yet. Ship it and use it internally for a month.
  2. Phase 2 — metrics + auto-pause (2 weeks). Build the rolling metrics comparator. Wire auto-pause. Add the canary health dashboard.
  3. Phase 3 — agency-facing UI (1 week). The "you're on rev X" panel, version history, tier preferences.
  4. Phase 4 — skill rings (2 weeks). Move skills to a versioned table, apply the same pipeline. Metrics are skill-specific but the comparator infrastructure is reused.

Total: ~7 weeks of focused work. The first useful milestone (phase 1) ships in 2 weeks and we dogfood it before exposing it to agencies. That's the right shape — we catch our own foot-guns before clients see any of it.
