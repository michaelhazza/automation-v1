## 4. Failure breakdown

### Run 2 (canonical)

| Class | Count | Cause |
|---|---|---|
| `gemini_other` (429 quota) | 18 | Free-tier daily quota of **20 requests/day** for `gemini-2.5-flash` exhausted by Run 1. Quota body explicitly says: `"Quota exceeded for metric: generativelanguage.googleapis.com/generate_content_free_tier_requests, limit: 20"`. |
| `gemini_other` (503) | 2 | Transient model overload, "This model is currently experiencing high demand. Spikes in demand are usually temporary." |
| `supadata_4xx` (202 async) | 1 | Evolution of Dance (6 min). Supadata returned 202 — async transcript queued. Script doesn't poll. |

### Run 1 (informational, superseded by Run 2)

In Run 1 (parallel Supadata calls), the pattern was:
- **Supadata 429s (10 videos):** parallel `transcript` + `metadata` calls per video tripped per-second rate limit. **Self-inflicted** — fixed in Run 2 by serializing.
- **Supadata 524s (2 videos):** Cloudflare timeouts on Baby Shark + Thrift Shop (both very long, very high-view-count videos that Supadata's scraper takes >100s to process). May also reflect async-pending behavior; would need polling.
- **Gemini 503s + 429s (19 videos):** mix of high-demand transient errors and quota throttling.

## 5. Recommendation

**Verdict: GO with conditions.** The vendor stack is feasible. Specifically:

| Layer | Verdict | Evidence |
|---|---|---|
| **Supadata (fetch + transcript + metadata)** | **GO on YouTube.** Reliability holds. | 19/20 fetched cleanly + 1 async-pending (would resolve via polling). 100% on metadata. |
| **Gemini 2.5 Flash (analysis)** | **GO on capability.** Quality + cost both clear the bar. Operationally **needs paid tier** before any further volume testing. | One clean E2E success: $0.011 / 37.7s / high-quality structured JSON. Cost is **5 to 10× cheaper** than the modelled $0.07 to $0.13 envelope. |
| **TikTok / Instagram** | **UNVERIFIED.** | All 30 slots were TODO placeholders; not tested. Independent benchmarks suggest ~91% on TikTok via Supadata; we have no first-party data yet. |

### Conditions before integrating

1. **Upgrade Gemini billing to paid tier.** Free tier (20 req/day) is incompatible with even a single 50-video pilot run, let alone production. Paid tier removes the 429 quota wall observed here. _Pricing remains $0.30/M input + $2.50/M output, same as our model._
2. **Re-run pilot on paid tier with TikTok/Instagram URLs filled in.** This is where the real platform-coverage uncertainty lives. Specifically validate:
   - Does Gemini 2.5 Flash accept TikTok/Instagram URLs directly via `file_data.file_uri`? Vendor docs are explicit only about YouTube. If not, we'll need a download-and-upload-via-File-API path, which adds complexity and cost.
   - Does Supadata's 91% TikTok benchmark hold on real, recent URLs?
3. **Build async-fetch handling into the production fetcher.** Supadata's HTTP 202 response means "queued for processing" — production should poll, not treat 202 as failure. ~5% of YouTube cases need this; likely higher on TikTok where pages are more dynamic.
4. **Adopt media_resolution=LOW as the default.** Confirmed working; keeps token cost down. The Shape of You analysis is rich enough at LOW that we don't need MEDIUM/HIGH for the "why this video worked" use case.

### What I would not change

- Don't build native scrapers. Supadata's 95% YouTube reliability is high enough to justify the dependency.
- Don't pre-extract frames or downsample. Gemini's native video ingestion is the simpler path and the cost is well within budget.
- Don't over-engineer retry. The 429s are quota-shaped (a tier issue), not transient — retries don't help; tier upgrade does. The 503s are infrastructure-shaped; light retry (1 to 2 attempts with backoff) would smooth those out, but is not load-bearing for the feasibility decision.
