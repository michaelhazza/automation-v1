What each badge actually counts

7 Home (review queue)

What it is: Items sitting in the human-review queue for the active client.

Counts: Rows in reviewItems where subaccountId matches and reviewStatus IN ('pending', 'edited_pending').

Action: Click → review queue → approve / edit / reject each item.

Source: /api/subaccounts/:id/review-queue/count · permission REVIEW_VIEW · realtime via review:item_created socket.

3 live Dashboard (live agents) dead

What it would be: How many agents are currently running for the active client.

Counts: Live count via /api/subaccounts/:id/live-status; updated by live:agent_started / live:agent_completed socket events.

Action: None — it's awareness data. The Live Agent Execution Log page is where operators actually monitor runs.

Status: Computed and updated at all times right now, but not displayed anywhere — JSX was removed in a prior commit.

2 Incidents (system faults) sys-admin only

What it is: Active high-or-critical platform faults. Only system admins see this nav item.

Counts: Rows in system_incidents where status is one of open / investigating / remediating / escalated AND severity ∈ {high, critical} AND classification = system_fault.

Action: Click → incidents page → triage / suppress / escalate.

Source: /api/system/incidents/badge-count · system-admin permission only.

The take

Home and Incidents both signal action waiting — work the operator must triage. The dead "live agents" indicator is awareness data with no implied action. Per the project's frontend design principles ("default to hidden", "≥3 sidebar panels → cut one"), Option B keeps the sidebar focused on action affordances and removes the awareness-only signal. If you ever want live-run visibility again, the right place for it is the dedicated Live Agent Execution Log page, not the sidebar.
