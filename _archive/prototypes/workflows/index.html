Workflows · UI prototypes

Brainstorming the Phase-2 UI for the workflows feature. Backend (engine, schema, routes) is already built — these mocks explore the operator surface, the Studio authoring surface, and the open-task surface.

Current direction (the four A's)

04 · Four A's — per-step-type inspector showcase
Foundation
Single page with four state-switch buttons at the top (Agent · Action · Ask · Approval) — click each to see that step type selected in a real workflow with its inspector showing fields specific to that type. Same canvas across all four states; only the selected step + inspector content change. Approval state demonstrates the new humans-only approver picker (Specific people / A team / Task requester / Org admin) with quorum.
05 · Studio — route editor
Recommended
Canvas-first authoring (chat as a docked pill, slide-out inspector, hover-to-insert "+", floating bottom action bar). Step cards use the four A's vocabulary. Branching as an inline "Branch on X" chip below an Agent step. Approval-on-reject as a dashed back-arrow to a previous step. Selected step is the Marketing review approval — inspector shows the new humans-only approver picker with quorum + loop-back-on-reject configuration.
07 · Open task view — 3-panel master
Operator surface (master)
The user-facing task surface. Chat (26%) + Activity panel (22%, collapsible — can minimise to 36px strip) + Tabbed right panel (52%) with Now / Plan / Files (renamed v2 from Live / Flow / Files). Activity is its own column rather than a tab so chat + activity + visual context are all visible at once. Files tab supports portrait + landscape orientations with proper proportions in the reader pane. Document toolbar above each file with download + open-in-new-window actions. Same surface for ad-hoc and workflow-fired tasks — "Brief" retired as a noun.
08 · Task progression states (5 moments)
Real-time coordination · v2
Five states of one task, click between them to see how chat / activity / right pane stay coordinated: 1) just kicked off (Plan tab, plan forming), 2) multi-agent dispatch (Now tab, org chart pulsing), 3) fetching references (Files tab, Stripe rate card), 4) file created (Files tab, Acme proposal), 5) awaiting approval (Plan tab, Critical pill + confidence chip). Chat panel is the conversation plus a single thinking box at the bottom (not a per-agent log). Activity is the chronological event stream, newest at the bottom (matches chat). Right pane focuses on whichever tab the moment calls for. Demonstrates v2 design refinements (chat-as-thinking-box pattern + WebSocket-driven multi-pane coordination, brief §6.1 / §6.2).
09 · Ask step authoring (5 inspector states)
Authoring deep-dive · v2
Five inspector states zoomed into the Ask step type: A) default, B) Who-can-submit dropdown open (4 routing options), C) Auto-fill dropdown open (V1 + V2 deferrals), D) Add-a-field picker (7 V1 field types: text, textarea, select, multi-select, number, date, checkbox), E) Editing a select field with options list. Companion to mock 04 which shows all four A's together. Brief §7.3.
10 · Ask step runtime (form card + routing)
Runtime experience · v2
What happens when an Ask step queues at runtime. Four states: A) form card awaiting input in the chat panel, B) mid-fill with required-field validation, C) submitted (collapses to a receipt; next step starts), D) routed elsewhere (sidebar count, "Waiting on you" page, email — all paths land on the same form card in the same task view). Demonstrates the form-card primitive shares the same surface and lifecycle as Approval cards.
