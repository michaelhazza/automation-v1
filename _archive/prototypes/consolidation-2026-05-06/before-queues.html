Problems with the current two-page approach

Two pages for related data
Queue health and execution forensics are the same data at different aggregation levels. SREs must navigate between two URLs to correlate a queue's elevated state with its specific failing runs.
No drill-down from queue to executions
Clicking a queue row in JobQueueDashboardPage does nothing. To see failing executions for agent.run, the engineer must copy the queue name and manually search in SystemTaskQueuePage.
Tier grouping is data, not visual
The JobQueueDashboardPage groups queues by tier via table section rows with no color coding, no tier-level summary, and no visual differentiation between tiers.
No live tail in SystemTaskQueuePage
The execution log does not update in real time. During an incident, the engineer must manually reload the page to see new failure events as they arrive.
DLQ retry requires CLI
Dead-letter queue jobs cannot be retried or discarded from the UI. The only remediation path is a separate admin script executed via command line, requiring terminal access and environment knowledge.
No queue pause / resume in UI
Pausing a queue to stop processing during an outage requires direct Redis CLI access. There is no operator-facing pause control in either page.
