## 1. Purpose and scope

This document is the structural contract between two consumers of the `investigate_prompt` field on `system_incidents`:

1. **The `system_monitor` agent** (server-side). Its prompt-authoring instructions reference this protocol — "when emitting `investigate_prompt`, format per `docs/investigate-fix-protocol.md`." The contract makes prompt format predictable and tunable in one place.
2. **Claude Code** (operator's local development environment). A hook in `CLAUDE.md` instructs every Claude Code session in this repo: "if given a prompt that begins with `# Investigate-Fix Request`, follow `docs/investigate-fix-protocol.md` for execution behaviour."

The protocol is the structural contract. Both ends iterate against it. The agent's prompt-writing improves; Claude Code's prompt-consumption behaviour improves; the contract holds the iteration coherent.

**Scope.** This document governs:

- The required and optional sections of every `investigate_prompt` value.
- Forbidden content (instructions that would bypass operator approval).
- Authoring rules for the monitor agent when generating prompts.
- Execution rules for Claude Code when consuming prompts.
- The version-stamping scheme that allows the protocol to evolve without breaking archived prompts.

**Out of scope.** The heuristics that decide *whether* to fire a prompt (see `server/services/systemMonitor/heuristics/`), the scheduling of the monitor agent's sweep job, and the UI surface for displaying the prompt (see `client/src/components/system-incidents/`) are governed by the spec and the code, not this document.

**Forward path.** When Phase 3 (auto-remediation) ships, a server-side worker will follow this same protocol to consume prompts unattended. No architectural change is needed — just a new executor pointed at the same contract. The protocol-doc approach is the lever that makes Phase 3 incremental rather than a rewrite.

## 2. Prompt structure — the contract

Every `investigate_prompt` value generated by the monitor agent must conform to the following structure. Sections are markdown-headered for human readability and machine parseability.

```markdown
# Investigate-Fix Request

## Protocol
v1 (per docs/investigate-fix-protocol.md)

## Incident
- ID: <system_incidents.id>
- Fingerprint: <fingerprint>
- Severity: <low|medium|high|critical>
- First seen: <ISO8601>
- Occurrence count: <integer>
- Source: <route|agent|job|connector|skill|llm_router|synthetic|self_check>

## Problem statement
<One paragraph. What looks wrong. Plain English. No internal jargon
without expansion. Example: "Agent run abc123 completed successfully
but produced no output text. The agent is configured to summarise
emails; the user-facing result is empty. This is the third occurrence
in the last 6 hours.">

## Evidence
<Bullet list. Each bullet must include a file:line reference where applicable,
or a stable resource identifier (agent_runs.id, pgboss.job.id, etc.).
Example:
- agent_runs.id = abc123 (server/services/agentRunner.ts:147 logs the
  empty-output case)
- Recent runs: see system_incidents row, sweep_evidence_run_ids
- Heuristic that fired: empty_output_baseline_aware (confidence 0.78)
  defined in server/services/systemMonitor/heuristics/agentQuality.ts:34
- Baseline median output for this agent: 1,247 chars (this run: 0 chars)>

## Hypothesis
<One paragraph. Best guess at root cause, with confidence stated.
Example: "Likely cause is the upstream Gmail connector returning an
empty thread payload — see connector_configs row id 'gmail-default'
where lastSyncError contained 'rate-limited' 8 minutes before this
run. Confidence: medium. Alternative hypothesis: a recent change to
the email-summary skill prompt (commit hash if known); see
git log -- server/skills/emailSummary.ts.">

## Investigation steps
<Numbered list. What Claude Code should do, in order. Each step concrete
enough to execute without follow-up clarification.
Example:
1. Read server/services/agentRunner.ts:140-180 for the empty-output
   handling.
2. Read server/skills/emailSummary.ts to verify prompt and tool calls.
3. Query agent_runs WHERE agent_id = '...' AND created_at > NOW() - INTERVAL '24 hours'
   — count how many produced empty output vs non-empty.
4. Check git log on server/skills/emailSummary.ts for changes in the last 7 days.>

## Scope
- In scope: <comma-separated list of files or resources the executor may read and modify>
- Out of scope: <comma-separated list of files or resources the executor must not touch>

## Do not change without confirmation
<Optional. List of files or behaviours the operator should be asked
about before modifying. Example: "Do not modify the agent's system
prompt without explicit operator confirmation — the prompt is part of
the agent's tuned behaviour and changes affect all subaccounts using it.">

## Expected output
A diff or set of proposed changes. The operator (human in the loop)
will review and approve before merge. Do not commit or push without
approval.

## Approval gate
The user (operator) must explicitly approve any code change before it
is committed.
```

## 3. Required sections, forbidden content, and worked example

### 3.1 Required sections

Every `investigate_prompt` must contain all nine of these sections, in order. Empty sections are explicitly disallowed — the agent must either provide content or state `(none — see Hypothesis)`.

| Section | Requirement |
|---|---|
| `## Protocol` | Version stamp. Format: `v<n> (per docs/investigate-fix-protocol.md)`. |
| `## Incident` | Structured metadata: ID, fingerprint, severity, first seen, occurrence count, source. |
| `## Problem statement` | One paragraph. Plain English. No unexpanded jargon. |
| `## Evidence` | Bullet list. Each bullet includes a `file:line` reference or a stable resource identifier. |
| `## Hypothesis` | One paragraph. Root cause guess with explicit confidence statement (`low`, `medium`, `high`). |
| `## Investigation steps` | Numbered list. Each step concrete enough to execute without follow-up clarification. |
| `## Scope` | Two sub-bullets: `In scope:` and `Out of scope:`. Concrete file paths or resource names. |
| `## Expected output` | Must include the instruction: "The operator will review and approve before merge. Do not commit or push without approval." |
| `## Approval gate` | Must state that the operator must explicitly approve any code change before it is committed. |

### 3.2 Optional section

`## Do not change without confirmation` may be omitted if there are no special files or behaviours requiring pre-modification confirmation. When present, it lists files or behaviours the operator must be asked about before the executor proceeds.

### 3.3 Forbidden content

A prompt is malformed if it contains any of the following:

- An instruction to commit, push, deploy, or merge without explicit operator approval.
- An instruction to modify files listed in `## Scope / Out of scope` without first surfacing this to the operator.
- An instruction to fabricate file paths or line numbers not derivable from the evidence.
- An `## Investigation steps` step that reads as "investigate as you see fit" or similar open-ended exploration without a concrete termination condition.

The CLAUDE.md hook (see §1 and the CLAUDE.md file) causes Claude Code to refuse execution of any prompt that violates these rules.

### 3.4 Worked example

The following is a short well-formed prompt illustrating all required sections. This example is anonymised and does not represent a real incident.

```markdown
# Investigate-Fix Request

## Protocol
v1 (per docs/investigate-fix-protocol.md)

## Incident
- ID: 3a7b9c2d-1234-5678-abcd-ef0123456789
- Fingerprint: a4b1c2d3e4f5a6b7
- Severity: high
- First seen: 2026-04-20T08:14:00Z
- Occurrence count: 3
- Source: agent

## Problem statement
Agent run 7f3c8a2b completed successfully (exit status: ok, no error in
agent_runs) but produced zero output text. The agent is configured to
summarise incoming emails; the user-facing result was an empty summary.
This is the third such occurrence in 6 hours, all for the same agent
instance (agent_id: a1b2c3d4).

## Evidence
- agent_runs.id = 7f3c8a2b, output_text length = 0
  (server/services/agentRunner.ts:147 logs the empty-output path)
- Heuristic that fired: empty_output_baseline_aware (confidence 0.81),
  server/services/systemMonitor/heuristics/agentQuality/emptyOutputBaselineAware.ts:28
- Baseline median output length for this agent: 1,182 chars
  (system_monitor_baselines WHERE entity_kind='agent_run' AND
   entity_id='a1b2c3d4' AND metric_name='output_length_chars')
- connector_configs id='gmail-prod': lastSyncError = 'rate_limit_exceeded'
  at 2026-04-20T08:06:00Z (8 minutes before agent run)

## Hypothesis
Likely cause: the Gmail connector returned an empty thread payload due to
a rate-limit error 8 minutes before the agent run. The email-summary skill
received an empty input and produced an empty output without raising an
error. Confidence: medium. Alternative: a recent change to the
email-summary skill prompt changed how empty input is handled — check
git log on server/skills/emailSummary.ts.

## Investigation steps
1. Read server/services/agentRunner.ts:140-180 to understand the
   empty-output logging path.
2. Read server/skills/emailSummary.ts to inspect the prompt template and
   any empty-input guard.
3. Query: SELECT output_text IS NULL OR LENGTH(output_text) = 0 AS empty,
   COUNT(*) FROM agent_runs WHERE agent_id = 'a1b2c3d4'
   AND created_at > NOW() - INTERVAL '48 hours' GROUP BY 1 — check
   frequency of empty output over 48 hours.
4. Run: git log --oneline -- server/skills/emailSummary.ts — look for
   changes in the last 7 days.
5. If step 4 shows a recent change, diff the commit to see what changed.

## Scope
- In scope: server/skills/emailSummary.ts (read + modify if a bug is
  found), server/services/agentRunner.ts (read only), connector_configs
  table (read only).
- Out of scope: system_incidents table, system_monitor_baselines,
  the monitor agent itself, any RLS policy.

## Do not change without confirmation
Do not modify the email-summary skill's system prompt without explicit
operator confirmation — prompt changes affect all subaccounts using this
agent.

## Expected output
A diff or set of proposed changes addressing the root cause, if one is
found. If no code bug is found (e.g. the connector rate-limit is the sole
cause), report this conclusion with supporting evidence so the operator
can decide whether to tune the connector's retry policy. The operator
will review and approve before merge. Do not commit or push without
approval.

## Approval gate
The user (operator) must explicitly approve any code change before it is
committed.
```

## 4. Authoring instructions for the monitor agent

These rules are included in the monitor agent's system prompt (see `server/services/systemMonitor/`). They are reproduced here as the canonical source of truth. The system prompt carries these rules verbatim; if they diverge, this document wins.

### 4.1 Template adherence

The agent's system prompt includes a copy of the §2 template structure, the §3.1 required-section table, and the §3.3 forbidden-content list. The agent is instructed to fill in every required section. Empty sections are disallowed — if a section has no content, the agent must write `(none — see Hypothesis)` rather than omitting the heading or leaving it blank.

### 4.2 Evidence quality rules

- Always include a `file:line` reference when the evidence is traceable to a specific code location. Never fabricate a file path or line number.
- When the evidence is purely behavioural (e.g. a job that did not run, an empty-output run), use a stable resource identifier instead: `agent_runs.id`, `pgboss.job.id`, `system_incidents.id`, `connector_configs.id`, etc.
- The Evidence section must contain at least one bullet. "No evidence available" is not a valid entry — if no specific evidence exists, the agent must not fire a prompt.

### 4.3 Length constraints

- **Target:** 400–800 tokens per prompt.
- **Hard cap:** 1,500 tokens. Above this cap the agent must trim the Evidence or Investigation Steps sections and append a note: `(Evidence truncated to stay within the 1,500-token hard cap. Full evidence is in the system_incidents row and associated agent_runs/skill_executions.)`.
- The hard cap exists to prevent prompts that are too long for the operator to read quickly and too expensive to feed back into Phase 3 auto-fix without truncation.

### 4.4 Hypothesis humility rules

- The hypothesis is always stated with an explicit confidence: `low`, `medium`, or `high`.
- If confidence is `low`, the agent must include the sentence: "This hypothesis is low-confidence. The operator should verify it before assuming it is correct."
- If the agent cannot form any hypothesis (insufficient data), it must write `Hypothesis: insufficient data to form a hypothesis. The operator should investigate the Evidence section directly.` — it must not fabricate a hypothesis.
- Alternative hypotheses are encouraged when the primary hypothesis is not high-confidence.

### 4.5 Scope rules

- The Scope section must list specific file paths or resource names (not categories like "all server files").
- The `Out of scope` list must always include: `system_incidents table, system_monitor_baselines, the monitor agent itself, any RLS policy` — these are constant out-of-scope items for every prompt.
- Additional out-of-scope items are added based on the incident's context.

### 4.6 Version stamp

Every prompt must begin with:

```
# Investigate-Fix Request

## Protocol
v1 (per docs/investigate-fix-protocol.md)
```

When the protocol bumps to `v2`, the agent's system prompt is updated to emit `v2 (per docs/investigate-fix-protocol.md)` in the same commit that updates this document. The agent must not emit a version number it was not explicitly instructed to emit.

## 5. Iteration loop and version stamping

### 5.1 The protocol is a living document

This document will be wrong on day one. Iteration is the point.

**Inputs to iteration:**

- The `prompt_was_useful: bool` and `prompt_feedback_text` fields on `system_incidents` (set by the operator via the triage drawer feedback widget).
- The `investigate_prompt_outcome` event log in `system_incident_events`.
- Operator-side observations: which prompts led to one-shot fixes, which required prompt rewriting, which led to wrong-direction investigations.

**Iteration cadence:** weekly review of the prior week's prompts and outcomes. Adjustments land as edits to this document. Each edit is a normal commit with a clear message — protocol drift is observable from git log.

**Phase 3 readiness signal.** When the agent's prompts consistently lead to one-shot fixes that operators accept without rewriting, that is the signal Phase 3 (auto-fix) is on solid ground. The protocol holds; the agent's diagnosis quality has stabilised; the auto-fix executor can be pointed at the same contract. There is no formal threshold in this spec — Phase 3 is its own design exercise — but the feedback data accumulated here is the input to that decision.

### 5.2 Version stamping

Every generated `investigate_prompt` carries a `## Protocol\nv<n> (per docs/investigate-fix-protocol.md)` line at the top per §2. The version stamp is stored in the prompt body — not in a separate column on `system_incidents` — because the prompt body is the historical record (the prompt text is the work product; the work product is the source of truth).

**What happens when the protocol bumps to v2:**

1. The agent's system prompt is updated to emit `## Protocol\nv2 (per docs/investigate-fix-protocol.md)` in the same commit that updates this document.
2. Old incidents retain their original `## Protocol\nv1` text indefinitely. Pre-bump prompts are not retroactively rewritten. The original protocol-doc revision can be recovered via git log on `docs/investigate-fix-protocol.md` if needed.
3. Consumers (Claude Code via the CLAUDE.md hook, Phase 3 auto-fix executor, future analytics) must branch on the `## Protocol` line if they need version-specific behaviour. Reading a `v1` prompt under `v2` execution rules is a contract violation. Consumers must tolerate older versions or refuse to act on them.
4. No `system_incidents.investigate_protocol_version` column is added. Adding one would create a duplicate source of truth (column vs prompt text) and require a backfill story for pre-bump rows. The prompt-body stamp is sufficient.

**Why a version stamp at all.** Without it, a future operator reading an old prompt under new protocol assumptions could mis-execute — for example, assuming a section is required that did not exist in `v1`. The stamp lets readers self-validate before acting.

### 5.3 Updating this document

Edits to this document are normal commits on the main branch. The commit message must include the word `protocol` so git log can locate protocol-revision history:

```
docs(investigate-fix-protocol): <what changed and why>
```

When a change to this document requires a corresponding change to the agent's system prompt, both changes land in the same commit. When a change only affects the document (clarification, example update, iteration note), the system prompt need not change.

After any edit to §2 (prompt structure) or §3 (required sections, forbidden content), the agent's system prompt must be reviewed to confirm it still satisfies the new contract.
