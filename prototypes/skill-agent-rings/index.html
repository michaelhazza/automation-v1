Skill & Agent Ring Rollout — Mockups

How a system agent or system skill revision travels from the author's desk to every organisation — safely, in stages, with automatic pause if anything regresses. Mockups are grouped by user class; each file is one screen, one primary task.

Grounded in the codebase. Each mockup references the actual existing surface it sits on (or notes when it's a net-new route). Source files named in each mockup's footer. Per the brief's recommendation, this feature is on hold until closed-loop skill-improvement (separate brief on main) ships and validates its case — see tasks/dev-brief-staged-rollout.md §11.3.

Design notes

  • Two user classes, different rules. Admin screens (01–07, 10) use the relaxed budget per docs/frontend-design-principles.md. Agency screens (08, 09) obey the strict budget — no metrics, no version numbers in primary copy, version info lives inside a collapsed Advanced section.
  • Existing surfaces vs new routes. Mockups 01 (Agents list), 02 (Agent edit), 07 (Skills list), 08 (Subaccount agent config), 10 (kill switch on skill edit) sit on existing pages. Mockup 03 (Rollout pipeline) and 05 (Canary health) are new routes reachable from the existing edit pages. Mockups 04, 06 are modals. Mockup 09 is a notification surface (in-app banner + email).
  • Versioning already exists. agent_prompt_revisions and skill_versions are live. Rings add per-run pinning columns on agent_runs, a rollout_cohorts table, a skill_disable_flags table, and a reversibility_class column on system_skills.
  • Composes with Explore/Execute Mode. Rings are per-revision safety (platform authors shipping changes); Explore/Execute is per-run safety (operators trying things). Orthogonal.
  • Platform stance: docs/spec-context.md currently says rollout_model: commit_and_revert. Phase 1 (internal release) is technically permitted today; Phase 2 (customer rings) activates when first live agency client onboards.
  • Build sequencing. See brief §11. Closed-loop skill improvement ships first; staged rollout follows only if closed-loop validates its case. Per brief §11.3, holding this work is recommended until both: (i) closed-loop has 4 weeks of internal operation showing measurable improvement, and (ii) there's at least one live agency client.
