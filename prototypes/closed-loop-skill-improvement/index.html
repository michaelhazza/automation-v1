Closed-loop skill improvement

Hi-fi prototype, Round 1. Agents propose improvements; operators approve them.

Flow: Operator opens Inbox (S1), clicks a skill improvement row, reviews it in the side drawer (S2), and accepts, edits, or rejects. Accepted improvements appear in the skill detail page (S3). The run trace (S4) shows how the proposed improvement event surfaces inline after a scorecard fail.
1
Review queue — Skill improvements tab
Extends ReviewQueuePage. New "Skill improvements" tab added to the existing Briefs / Needs Review tabs. Toggle the ViewModeSwitcher to see Workspace vs. Org (cross-workspace roll-up with Subaccount column).
Toggle Workspace / Org in the header to see both variants
2
Amendment review drawer
Side drawer opened from a queue row. Shows skill name, kind pill, proposed text (editable with char counter), diff view, collapsible "Why proposed" block with failing run details, and peer-review verdict. Footer: Accept / Edit & accept (CTA swaps when you type) / Reject.
Type in the textarea to swap "Accept" to "Edit & accept". Click Reject to see the regression toast.
3
Skill detail with active improvements
Extends SubaccountSkillsPage. New "Active improvements (N / 20)" section below skill body. Each row shows kind pill, summary, source badge (only on agent-proposed items), and a Retire action. Click the cap counter to demo amber and red states. One improvement is pre-expanded.
Click the "3 / 20" counter to cycle through cap warning states
4
Run trace — "Improvement proposed" event
Extends RunTracePage / RunTraceEventRenderer. Shows a failed scorecard event (red) immediately above the new improvement-proposed event (violet). The new event card shows kind pill, one-line summary, and a "Review in queue" link to the drawer.
Click the tool-call event cards to expand input/output. "Review in queue" links to S2.
Design decisions, Round 1
  • Multi-screen directory format (four distinct screens with cross-screen navigation).
  • Drawer over full page for the amendment review: the operator's mental context is the queue list, not a new page hierarchy.
  • ViewModeSwitcher reuses the existing pill component exactly (Workspace / Org segments).
  • Worked example uses "Draft Email" skill for all four screens for narrative coherence.
  • Source badge only on agent-proposed improvements; operator-authored rows carry no badge (default case per design rules).
  • Char counter turns amber at 150 chars remaining, red when over limit.
  • Cap counter (N / 20) cycles amber at 15+, red at 20 — demo by clicking the counter on S3.
  • No em-dashes in any copy. Framing is "agents propose, you approve" throughout.
