Workflows · UI prototypes

Brainstorming the Phase-2 UI for the workflows feature. Backend (engine, schema, routes, step types including invoke_automation for n8n / Make / Zapier / GHL) is already built — these mocks explore the operator surface.

01 · Chat-driven Studio with live preview
Rejected as too busy
Three-pane workspace: chat / canvas / inspector all permanently visible. Per-step cost and side-effect badges. Multi-metric validation summary. Felt like Zapier with a chat overlay — violated frontend principles (≥2 primary actions, KPI chrome by default).
02 · Canvas-first, chat collapsed
Recommended · resting state
Canvas is the whole screen. Direct manipulation handles 80% of edits — click to edit, hover-to-insert "+" between steps. Chat is a docked pill bottom-left, opens on demand for big restructures. Step cards show only icon, name, action verb. Cost rolls up to one bottom-bar number. Only irreversible steps wear a flag. Inspector is a slide-out, not a permanent column.
03 · Canvas-first, chat activated
Recommended · activated state
Same workspace as 02 once "Ask Studio" / ⌘K is invoked. Chat slides in from the left as a workspace panel; canvas shrinks but stays primary; inspector closes (one side panel at a time). Demonstrates the "big move" use case: one sentence proposes a multi-field edit that would take several clicks. Agent never silently applies — every change renders as a diff card the user must Apply or Discard, with the canvas previewing the affected step in indigo.
