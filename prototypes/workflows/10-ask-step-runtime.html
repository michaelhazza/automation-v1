State A · Awaiting input. The form card lands in the chat panel as a new message. Same card primitive used for Approvals, just with form fields instead of approve/reject. Fields are required-marked with the red asterisk; the Submit button stays enabled (validation runs on click and surfaces inline). The thinking box reads "Waiting for you to confirm the campaign target." Activity has a single Ask-queued event at the bottom (newest). Plan tab shows step 1 in indigo as the awaiting step.
State B · Validating. Operator clicked Submit without picking an audience segment; the field gets a red border and an inline error message ("Pick a segment to continue"). The Tone field is filled. Submit stays enabled — clicking it without fixing the error just re-shows the validation. No modal, no toast, no extra surface. Same shape as standard form validation across the web.
State C · Submitted. The form card collapses to a green-tinted receipt showing what was submitted, by whom, when. The orchestrator's next message lands in chat ("Got it. Pulling Tier A accounts now."). Activity shows the Ask-submitted event with the values, and the next step starting. Plan tab marks step 1 as ✓ done; step 2 (Find stale leads) is the new current step. The values are now bindings: steps.confirm_campaign_target.outputs.audience = "Tier A".
State D · Routed elsewhere. The Ask was authored with "A team" routing (Marketing team). Mike (the requester) is not the submitter. A team member arrives at the form via one of three paths: sidebar nav with a count badge, a dedicated "Waiting on you" page, or an opt-in email notification. All three paths land on the same task view with the same form card — there is no separate forms surface. The form card text adapts slightly ("Mike H. asked you (you're in the Marketing team)") so the team member has context. The thinking box on the requester's view reads "Waiting on Marketing team to confirm" — same task, same surface, just different audience.

The form-card primitive is one component. Author defines the schema in the Studio inspector (mock 09); the renderer at runtime consumes it and produces the card in this chat panel. Submission persists values on the step run record, exposes them as bindings, and unblocks the workflow. Same lifecycle as Approval cards. See brief §7.3 for the engineering shape.
