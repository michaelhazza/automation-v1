Round 7d note: OrgMemoryPage used "block" terminology throughout ("memory blocks", "block type", "suppress block", "promote block"). These are schema/engineering terms that do not survive translation to non-engineering operators. The Round 7d rebuild drops all schema-verb language and uses plain-language equivalents: "Authored memory", "Auto-memory", "Pending review", "Ignored", "Track again", "Keep ignoring". Additionally, "Tier 1 / Tier 2 / Tier 3" was displayed as a UI category in the before state. This is internal schema jargon. Non-technical operators could not decode what tier numbers meant. Replaced with named categories: Identity, Voice, Positioning, Audience, Constraints, Proof.

Issue: Scope filter is confusing Org-level documents are managed through the subaccount Knowledge page with a "Organisation" scope filter — an admin must know to toggle the scope pill to see org documents. No "Used by N workspaces" indicator. No inheritance controls. No version history. Scope ownership is invisible at-a-glance.

Problems with current org-level knowledge management

  • No unified org Knowledge view: memory blocks live in OrgMemoryPage (/org/memory); documents are a scope filter toggle buried inside the subaccount page. An admin needs to visit two different surfaces.
  • Inheritance not visible from org page: admin cannot see which of the 4 subaccounts inherit each org-level item. The subaccount view shows "Org-inherited" but the org page shows nothing about inheritance.
  • No per-subaccount opt-out: there is no way to disable a specific org-level item for one subaccount without the subaccount creating a local override. The org admin has no inheritance control panel.
  • Documents at org scope are second-class: they have no dedicated UI. Org-scope document management is a mode of the subaccount knowledge page, not a first-class org capability.
  • No "Used by" indicator: the admin cannot tell at a glance whether an org-level item is actually inherited anywhere, or how many workspaces are using it.
  • Org-level rich-text authoring missing: OrgMemoryPage uses a plain textarea for memory block content. The rich-text editor (Bold, Italic, headings, lists, code blocks) exists only in the subaccount knowledge drawer.
  • Authored entries not supported at org scope: freeform reference entries (not memory blocks) cannot be created at org scope. Only memory blocks with a fixed schema exist in OrgMemoryPage.
  • Tier grouping absent: OrgMemoryPage shows a flat list. No Tier 1 / Tier 2 / Tier 3 grouping, even though the schema has the tier field. Admin cannot quickly identify the "system-level" tier-1 blocks from the operational tier-3 items.
After: unified org Knowledge page with Memory + Documents tabs, "Used by N workspaces" indicators, per-subaccount inheritance toggles, rich-text authoring, and tier-grouped memory list. View org-knowledge.html →
