Problems with the current state (AdminAgentEditPage)

  • No indication that this is an org-level definition. The page title says "Edit Agent" with no org-vs-workspace distinction, so admins do not know they are editing a shared template.
  • No visibility into which workspaces are using this agent or how many of them exist. Saving a change affects all 4 workspaces silently.
  • No per-workspace override visibility. Admins cannot see that Revel Group has 5 local overrides and Acme Corp has 2, so changes may unexpectedly conflict with or overwrite those overrides.
  • No tier indicator on skills. System-built skills, org-custom skills, and workspace-scoped skills appear identically in the list. Deleting an org-custom skill here would remove it from all inheriting workspaces with no warning.
  • No Deployments view. There is no surface to see which subaccounts have this agent deployed, their run counts, their override count, or their custom skill additions. That information requires navigating to a separate subaccount admin page.
  • Schedule and budget are mixed inline with identity and model settings. There is no visual hierarchy indicating that "Schedule" and "Budget" are org-level defaults that workspaces can override.
  • Deploying this agent to a new workspace is an entirely separate workflow. There is no "Deploy to workspaces" action on this page, so org admins must navigate away to provision a new deployment.
  • Editing a subaccount-level agent override requires navigating to a separate page (the subaccount's own agent edit). There is no shared-state context between org-level and workspace-level editing flows.
