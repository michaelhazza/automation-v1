Problems with the current state (8 items)

1 Two separate pages for conceptually the same thing: a rule that describes when to invoke an agent. The only difference is the fire condition (time vs event).
2 Mental model split forces operators to ask "is this a schedule or a trigger?" before they can even find the right page. The answer is a detail of the rule, not its identity.
3 Switching between pages mid-flow when configuring related rules: e.g., an operator sets up a daily schedule and then wants a companion event rule. They must navigate away and lose context.
4 No unified history view. Operators cannot see all recent fires across both schedules and triggers in one place without running separate queries or switching pages.
5 Feature parity inconsistency: agent_triggers can only fire agents (no workflow pinning). scheduled_tasks supports an optional pinned workflow. Operators on the Triggers page cannot pin a workflow for their event-fired rules.
6 No clear home for manual one-off invocations (run agent on demand without a schedule or event). Neither page exposes a "Manual" fire condition, so operators must use a separate admin action or the API.
7 No "Run now" button on either page for ad-hoc fires. Operators wanting to test or manually trigger a scheduled task must wait for the next scheduled time or use the API.
8 Schema split (scheduled_tasks plus agent_triggers) means dashboards aggregating fire statistics across all rules require JOIN queries and multiple API calls, adding backend complexity for a single conceptual entity.
