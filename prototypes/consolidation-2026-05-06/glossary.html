System model

Plain-language definitions of every concept in Automation OS. Use this when you need to understand how pieces fit together.

This page is linked from the help icon in the shell. It is a reference only -- nothing on this page changes your configuration.
How the pieces relate
Tenant / Org → has many → Agents → run → Runs
Agents → use → Skills → composed of → Blocks
Automations → triggered by → webhook / schedule / agent action
Workflows → multi-step orchestration, can call → Automations
Knowledge → provides context to → Agents
Core primitives
Agent
Primitive
An AI worker with a name, a role, a set of skills, and a personality defined by its system prompt. Agents run tasks, communicate with external services, and report back. Think of an agent as a persistent employee who is always ready to take instructions.
Not to confuse with: A "run" is a single execution of the agent doing something. The agent is the worker; the run is the work session.
Research Agent CRM Agent Outreach Agent Daily Brief Agent

Automation
Primitive
A single-action trigger-and-run unit. Automations are triggered by a webhook, a schedule, or an agent action. They execute a defined task (e.g. "send a summary email", "create a CRM record") and complete. They are stateless -- each run is independent.
Not to confuse with: A Workflow is a multi-step orchestration that can chain automations together. An Automation is one step; a Workflow is a process with multiple steps.
View Automations page →
Workflow
Primitive
A multi-step orchestration that coordinates agents, automations, and human review checkpoints in a defined sequence. Workflows are authored using the Workflow Studio (system admin) and instantiated as runs. Each step can wait for human approval before proceeding.
Not to confuse with: An Automation is one discrete action. A Workflow chains many actions together, can branch, can pause for HITL, and has an overall lifecycle from "started" to "complete".
View Workflows page →
Run
Execution
A single execution of an Agent, Automation, or Workflow. A run has a start time, an end time, a status (pending, running, awaiting input, completed, failed), and a trace log of every step taken. Runs are immutable once complete.
Not to confuse with: A Job is a queued unit of work waiting to become a run. The run is the live or completed execution; the job is the scheduler or queue entry that produced it.

Job
Execution
A unit of work in the queue, waiting to be picked up and executed. Jobs are created by schedules, webhooks, agent tool calls, and API requests. A job transitions to a run when it starts executing. If no worker is available the job waits; if it fails to start it can be retried.
Not to confuse with: A run is the live execution. A job is the entry in the queue before execution starts. Most operators never need to see jobs directly.
Skill
Capability
A packaged capability that an Agent can use. Skills define what an agent is allowed to do: search the web, read CRM records, send emails, create tasks. Each skill has a name, a tool definition, and optional parameters. Agents can have many skills; skills can be shared across agents.
Not to confuse with: A Block is the implementation unit inside a skill (the actual code or API call). A skill is the contract; a block is the implementation.
web_search send_email create_crm_contact read_calendar
Block
Capability
The implementation unit inside a Skill. A block is a single executable step: call an API, transform data, run a subagent, wait for a condition. Skills are composed of one or more blocks. Blocks are versioned and can be reused across skills.
Not to confuse with: A Block Type is the template or class. A Block is an instance of a Block Type configured with specific parameters.
Block Type
Capability
The schema and behaviour template for a class of Blocks. Block Types are defined by system administrators (e.g. "HTTP request", "Run subagent", "Wait for approval", "Evaluate condition"). Operators instantiate blocks from block types when building skills.
Not to confuse with: A Block is a configured instance. A Block Type is the template. The relationship is the same as a class (Block Type) and an object (Block).
Tenancy
Org (Tenant)
Tenancy
An Organisation is the top-level tenant in Automation OS. It owns agents, automations, workflows, connections, and knowledge. All billing, permissions, and data isolation are per-org. When you log in, you are always acting inside one org.
Not to confuse with: A Subaccount (Client) is a workspace inside your org, typically representing one of your end clients. The org is your agency; the subaccount is each client you serve.
Subaccount (Client)
Tenancy
A Subaccount represents one client (end customer) inside your org. Agents can be scoped to run work for a specific subaccount -- reading that client's data, generating reports for them, or sending communications on their behalf. Subaccounts have their own knowledge, tags, team members, and permissions.
Not to confuse with: The Org is your agency. A Subaccount is one client. An Agent can work for multiple subaccounts, but its runs are always tagged to the subaccount they were running for.
Project
Container
A container that groups work items (tasks, boards, deliverables) together for a specific objective. Projects belong to a subaccount. An agent working on a project can create tasks inside it, move items between stages, and generate deliverables.
Knowledge
Knowledge
Knowledge
Everything an Agent knows about your organisation, clients, and rules. Knowledge has two sources: Auto-memory (things the agent observed and stored automatically from past runs) and Authored knowledge (documents, rules, and facts you wrote explicitly). Agents load relevant knowledge into context before each run.

Memory Block
Knowledge
A single stored memory -- a fact, a rule, a preference, or an observation -- that an agent extracted from past runs or that you authored explicitly. Memory blocks have a type (rule, fact, preference, observation) and a dot-notation label for identification (e.g. client.communication_style.tone).
Not to confuse with: An Entry is an auto-memory record tied to a specific run event. A Memory Block is a distilled, reusable fact surfaced from entries.
