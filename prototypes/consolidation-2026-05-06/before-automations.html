Mental model: one surface for event-driven work

Problems with the three-page before-state

  • Three separate nav items (Workflows, Automations, Triggers) for what is conceptually one capability: event-driven automated work.
  • A trigger's target workflow lives on a different page; clicking a trigger chip requires navigating away and re-finding the workflow.
  • Creating a new automated process that needs a trigger requires switching pages mid-flow: Automations to create the recipe, then Triggers to wire the event, then Workflows if a multi-step flow is involved.
  • AutomationsPage and AdminAutomationsPage are near-duplicate routes showing the same data with minor permission differences, creating two separate code paths and two separate cognitive contexts for the same list.
  • Different visual patterns per page: Workflows uses card grid, Automations uses table, Triggers uses table with no cross-reference links. Inconsistent mental model.
  • An operator viewing a failed workflow has no quick path to the trigger that fired it; they must manually find it on the Triggers page by name.
