Personal Assistant V1 — Round 2 Prototypes
Hi-fi clickable prototypes after the round-1 design pushback: capability-first (not provider-first) wizard, Personal zone on existing home (no dedicated EA dashboard), tabbed per-agent detail page.
EA V1 brief → Foundation brief (user-owned-agents) →
Round-2 changes from round 1:
  • Mockup 01: wizard cards changed from provider names (Gmail / Slack) to capabilities (Email / Calendar / Files / Team chat) — aligns with existing capability taxonomy in integration-reference.md. Outlook / Teams slot in as future providers.
  • Mockup 02: no longer a dedicated "My EA Home" page. The EA's home card now lives in a new Personal zone at the top of the existing home page. Same data-driven pattern handles any future user-principal agent (Dev Agent, etc.).
  • Mockup 03: reorganised as a tabbed per-agent detail page: Workspace (drafts inbox + briefing) | Activity (run trace) | Settings (config). Generic tab pattern reusable for any user-principal agent.
Foundation predecessor: these mockups assume user-owned-agents has merged (renamed and simplified from the earlier principal-scoped-agents per the 2026-05-12 challenge round — single agents.owner_user_id column, no principal abstraction). The EA is a user-owned agent — each user has their own agent row bound to their identity, with credentials and runs scoped to them. Org admins see metadata only; content is redacted by default.
V1 scope tightening (2026-05-12): Calendar deferred to read-only V1 (write capability follows in V1.5). All third-party email/Slack sends review-gated. Auto-send only to self (Slack DM to operator). Reason: prove the user-owned primitive without competing on personal-productivity richness against Claude / ChatGPT / Codex. Detail in EA V1 brief §0.5.

Architectural alignment to the existing codebase
What's reused, not invented:
  • Capability taxonomy (read/write capability slugs + aliases) — already lives in docs/integration-reference.md, CI-enforced.
  • WorkspaceAdapter contract (email + calendar backend abstraction) — already exists with synthetos_native + google_workspace backends. Outlook slots in here.
  • Capability map per agent — already computed by capabilityMapService.ts from skill links × integration reference.
  • Live-fetch vs canonical readPath — already on every action. EA stays live-fetch for source data; canonical only for derived artefacts (voice, memory, drafts).
  • Scheduled tasks + agent triggers + sidebar config + OAuth providers — all reused unchanged.
What's new but reusable:
  • Capability grouping layer (~30 lines in server/config/capabilityGroups.ts) — maps user-facing groups to existing slugs.
  • Home-widget contribution contract on system-agent template — Dev Agent and any future user-principal agent contributes its own card via the same pattern. Zero hardcoding.
  • Voice Profile primitive — generic scoped resource (per user / per subaccount / per org) with pluggable samplers. EA is the first consumer; Riley / Helena / Sarah become future consumers.
Review questions to look for as you click through
  • 01: Capability-first framing — does "Email" / "Calendar" / "Files" / "Team chat" read naturally? Does the provider picker (shown inline under Files) feel right or should it be a modal?
  • 02: Is the Personal zone visually distinct from the existing home content without being jarring? Does the dashed Dev Agent placeholder make the abstraction clear? Should the zone collapse when the user has no user-principal agents (current design: hidden entirely)?
  • 03: Do the three tabs (Workspace / Activity / Settings) cover the work surfaces? Is "Workspace" the right name for the drafts + briefing view, or should it be "Today" / "Inbox" / something else?
