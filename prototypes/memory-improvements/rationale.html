Memory Improvements — UI Rationale

Honest CEO-level summary: who clicks each surface, why, and whether it is actually required in v1 or whether a cheaper non-UI path would do.

Important framing: none of these surfaces are user-facing in the customer sense. All three are operator / admin tooling. The customer benefit is indirect: better AI memory behaviour, fewer hallucinations, faster trust building. The UI gain accrues to the team running the platform, not to the end user.

1. Memory Block Sources tab Required

Who clicks here

An operator (or support engineer) investigating "why does this agent think X?" An end user does not see this.

Why it exists

A separate in-flight build, Trust & Verification Layer, is already adding a Source: Auto | Manual | Correction pill to memory blocks. Today, the Auto case has nothing behind it. Operators see "this came from AI synthesis" and cannot answer "from what?" This mockup is the click-through target for that pill.

Is it actually required?

Yes, but conditionally. If Trust & Verification ships its Source pill, this surface must exist or the pill is a dead-end. If Trust & Verification is paused or descoped, this becomes optional internal tooling.

Where it lives

Extends the existing MemoryBlockDetailPage. New tab alongside Version History and Diff vs Canonical. No new top-level page, no new navigation.

Cheapest alternative

API endpoint only (/api/memory-blocks/:id/sources) with no UI. Operators query via internal tools. Loses the click-through audit story; keeps the data integrity.

2. Citation Utility dashboard Optional in v1

Who clicks here

An admin tracking whether the AI memory we inject is actually being used by agents. End users do not see this.

Why it exists

We have no way today to answer "of the memory we put into agent prompts, what percentage was actually used?" Without this signal, every retrieval improvement is unfalsifiable. We could ship "improvements" indefinitely with no way to know if they help. This is the difference between governing AI memory and guessing about it.

Is it actually required?

The data layer (B1) is required for safe rollout of Proposal D. The dashboard (B2) is convenience on top of it. The reviewer explicitly split these: D only needs the queryable data substrate. A dashboard becomes valuable once retrieval changes are happening regularly, but it does not block anything in v1.

Where it lives

Extends the existing UsagePage. New tab labelled "Memory Utility" alongside Runs / Routing / Spend. No new top-level page, no new navigation. Admin-only label.

Cheapest alternative

Ship B1 (the data layer) with no UI. Operators run SQL against the materialised view during retrieval-change rollouts. Add B2 only when retrieval tuning becomes a recurring activity.

3. AKR Semantic Ranker (Proposal D) Backend only — no UI

Rev 6 simplification: earlier revisions of the brief included a four-mode staged rollout (off / shadow / sampled / on) with a per-subaccount admin UI, a shadow-mode comparison panel, and a baseline-window invariant. All of that was dropped in Rev 6 after CEO review identified it as over-engineering for a pre-production codebase. D ships as a backend algorithm change behind a single env-driven on/off flag. No subaccount-admin settings page. No shadow telemetry. No per-run comparison panel. The mockup file akr-ranker-settings.html was retired in the same revision.

Who clicks anything for D

Nobody. Customers see only the resulting retrieval quality. Platform engineering controls the env flag; if a regression appears in dev testing, the flag flips off and the system reverts to today's behaviour in seconds.

Why no UI

The retrieval ranker is internal AI infrastructure. A subaccount admin choosing between "scope-tier" and "semantic" ranking is like a search-engine user choosing between BM25 and vector retrieval — they do not care about the algorithm, they want good results. Building a UI to expose this choice would surface engineering rollout machinery as a product feature it is not.

What still matters even without UI

Two algorithm safety invariants from §4 of the brief remain mandatory. They are properties of the ranker itself, not staged-rollout discipline:

  • D-Recall invariant: if semantic filtering at the configured threshold would empty a previously non-empty candidate category, the ranker must fall back to legacy top-N ordering for that category and emit a degraded reason. Prevents accidental memory starvation regardless of flag state.
  • D-Embedding-failure invariant: OpenAI failures must fail open to legacy retrieval and emit a degraded reason on the run trace. They must never block an agent run.

When the rollout-machinery discussion comes back

When the codebase approaches live-user readiness and the production risk profile materially changes. The Rev 5 staged-rollout shapes are preserved in git history for that reopening; they are deferred, not deleted.
