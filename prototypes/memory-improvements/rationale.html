Memory Improvements — UI Rationale

Honest CEO-level summary: who clicks each surface, why, and whether it is actually required in v1 or whether a cheaper non-UI path would do.

Important framing: none of these surfaces are user-facing in the customer sense. All three are operator / admin tooling. The customer benefit is indirect: better AI memory behaviour, fewer hallucinations, faster trust building. The UI gain accrues to the team running the platform, not to the end user.

1. Memory Block Sources tab Required

Who clicks here

An operator (or support engineer) investigating "why does this agent think X?" An end user does not see this.

Why it exists

A separate in-flight build, Trust & Verification Layer, is already adding a Source: Auto | Manual | Correction pill to memory blocks. Today, the Auto case has nothing behind it. Operators see "this came from AI synthesis" and cannot answer "from what?" This mockup is the click-through target for that pill.

Is it actually required?

Yes, but conditionally. If Trust & Verification ships its Source pill, this surface must exist or the pill is a dead-end. If Trust & Verification is paused or descoped, this becomes optional internal tooling.

Where it lives

Extends the existing MemoryBlockDetailPage. New tab alongside Version History and Diff vs Canonical. No new top-level page, no new navigation.

Cheapest alternative

API endpoint only (/api/memory-blocks/:id/sources) with no UI. Operators query via internal tools. Loses the click-through audit story; keeps the data integrity.

2. Citation Utility dashboard Optional in v1

Who clicks here

An admin tracking whether the AI memory we inject is actually being used by agents. End users do not see this.

Why it exists

We have no way today to answer "of the memory we put into agent prompts, what percentage was actually used?" Without this signal, every retrieval improvement is unfalsifiable. We could ship "improvements" indefinitely with no way to know if they help. This is the difference between governing AI memory and guessing about it.

Is it actually required?

The data layer (B1) is required for safe rollout of Proposal D. The dashboard (B2) is convenience on top of it. The reviewer explicitly split these: D only needs the queryable data substrate. A dashboard becomes valuable once retrieval changes are happening regularly, but it does not block anything in v1.

Where it lives

Extends the existing UsagePage. New tab labelled "Memory Utility" alongside Runs / Routing / Spend. No new top-level page, no new navigation. Admin-only label.

Cheapest alternative

Ship B1 (the data layer) with no UI. Operators run SQL against the materialised view during retrieval-change rollouts. Add B2 only when retrieval tuning becomes a recurring activity.

3. AKR Ranker Settings + Shadow Comparison Mode selector required, comparison optional

Codebase reality check + CEO scope correction: the original mockup labelled this "Settings → Retrieval" (org-level). That subsection does not exist, and more importantly, org-level is the wrong scope — each subaccount must act independently. The mode selector lives as a section on SubaccountKnowledgePage (per-subaccount). An org-level default may exist behind the scenes, but the subaccount admin sees this as their own setting (no "inherited from org" badges, no greyed fields). The shadow-comparison panel lives on the existing RunTracePage, where operators already debug individual runs.

Who clicks the mode selector

A subaccount admin rolling out a retrieval-quality change for their workspace. End users do not see this.

Why the mode selector exists

Proposal D changes how every agent run picks memory. The reviewer mandated a four-mode rollout (Off → Shadow → Sampled → On) for safety. Each subaccount must flip its own mode independently — an agency's clients evolve at different paces and risk profiles. Someone has to control this per subaccount; that someone is the subaccount admin, and they need a UI surface to do it without engineering involvement.

Is the mode selector actually required?

Yes — unless we accept env-only configuration. Env-only means engineering deploys a config change every time we want to enable shadow mode for a new subaccount. Does not scale past a handful of subaccounts. With more, you need an admin UI. It can be small.

Where the mode selector lives

A new section on SubaccountKnowledgePage (per-subaccount), not a new page. The mode is a first-class subaccount control — the subaccount admin sees this as their setting, with no visible inheritance chain. Behind the scenes, resolution at runtime is: subaccount value if set → org default → env-driven platform default. Subaccount-level override is stored via a subaccount-scoped settings primitive; org-level defaults via the existing orgSettingsService. No new column on the organisations table.

Who clicks the shadow-comparison panel

An operator inspecting a specific agent run during D rollout, asking "would the new ranker have changed what got loaded?" Spot-check tool. Not used routinely once D ships at on.

Is the comparison panel actually required?

No. It is convenience. The data behind it (the shadow-comparison telemetry) is mandatory for the rollout invariant. The visual diff is a debugging affordance. Could ship as telemetry-only and inspect via SQL.

Where the comparison panel lives

If we ship it, it extends the existing RunTracePage — a collapsible panel below the existing trace, defaulted closed. It does not belong in Settings.

Cheapest alternative

  • Mode selector: env-only config, no UI. Acceptable for first 1-3 tenants only.
  • Comparison panel: telemetry-only, no UI. Operators query the shadow-compare table directly during rollout investigation.

Bottom line. The only UI surface unconditionally required is the D mode selector, and only because per-tenant operator control is required for safe rollout. Everything else is either conditional on another in-flight build (A on Trust & Verification) or a convenience layer over a mandatory data substrate (B2 on B1, D-comparison on D-telemetry). If we want a truly minimal v1, we can ship: lineage data + minimal Sources tab, B1 substrate only with no dashboard, D mode selector only, no shadow-comparison panel. That is the cheapest path through the brief. The mockups show the maximal path; the spec should pick where on the spectrum to land.
